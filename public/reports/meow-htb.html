Resumen Ejecutivo

La máquina Meow expone el servicio Telnet en el puerto 23 sin ningún tipo de autenticación para el usuario root. Esto significa que cualquier persona con acceso a la red puede obtener una shell con privilegios máximos en cuestión de segundos, sin necesitar contraseña alguna. El ataque requirió 4 comandos y aproximadamente 5 minutos desde el inicio hasta obtener la flag.

¿Qué es Telnet y por qué es peligroso?

Telnet es un protocolo de red para controlar remotamente otro computador a través de una terminal de texto. Fue creado en 1969 y fue el estándar de administración remota durante décadas, hasta que SSH lo reemplazó en los años 90.

⚠ ¿Por qué es peligroso hoy?

  • Texto plano: Todo lo que se transmite — contraseñas, comandos, datos — viaja sin cifrar. Cualquiera que capture el tráfico de red puede leerlo directamente.
  • Sin autenticación fuerte: Telnet solo usa usuario/contraseña, sin criptografía moderna ni certificados.
  • Configuraciones inseguras: Algunos sistemas dejan Telnet habilitado con root accesible y sin contraseña, especialmente dispositivos legacy o mal administrados.

Analogía: Usar Telnet hoy es como enviar una carta con tu contraseña escrita en el sobre, sin sellarlo, por un mensajero que puede leerla.

Reconocimiento — Paso a Paso

Paso 1 — Reconocimiento

Verificar conectividad con ping

¿Por qué este comando?

Antes de cualquier escaneo, confirmamos que la máquina está encendida y que nuestra VPN de HTB funciona correctamente. Si ping falla, ningún otro comando funcionará. Además, el campo TTL de la respuesta nos permite identificar el sistema operativo sin lanzar ningún escaneo agresivo.

ping -c2 10.129.18.182
ping → herramienta de diagnóstico ICMP — envía Echo Request y espera Echo Reply
-c2 → (count 2) enviar exactamente 2 paquetes y detenerse. Sin este flag, ping correría indefinidamente hasta Ctrl+C.
10.129.18.182 → dirección IP del objetivo en la red VPN de HTB
Output

64 bytes from 10.129.18.182: icmp_seq=2 ttl=63 time=44.8 ms

2 packets transmitted, 2 received, 0% packet loss
¿Qué nos dice este output?
  • 2 received, 0% packet loss → máquina activa y VPN funcionando correctamente
  • ttl=63 → Linux confirmado. El TTL (Time To Live) es un campo en paquetes IP que se decrementa en 1 en cada router. Linux parte con TTL inicial de 64; con 1 salto de red VPN llega como 63. Windows partiría de 128 y llegaría como 127.

Decisión tomada

Máquina Linux activa y accesible. Procedemos a descubrir qué servicios tiene expuestos con nmap.

Paso 2 — Reconocimiento

Descubrir puertos abiertos — Nmap fase rápida

¿Por qué este comando?

Necesitamos un mapa completo de los servicios expuestos en el objetivo. Cada puerto abierto es una posible "puerta" de entrada al sistema. Escaneamos todos los 65535 puertos para no perdernos ningún servicio, incluso en puertos no estándar.

nmap -p- --open -n -T4 -sS 10.129.18.182
nmap → Network Mapper — herramienta estándar de escaneo de puertos en seguridad
-p- → escanear todos los 65535 puertos TCP (del 1 al 65535). Sin este flag, solo escanea los 1000 más comunes.
--open → mostrar solo puertos abiertos. Los cerrados y filtrados no son vectores inmediatos.
-n → no resolver nombres DNS — acelera el escaneo evitando consultas DNS innecesarias.
-T4 → velocidad "aggressive". Escala T0-T5: T4 es rápido sin causar pérdida excesiva de paquetes.
-sS → SYN scan (half-open): envía SYN, analiza respuesta, no completa el handshake TCP. Más rápido y sigiloso.
Output

PORT   STATE SERVICE
23/tcp open  telnet

Nmap done: 1 IP address scanned in 47.23 seconds
¿Qué nos dice este output?
  • Solo 1 puerto abierto de 65535: el 23/tcp
  • Puerto 23 = Telnet — el protocolo de terminal remota sin cifrado
  • Superficie de ataque mínima: un único servicio = un único vector de ataque. Sin alternativas.
Decisión tomada

Un único vector: Telnet en puerto 23. Pasamos a la fase profunda para obtener la versión exacta del servicio.

Paso 3 — Reconocimiento

Enumerar Telnet en detalle — Nmap fase profunda

¿Por qué esta segunda fase de nmap?

El primer escaneo (-sS) solo nos dice si el puerto está abierto. Esta segunda fase detecta la versión exacta del software y ejecuta scripts NSE (Nmap Scripting Engine) que automatizan reconocimiento adicional — en algunos casos, el script ya revela la vulnerabilidad directamente.

nmap -p23 -sC -sV -n 10.129.18.182
-p23 → solo el puerto 23. No re-escanear todos los 65535.
-sC → ejecutar scripts NSE por defecto. Para Telnet: telnet-* que captura el banner y extrae información del sistema.
-sV → detectar la versión exacta del servicio. Nmap envía sondas específicas y analiza respuestas para identificar software y versión.
-n → sin resolución DNS, igual que antes.
Output
PORT   STATE SERVICE VERSION
23/tcp open  telnet  Linux telnetd

Service Info: OS: Linux; CPE: cpe:/o:linux:linux_kernel
¿Qué nos dice este output?
  • Linux telnetd → daemon de Telnet estándar de Linux, no una implementación custom
  • OS: Linux → nmap confirma Linux por segunda vía independiente (fingerprinting de TCP/IP)
  • Nmap no detectó requerimiento de autenticación en el banner → señal de alerta de misconfiguration
Decisión tomada

Telnet en Linux confirmado. El árbol de decisión Ghost-Ops indica: intentar root sin contraseña como primer paso — es el más probable en máquinas Very Easy con un único puerto Telnet.

Explotación — Paso a Paso

Paso 4 — Explotación

Conectarse a Telnet como root sin contraseña

¿Por qué root sin contraseña?

Cuando Telnet está expuesto en un sistema Linux, la misconfiguration más común es el usuario root sin contraseña. En dispositivos legacy, routers, o sistemas de desarrollo mal configurados, el administrador desactivó la autenticación de root para agilizar el setup y nunca la reactivó.

¿Por qué no funciona un pipe simple? Telnet requiere una negociación inicial (handshake de opciones: WILL/WONT/DO/DONT) antes de mostrar el login. Un pipe simple (echo "root" | telnet IP) no puede manejar este handshake y la conexión se cierra inmediatamente. La solución es usar expect — herramienta que automatiza interacciones con programas interactivos.

expect -c 'spawn telnet 10.129.18.182; expect "login:"; send "root\r"; expect -re "(Password:|#)"; send "\r"; expect "# "; send "cat /root/flag.txt\r"; expect eof'
expect → herramienta que automatiza interacciones con programas interactivos de terminal
-c → ejecutar el script expect como string inline (sin archivo separado)
spawn telnet IP → lanza telnet como proceso hijo que expect controlará
expect "login:" → esperar hasta recibir el texto "login:" del servidor
send "root\r" → enviar "root" + retorno de carro (equivalente a Enter)
expect -re "(Password:|#)" → esperar cualquiera: si pide password continúa; si da prompt (#) ya entró
send "\r" → enviar Enter vacío (contraseña vacía)
expect "# " → esperar el prompt de shell root que indica login exitoso
send "cat /root/flag.txt\r" → leer la flag una vez dentro

¿Por qué existe esta vulnerabilidad?

Telnet + Root sin contraseña — Misconfiguration crítica

Esta misconfiguration ocurre típicamente por:

  • Setup de desarrollo: El administrador desactivó la contraseña de root para agilizar la configuración inicial y nunca la reactivó.
  • Sistema legacy: Telnet fue configurado décadas atrás cuando la seguridad no era prioritaria.
  • Red "aislada": El equipo de IT asumió que al estar en red interna no había riesgo — olvidando atacantes internos o errores de segmentación.
  • Falta de gestión: Nadie revisa qué servicios están corriendo ni con qué configuración.

En producción real, un atacante con acceso root puede instalar backdoors, leer /etc/shadow, pivotar hacia otros sistemas, destruir evidencia y cifrar archivos para ransomware.

Lecciones Aprendidas

⚔ Para el Atacante

  • Puerto 23 abierto → siempre intentar Telnet primero
  • TTL 63-64 → Linux (cambia estrategia si 127 → Windows)
  • Orden de intentos: root vacío → root:root → admin:admin
  • Usar expect para automatizar Telnet (pipes simples fallan)
  • Un único puerto = un único vector → no perder tiempo buscando alternativas

🛡 Para el Defensor

  • Deshabilitar Telnet: systemctl disable telnetd
  • Usar SSH en su lugar — cifrado y seguro
  • Establecer contraseña para root: passwd root
  • Deshabilitar login root directo: PermitRootLogin no
  • Firewall: iptables -A INPUT -p tcp --dport 23 -j DROP
  • Detectar en logs: /var/log/auth.log → ROOT LOGIN FROM IP
