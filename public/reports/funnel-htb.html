Descripción

Funnel es una máquina Linux de nivel Very Easy en HackTheBox Starting Point. Simula un escenario empresarial realista donde el FTP anónimo expone documentos internos —un email de bienvenida y una política de contraseñas— que revelan una contraseña por defecto. Un usuario del equipo no la cambió. Con esas credenciales se accede por SSH y se descubre un PostgreSQL solo en loopback (no visible desde fuera). Se utiliza SSH Port Forwarding para crear un túnel local y así conectarse a la base de datos interna donde está la flag.

→
PDF → pass defecto
→
SSH christine
→
SSH -L túnel 5432
→
psql → FLAG ✅

El nombre "Funnel" hace referencia al embudo (funnel) de información: el FTP filtra datos que llevan a credenciales, que llevan al túnel, que llevan a la base de datos. Cada capa alimenta la siguiente.

Paso 1 — Nmap

Descubrimiento de puertos y versiones

¿Por qué este comando?

Necesitamos saber qué servicios están expuestos. -sC activa scripts por defecto de nmap que prueban FTP anónimo automáticamente. -sV detecta versiones para identificar vulnerabilidades conocidas. -p- garantiza que no nos perdemos puertos no estándar. -T4 acelera el escaneo sin disparar falsos positivos.

nmap -sC -sV -p- --open -T4 10.129.7.157
-sC → ejecuta scripts NSE por defecto (incluye ftp-anon que detecta FTP anónimo)
-sV → detecta versión exacta de cada servicio activo
-p- → escanea los 65535 puertos TCP, no solo el top 1000
--open → muestra solo puertos en estado OPEN (filtra filtered/closed)
-T4 → plantilla de velocidad "agresiva": timeouts cortos, paquetes más rápidos
10.129.7.157 → IP objetivo en la red de HTB

Paso 2 — FTP Anónimo

Exploración y descarga de archivos sensibles

¿Qué es FTP anónimo y por qué es peligroso?

FTP (File Transfer Protocol, puerto 21) permite transferir archivos entre cliente y servidor. Cuando está configurado con acceso anónimo, cualquier usuario puede conectarse usando el nombre de usuario anonymous y cualquier texto como contraseña (convencionalmente, el email del usuario). Administradores lo usan para distribución pública de archivos, pero a veces quedan expuestos directorios con información interna.

ftp anonymous@10.129.7.157
ftp → cliente FTP de línea de comandos de Linux
anonymous@IP → especifica usuario anónimo. Contraseña: cualquier texto (dejamos vacío o escribimos un email)

¿Por qué extraer el PDF con Python?

Los PDFs almacenan su contenido en streams comprimidos con zlib (filtro FlateDecode). Herramientas como pdftotext o strings no siempre extraen texto de estos streams correctamente. Python con zlib.decompress() descomprime el stream raw y nos da el texto plano del PDF.

python3 -c "import re,zlib; d=open('password_policy.pdf','rb').read(); [print(zlib.decompress(bytes.fromhex(m.group(1).hex()))) for m in re.finditer(b'stream\r?\n(.*?)\r?\nendstream',d,re.DOTALL) if b'\x78' in bytes.fromhex(m.group(1)[:2].hex())]"
open('password_policy.pdf','rb') → leer el PDF en modo binario
re.finditer(b'stream...endstream') → encontrar todos los streams de datos dentro del PDF
zlib.decompress() → descomprimir el stream FlateDecode (compresión zlib/deflate)
b'\x78' → magic byte que indica compresión zlib (0x78 = inicio del deflate header)

Explotación — Paso a Paso Detallado

Paso 4 — SSH con contraseña por defecto

Probar funnel123#!# en los 5 usuarios

¿Por qué sshpass en lugar de SSH interactivo?

SSH no acepta contraseñas en argumentos de línea de comandos por seguridad. sshpass es una utilidad que inyecta la contraseña de forma no interactiva, permitiendo automatizar con bucles de shell. Es el equivalente simple al módulo ssh_login de Metasploit para este caso básico.

for user in optimus albert andreas christine maria; do
  sshpass -p 'funnel123#!#' ssh -o StrictHostKeyChecking=no $user@10.129.7.157 'id' 2>/dev/null && echo "EXITO: $user"; done
for user in ... → iterar sobre los 5 usuarios encontrados en el email
sshpass -p 'funnel123#!#' → proporcionar contraseña de forma no interactiva
-o StrictHostKeyChecking=no → no preguntar por confirmación de host key (automatización)
'id' → comando a ejecutar: si el login funciona, muestra uid/gid del usuario
2>/dev/null → suprimir mensajes de error de conexiones fallidas
&& echo "EXITO: $user" → solo imprime si el comando anterior tuvo exit code 0
Output del bucle
# optimus → Permission denied (publickey,password)
# albert  → Permission denied (publickey,password)
# andreas → Permission denied (publickey,password)
uid=1000(christine) gid=1000(christine) groups=1000(christine)
EXITO: christine
# maria   → Permission denied (publickey,password)
¿Qué nos dice?
  • 4 usuarios cambiaron su contraseña correctamente
  • christine no siguió la política corporativa y mantiene la contraseña por defecto
  • uid=1000 → usuario normal del sistema, no root. Necesitamos explorar qué servicios internos están accesibles
Decisión

Paso 5 — Enumeración de servicios internos

ss -tlnp — Ver qué escucha en loopback

¿Por qué ss y no netstat?

ss (Socket Statistics) es el reemplazo moderno de netstat en sistemas Linux actuales. Más rápido y con más información. Nos muestra todos los sockets TCP activos — especialmente los que escuchan solo en 127.0.0.1 (loopback), que no son visibles desde fuera con nmap pero sí son accesibles internamente.

ss -t l n p
-t → solo sockets TCP (no UDP)
-l → solo sockets en estado LISTEN (escuchando, esperando conexiones)
-n → mostrar números de puerto, no nombres de servicio (más rápido, más claro)
-p → mostrar el proceso/PID que tiene abierto cada socket
Output ss -tlnp (ejecutado vía SSH)
Netid  State  Recv-Q  Send-Q  Local Address:Port  Peer Address:Port
tcp    LISTEN   0      128     0.0.0.0:22          0.0.0.0:*         users:(("sshd",...))
tcp    LISTEN   0      128     127.0.0.1:5432      0.0.0.0:*         users:(("postgres",...))
tcp    LISTEN   0      128     [::]:22             [::]:*
¿Qué encontramos?
  • 0.0.0.0:22 → SSH escucha en todas las interfaces (ya lo sabíamos por nmap)
  • 127.0.0.1:5432 → PostgreSQL escucha solo en loopback. El 127.0.0.1 significa que solo acepta conexiones locales — por eso nmap no lo vio desde fuera
  • Puerto 5432 → puerto estándar de PostgreSQL
  • Proceso postgres → confirma que es el motor de base de datos PostgreSQL
Decisión

PostgreSQL interno en 5432. No es accesible directamente desde Kali. Usar SSH Port Forwarding para crear un túnel local y "traer" el puerto a nuestra máquina.

Paso 6 — SSH Port Forwarding (Local Tunnel)

Redirigir PostgreSQL interno a puerto local

¿Qué es SSH Local Port Forwarding y por qué funciona aquí?

SSH no solo sirve para abrir shells. La opción -L crea un túnel TCP: cualquier conexión al puerto local que especificamos se encripta por SSH y se redirige al servidor remoto, que la reenvía al destino. En este caso: localhost:15432 → [encrypted SSH] → 10.129.7.157 → 127.0.0.1:5432. El servidor hace el salto final a su propio loopback — algo que nosotros no podemos hacer directamente.

ssh -f -N -L 15432:127.0.0.1:5432 christine@10.129.7.157
-f → fork: poner el proceso SSH en background automáticamente (no bloquea la terminal)
-N → no ejecutar ningún comando remoto (solo mantener el túnel activo, sin abrir shell)
-L 15432:127.0.0.1:5432 → Local forward:
   • 15432 = puerto en NUESTRA máquina Kali donde escucha el túnel
   • 127.0.0.1:5432 = destino desde el punto de vista del servidor remoto (su propio loopback PostgreSQL)
christine@10.129.7.157 → usuario SSH y servidor que actúa como "puente" del túnel
Usamos 15432 en lugar de 5432 para evitar conflictos con PostgreSQL local en Kali (si existiera)
Verificación del túnel
# Confirmar que el puerto local está escuchando:
ss -tlnp | grep 15432
tcp  LISTEN  0  128  127.0.0.1:15432  0.0.0.0:*  users:(("ssh",...))

# El proceso ssh está forwarding el puerto — túnel activo ✅
Cómo funciona el túnel
  • Kali localhost:15432 → proceso ssh en Kali recibe la conexión
  • ssh encripta el tráfico y lo envía por el canal SSH a 10.129.7.157
  • El servidor SSH en 10.129.7.157 desencripta y conecta a 127.0.0.1:5432 (su PostgreSQL local)
  • Para PostgreSQL, la conexión parece venir de 127.0.0.1 (local) → acepta sin restricciones de red
Decisión

Túnel activo en localhost:15432. Conectar con psql usando las credenciales de christine.

Paso 7 — PostgreSQL: encontrar la flag

Conexión, exploración de bases de datos y extracción

¿Por qué usamos PGPASSWORD en lugar de introducir la contraseña interactivamente?

psql pide contraseña interactivamente por defecto. La variable de entorno PGPASSWORD permite pasarla de forma no interactiva, útil para scripting y automatización. psql es el cliente oficial de línea de comandos de PostgreSQL.

PGPASSWORD='funnel123#!#' psql -h 127.0.0.1 -p 15432 -U christine
PGPASSWORD='funnel123#!#' → variable de entorno con la contraseña (misma que SSH — reutilización de credenciales)
-h 127.0.0.1 → conectar a localhost (donde escucha nuestro túnel SSH)
-p 15432 → puerto del túnel local (no el 5432 real del servidor)
-U christine → usuario de PostgreSQL (mismo nombre que el usuario del sistema)

Recomendaciones

Alta Prioridad

Deshabilitar FTP anónimo

En /etc/vsftpd.conf establecer anonymous_enable=NO. Si se necesita compartir archivos externamente, usar SFTP con autenticación por clave o un servicio con control de acceso adecuado. Nunca almacenar documentos corporativos en directorios de FTP anónimo.

Alta Prioridad

Forzar cambio de contraseña en primer login

Usar chage -d 0 usuario en Linux para marcar la contraseña como expirada, forzando el cambio en el siguiente login SSH. Implementar un sistema de gestión de identidades que verifique que el cambio se realizó. Auditar periódicamente cuentas con contraseñas por defecto.

Prioridad Media

No documentar contraseñas por defecto en PDFs

La política de contraseñas no debería incluir la contraseña por defecto en texto plano. En su lugar, usar un sistema de onboarding que genere contraseñas únicas por usuario enviadas por canal seguro (no correo interno). El PDF debería describir los requisitos sin revelar la contraseña.

Prioridad Media

Segmentar acceso a PostgreSQL

Crear usuarios PostgreSQL dedicados con permisos mínimos (solo SELECT en tablas necesarias). No reutilizar credenciales del sistema operativo en bases de datos. Configurar pg_hba.conf para requerir autenticación fuerte. Considerar cifrado de datos sensibles en la BD.

Baja Prioridad

Monitoreo de FTP y SSH

Activar logging detallado en vsftpd (xferlog_enable=YES). Implementar fail2ban para SSH. Alertar sobre logins FTP anónimos con descarga de archivos. Monitorear comandos SSH ejecutados (ForceCommand + logging).

Baja Prioridad

Principio de mínimo privilegio en FTP

Si se necesita FTP para distribución de archivos, crear un directorio con solo archivos públicos (manuales de uso, no documentos corporativos). Revisar qué archivos están accesibles. Nunca exponer backups de correo o documentos internos por FTP.

8

Panel de Agentes Ghost-Ops

⚔ RTO — Red Team

FTP anon siempre debe explorarse antes de intentar exploits complejos. Buscar archivos .pdf, .docx, .txt — especialmente en directorios con nombres como backup, mail, internal. Con usuarios del email → brute SSH/FTP/RDP/SMB con pass por defecto. En el host: ss -tlnp inmediatamente para descubrir servicios internos. Port forwarding SSH es invisble para firewalls perimetrales.

🛡 BTA — Blue Team

Ruido generado: BAJO. FTP anon = 1 conexión TCP limpia. SSH login = 1 intento por usuario (no es brute fuerza voluminosa). El túnel SSH-L es encriptado y pasa por el puerto 22 → casi invisible para DPI básico. Detectable con: monitoreo de sesiones SSH con comandos sospechosos (ss -tlnp es legítimo). Alertar sobre FTP anon + descarga de archivos PDF/DOC.

🎭 BHM — Black Hat

"Funnel" = embudo de información. Siempre buscar el elemento más débil de la cadena: aquí fue el usuario que no cambió la pass. En entornos reales: después de PostgreSQL, buscar credenciales hardcodeadas en la BD para movimiento lateral. SSH -L también sirve para tunelizar RDP, SMB, HTTP interno. Combinar con -D (SOCKS proxy) para acceso completo a red interna.

Lecciones Aprendidas

⚔ Para el Atacante

  • FTP anónimo = gold mine de información. Siempre descargar TODOS los archivos y leerlos
  • PDFs tienen texto en streams zlib comprimidos — pdftotext o Python + zlib los extrae
  • Lista de usuarios del sistema = lista de targets para brute force
  • ss -tlnp (o netstat -tlnp) es el primer comando tras obtener shell — ver servicios internos
  • SSH -L localport:127.0.0.1:remoteport = acceso a cualquier servicio en loopback del servidor
  • PostgreSQL por defecto crea rol con el mismo nombre que el usuario del SO → probar misma contraseña
  • La contraseña por defecto puede valer para múltiples servicios (SSH + BD)

🛡 Para el Defensor

  • FTP anónimo = NO en producción. Si necesitas distribución de archivos, usa HTTPS o SFTP con auth
  • Jamás almacenar documentos corporativos sensibles en directorios FTP accesibles

Glosario

FTP (File Transfer Protocol) Protocolo de red para transferencia de archivos. Puerto 21 TCP. Sin cifrado. El modo anónimo permite acceso sin credenciales.
vsftpd Very Secure FTP Daemon — servidor FTP para Linux. Nombre irónico dado que la seguridad depende de su configuración. Aquí tiene anonymous login habilitado.
FlateDecode / zlib Algoritmo de compresión usado en PDFs para comprimir streams de datos (texto, imágenes). zlib.decompress() en Python lo descomprime para extraer texto plano.
sshpass Utilidad Linux que proporciona contraseñas SSH de forma no interactiva a través de variable de entorno. Permite automatizar logins SSH en scripts.
SSH Port Forwarding (-L) Funcionalidad SSH que crea túneles TCP. Local forwarding (-L) redirige un puerto local al servidor remoto. Permite acceder a servicios internos no expuestos.
PostgreSQL Sistema de base de datos relacional open-source. Puerto 5432. psql es su cliente CLI. pg_hba.conf controla quién puede conectarse y desde dónde.
ss (Socket Statistics) Herramienta Linux moderna para ver sockets de red activos. Reemplaza a netstat. -tlnp muestra sockets TCP en LISTEN con proceso asociado.
Loopback (127.0.0.1) Interfaz de red virtual que solo acepta conexiones del mismo sistema. Servicios escuchando en 127.0.0.1 no son accesibles desde la red externa — solo localmente.
Credenciales por defecto Contraseñas predefinidas en sistemas o aplicaciones antes de que el usuario las personalice. Son la causa de numerosas brechas reales si no se cambian en el primer uso.
PGPASSWORD Variable de entorno de PostgreSQL que contiene la contraseña para conexiones no interactivas. Alternativa a escribir la contraseña manualmente en psql.
