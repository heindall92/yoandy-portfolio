Explosion — RDP Unauthenticated Access

Acceso a un servidor Windows mediante RDP (Remote Desktop Protocol) sin contraseña. Se utiliza xfreerdp para conectarse con el usuario Administrator sin credenciales, obteniendo acceso gráfico completo al escritorio del sistema.

Resumen Ejecutivo

La máquina Explosion es un sistema Windows Server 2019 que expone RDP (puerto 3389) y WinRM (puerto 5985). La misconfiguration central es que la cuenta Administrator no tiene contraseña configurada — o la contraseña es una cadena vacía. Esto permite acceso remoto completo sin credenciales. La flag se encuentra directamente en el Escritorio del Administrador.

La lección clave: en máquinas Windows HTB, probar siempre contraseña vacía con el usuario Administrator. Y cuando RDP falla con pass vacío, evil-winrm puede funcionar igualmente.

¿Qué es RDP y por qué es un vector de ataque?

RDP (Remote Desktop Protocol) es el protocolo de Microsoft para acceso remoto con interfaz gráfica. Puerto 3389 TCP. Permite controlar un Windows remotamente como si estuvieras frente al monitor. En entornos corporativos es ubicuo — y frecuentemente mal configurado.

⚠ Administrator sin contraseña = acceso total sin restricciones

Si la cuenta Administrator tiene contraseña vacía y RDP está habilitado, cualquier atacante en la red puede conectarse con acceso de nivel más alto posible en Windows. No hay exploit, no hay CVE — solo una misconfiguration básica que da las llaves del castillo.

Reconocimiento — Paso a Paso

Paso 1 — Reconocimiento

Nmap fase rápida — Todos los puertos

¿Por qué escanear todos los puertos en Windows?

Windows expone docenas de servicios por defecto (RPC, NetBIOS, SMB, RDP, WinRM). Necesitamos el mapa completo para priorizar. RDP en 3389 = acceso gráfico completo. WinRM en 5985 = shell PowerShell. Cualquiera de los dos con creds vacías = game over.

nmap -p- --open -n -T4 10.129.6.217
-p- → todos los 65535 puertos TCP
--open → mostrar solo puertos abiertos (filtra closed/filtered)
-n → sin resolución DNS (más rápido)
-T4 → velocidad agresiva, apropiado en lab HTB

¿Qué significa cada puerto?
  • 135/tcp msrpc → Microsoft RPC — comunicación interna Windows
  • 139/445 SMB → File sharing Windows — vector secundario
  • 3389/tcp ms-wbt-server → RDP — acceso gráfico remoto. Objetivo principal.

Decisión tomada

RDP (3389) y WinRM (5985) son los vectores. En Starting Point, siempre probar credenciales vacías/default primero. Fase profunda para confirmar OS y versión.

Explotación — Método 1: xfreerdp (acceso gráfico)

Paso 3A — Explotación (Método Principal)
✅ MÉTODO PRINCIPAL — xfreerdp

Conectar vía RDP con administrator sin contraseña

¿Qué es xfreerdp?

xfreerdp es el cliente RDP de código abierto para Linux, parte del proyecto FreeRDP. Permite conectarse a escritorios Windows remotos desde Kali. Es la alternativa libre a "Remote Desktop Connection" de Windows. El flag /d:htb es crítico — especifica el dominio y permite la autenticación sin contraseña.

xfreerdp /v:10.129.6.217 /u:administrator /p:"" /cert:ignore /dynamic-resolution +clipboard /d:htb
xfreerdp → cliente RDP para Linux (paquete freerdp2-x11)
/v:10.129.6.217 → (server) IP del objetivo
/u:administrator → usuario a autenticar
/p:"" → contraseña VACÍA (cadena vacía entre comillas dobles)
/cert:ignore → ignorar errores de certificado SSL/TLS (común en labs)
/dynamic-resolution → ajustar resolución de pantalla automáticamente
+clipboard → habilitar portapapeles compartido (copiar/pegar entre Kali y Windows)
/d:htb → ⚠ CRÍTICO: especificar dominio "htb". Sin este flag, xfreerdp falla con "connection reset" aunque las credenciales sean correctas
¿Por qué /d:htb es la clave?

Sin /d:htb, xfreerdp intenta autenticarse como usuario local y el servidor rechaza la conexión (connection reset by peer). Al especificar el dominio, el servidor procesa la autenticación correctamente y acepta la contraseña vacía. Esta es la diferencia entre que funcione o no.

Resultado

Se abre sesión RDP completa con escritorio gráfico de Windows Server 2019. La flag está en el Desktop del Administrator.

Explotación — Método 2: evil-winrm (shell de línea de comandos)

Paso 3B (Alternativo) — Explotación vía WinRM
🔵 MÉTODO ALTERNATIVO — evil-winrm

Shell remota PowerShell con administrator sin contraseña

¿Qué es evil-winrm?

evil-winrm es una herramienta de pentesting que aprovecha el servicio WinRM (Windows Remote Management) para obtener una shell interactiva de PowerShell. Es el equivalente a SSH para Windows. Útil cuando RDP no está disponible o cuando necesitamos automatizar comandos.

evil-winrm -i 10.129.6.217 -u administrator -p ''
evil-winrm → cliente WinRM para pentesting, genera shell PowerShell interactiva
-i 10.129.6.217 → IP del objetivo
-u administrator → usuario
-p '' → contraseña vacía (comillas simples = string vacío en bash)

¿Cuándo usar evil-winrm vs xfreerdp?
  • xfreerdp → cuando necesitas GUI, explorar archivos visualmente, o cuando no tienes automatización
  • evil-winrm → cuando necesitas ejecutar comandos, subir/bajar archivos, pivotar, o cuando RDP falla
  • Regla Ghost-Ops: Si xfreerdp falla sin /d:htb, probar evil-winrm. Si evil-winrm falla, revisar el dominio en xfreerdp.

Panel de Agentes Ghost-Ops

⚔ RTO — Red Team

Vector directo: RDP + WinRM ambos con administrator:empty. Dos vectores paralelos disponibles. RDP da GUI completa. WinRM da shell scriptable. En red real: dump del SAM/LSASS para extraer todos los hashes.

🛡 BTA — Blue Team

Nivel de ruido: ALTO. Login RDP genera Event ID 4624 (logon type 10) + 4648. WinRM genera 4624 (type 3) + logs en Microsoft-Windows-WinRM. Detectable con cualquier SIEM básico.

🎭 BHM — Black Hat

Administrator sin pass = acceso total. En red real: usar como pivote para moverse lateralmente, dump de credenciales del dominio si está unido a AD, instalar C2 agent. Flag fue lo menos interesante.

Lecciones Aprendidas

⚔ Para el Atacante

  • Siempre probar administrator:"" en RDP y WinRM antes de cualquier brute force
  • /d:htb en xfreerdp es clave para HTB — sin dominio, falla con pass vacío
  • Si RDP falla → probar WinRM con evil-winrm (mismo usuario/pass)
  • RDP (3389) + WinRM (5985) = dos puertas al mismo lugar
  • TTL=127 → Windows → pensar RDP, WinRM, SMB inmediatamente

🛡 Para el Defensor

  • Nunca dejar Administrator sin contraseña — política de contraseñas mínima obligatoria
  • Deshabilitar RDP si no es necesario: Set-ItemProperty -Path 'HKLM:\...\Terminal Server' -Name fDenyTSConnections -Value 1
  • Restringir WinRM por IP con GPO o firewall
  • Detectar: Event ID 4624 Logon Type 10 desde IPs externas
  • Usar NLA + contraseñas fuertes + MFA para RDP en producción
