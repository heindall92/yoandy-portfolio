CrownJewel-1 Sherlock

📋
Escenario

El Domain Controller de la organización Forela está bajo ataque. La cuenta del administrador de dominio está potencialmente comprometida y se sospecha que el atacante ha volcado la base de datos NTDS.dit. El equipo de respuesta ha recibido una alerta de uso de vssadmin en el DC, lo cual es inusual. Los artefactos proporcionados son la Master File Table ($MFT) y tres archivos de logs de eventos de Windows (.evtx).

🏆 ¿Qué es NTDS.dit?
NTDS.dit es la base de datos central de Active Directory. Reside en los Domain Controllers de Windows y contiene todos los datos de identidad del dominio: cuentas de usuario, grupos, políticas y — lo más crítico — los hashes de contraseña de todas las cuentas del dominio, incluida la cuenta krbtgt. El hash krbtgt es la clave raíz de Kerberos: quien lo posea puede fabricar Tickets de Concesión de Tickets (TGT) válidos para cualquier cuenta, un ataque conocido como Golden Ticket. De ahí el nombre: CrownJewel.
🔴 Scattered Spider — Threat Actor Scattered Spider (UNC3944) es un grupo de ciberdelincuentes activo desde 2022, conocido por ataques de ingeniería social sofisticados para obtener acceso inicial mediante el agotamiento de agentes de soporte técnico (MFA fatigue). Una vez dentro, escalan privilegios en Active Directory para acceder a NTDS.dit.

Preparación — Chainsaw + jq

Chainsaw es una herramienta de análisis forense de Windows diseñada para ejecutarse en Linux. Su función dump convierte los archivos EVTX y MFT a formato JSONL, permitiendo filtrado flexible con jq: 

chainsaw dump --jsonl -o events.jsonl Artifacts/*.evtx
chainsaw dump --jsonl -o mft.jsonl Artifacts/C/\$MFTBash — Preparación

Reconstrucción del Ataque

1
Inicio de VSS — Timestamp de Activación
Event ID 7036 — Service Control Manager

El primer paso es identificar cuándo se inició el servicio de Volume Shadow Copy. El EventID 7036 es generado por el Service Control Manager cada vez que un servicio cambia de estado. Filtrando por este ID y buscando el servicio de VSS:

jq — Filtro EventID 7036 + VSS
cat events.jsonl | jq '.Event | select(.System.EventID == 7036) | select(.EventData.param1 == "Volume Shadow Copy")'

El resultado revela el timestamp exacto en que el servicio de VSS entró en estado de ejecución, marcando el inicio del ataque.

2
Enumeración de Grupos — VSSVC.exe
Event ID 4799 — Group Membership Enumeration

Al crear una shadow copy, el proceso VSS (VSSVC.exe) valida privilegios consultando grupos de usuarios. El EventID 4799 registra estas consultas de enumeración de grupos. Esto permite identificar qué grupos consultó VSSVC y qué cuenta de máquina realizó la acción:

jq — Grupos consultados por VSSVC.exe
cat events.jsonl | jq '.Event | select(.System.EventID == 4799) | select(.EventData.CallerProcessName == "C:\\Windows\\System32\\VSSVC.exe") | .EventData.TargetUserName' | sort -u
CUENTA DE MÁQUINA
DC01$
La cuenta DC01$ (la cuenta de equipo del Domain Controller) es la que realiza la enumeración. Esto confirma que la operación está ocurriendo en el DC objetivo.
3
Identificación del GUID de Volumen
NTFS Operational Log — VolumeCorrelationId

Cuando VSS monta la shadow copy como unidad virtual, el driver NTFS registra el GUID del volumen en el log operacional Microsoft-Windows-Ntfs/Operational. Este GUID es fundamental para correlacionar eventos relacionados con la shadow copy específica:

jq — Extraer GUID del volumen shadow
cat events.jsonl | jq -c '.Event | select(.System.Channel == "Microsoft-Windows-Ntfs/Operational")' | grep -i "shadow" | jq '.EventData.VolumeCorrelationId'
4
Localización de NTDS.dit en el MFT
Ruta completa del archivo volcado

La Master File Table registra todos los archivos creados en el sistema. Filtrando por el nombre ntds.dit, se identifica la ubicación donde el atacante volcó la base de datos de Active Directory:

jq — Buscar ntds.dit en el MFT
cat mft.jsonl | jq -c | grep -i "ntds.dit" | jq .FullPath
RUTA DEL VOLCADO (NTDS.dit)
C:\Users\Administrator\Documents\backup_sync_Dc\Ntds.dit
El atacante nombró la carpeta backup_sync_Dc para simular una operación de backup legítima. Una táctica clásica de camuflaje.
5
Volcado del SYSTEM Hive
Artefacto adicional en la misma carpeta

NTDS.dit por sí solo no es suficiente para descifrar los hashes: el atacante también necesita el subárbol de registro SYSTEM, que contiene el SYSKEY (la clave de cifrado de los hashes almacenados en la base de datos AD). Filtrando por la carpeta del volcado en el MFT:

jq — Todos los archivos en la carpeta del volcado
cat mft.jsonl | jq '. | select(.FullPath | contains("Users/Administrator/Documents/backup_sync_dc/"))'

Junto a Ntds.dit, se encuentra el hive de registro SYSTEM volcado. Con ambos archivos, el atacante puede extraer todos los hashes del dominio usando herramientas como secretsdump.py de Impacket y proceder al ataque de Golden Ticket.

📊
Conclusiones

CrownJewel-1 ilustra perfectamente el riesgo de comprometer un Domain Controller en un entorno Windows. El acceso a NTDS.dit equivale al control total del dominio: el atacante puede impersonar cualquier usuario, incluyendo los administradores, durante meses sin necesidad de las contraseñas originales.

✅ Lecciones de Active Directory Forensics El análisis forense de un ataque de volcado de NTDS.dit requiere correlacionar tres fuentes de evidencia: logs de eventos del sistema (VSS), logs de seguridad (enumeración de grupos) y el MFT (archivos creados). Chainsaw + jq proporcionan un workflow eficiente para este análisis desde Linux.
🔴 El Riesgo del Golden Ticket Una vez que el atacante tiene el hash krbtgt, la solución no es simplemente cambiar contraseñas: el hash krbtgt debe rotarse dos veces (para invalidar tickets existentes), y todas las cuentas privilegiadas deben considerarse comprometidas hasta que se realice un reset completo.
