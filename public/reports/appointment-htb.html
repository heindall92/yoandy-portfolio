Appointment Machine

Aplicación web con formulario de login vulnerable a SQL Injection. Bypass de autenticación mediante inyección SQL clásica para obtener la flag.

AutorYoandy Ramírez Delgado
TipoHTB Starting Point

Resumen Ejecutivo

La máquina Appointment expone un servidor web Apache 2.4.38 en el puerto 80 con un formulario de login vulnerable a SQL Injection. Al inyectar admin'# como nombre de usuario, el carácter ' cierra el string SQL y # comenta el resto de la query (incluyendo la verificación de contraseña), permitiendo autenticarse como admin sin conocer su contraseña real. La flag apareció en el dashboard tras el login. 2 comandos — récord absoluto del Starting Point.

¿Qué es SQL Injection?

SQL Injection es una vulnerabilidad donde el input del usuario se interpreta directamente como código SQL, permitiendo al atacante modificar la lógica de las consultas a la base de datos. Es una de las vulnerabilidades más antiguas (documentada desde 1998) y figura en el OWASP Top 10 desde su primera edición.

⚠ Login bypass — el tipo más directo de SQLi

En lugar de extraer datos (que requiere más pasos), el login bypass simplemente hace que el servidor autentique al atacante sin verificar la contraseña. Resulta en acceso inmediato al panel de administración.

Paso 1 — Reconocimiento

Nmap — Descubrir Apache con formulario de login

¿Por qué nmap directamente?

Combinamos ping + nmap en este paso. El output de -sC en HTTP incluye el título de la página — si dice "Login", hay un formulario de autenticación que potencialmente es vulnerable a SQLi.

nmap -p80 -sC -sV -n 10.129.18.216
-p80 → solo el puerto 80 (HTTP). Un escaneo previo confirmó que es el único puerto.
-sC → scripts NSE por defecto. Para HTTP: http-title (extrae el título de la página), http-server-header (versión del servidor).
-sV → versión exacta del servidor web
-n → sin resolución DNS
Output
PORT   STATE SERVICE VERSION
80/tcp open  http    Apache httpd 2.4.38 ((Debian))
|_http-server-header: Apache/2.4.38 (Debian)
|_http-title: Login
¿Qué nos dice cada línea?
  • Apache httpd 2.4.38 (Debian) → versión específica de Apache en Debian (→ PHP como backend → MySQL como DB → vulnerable a SQLi con #)
  • http-title: Login → CLAVE: el título de la página es "Login" — hay un formulario de autenticación que es nuestro vector
Decisión tomada

Formulario de login en Apache+PHP. Stack Apache+Debian = MySQL como backend. Intentar directamente SQLi con payload admin'# (comentario MySQL).

SQL Injection Login Bypass con curl POST

¿Por qué curl y no el navegador?

curl permite hacer peticiones HTTP desde la línea de comandos de forma scriptable, reproducible y sin interfaz gráfica. Perfecto para probar payloads de forma rápida y automatizable. -X POST y -d replican exactamente lo que haría un navegador al enviar el formulario.

curl cliente HTTP de terminal
-s silent: sin barra de progreso
-X POST método HTTP POST (el form usa POST)
http://10.129.18.216/ URL destino (action del form)
-d datos del body de la petición
"username=admin'%23&password=anything"
admin → nombre de usuario objetivo
' → cierra el string SQL (¡el carácter mágico!)
%23 → URL encoding de # (ASCII 35 = hex 23). El # es el comentario de línea en MySQL. Las comillas del shell interpretarían # como comentario bash si no se URL-encode.
password=anything → cualquier valor — el comentario # hace que el servidor lo ignore completamente

Anatomía del SQL Injection — Visualización

Normal:
SELECT * FROM users WHERE username='admin' AND password='secreto'
Con payload:
SELECT * FROM users WHERE username='admin''#' AND password='anything'
Efectiva:
SELECT * FROM users WHERE username='admin' ← devuelve admin sin verificar password ✓

¿Por qué funciona? El carácter # es el delimitador de comentario de línea en MySQL. Todo lo que viene después de # es ignorado por el intérprete SQL. La query efectiva solo verifica que el usuario "admin" exista — sin importar la contraseña.

Código Vulnerable vs Código Seguro

❌ Código PHP Vulnerable
$username = $_POST['username'];
$password = $_POST['password'];

$query = "SELECT * FROM users
  WHERE username='$username'
  AND password='$password'";

El input del usuario se concatena directamente en la query SQL. Cualquier carácter SQL en el input modifica la lógica de la query.

✅ Código Seguro — Prepared Statements
$stmt = $pdo->prepare(
  "SELECT * FROM users
  WHERE username = ?
  AND password = ?");

$stmt->execute([$user, $pass]);

Los ? son placeholders. PDO envía la query y los valores por separado. El motor SQL los trata como datos, nunca como código.

Analogía: Con prepared statements es como darle a alguien un formulario con espacios en blanco — no puede alterar la estructura del documento, solo llenar los espacios asignados. Sin ellos, es como dejarle escribir el documento completo.

Variantes del Ataque — Diferentes Payloads SQLi

Payload usernameBackendExplicación
admin'#MySQL# comenta hasta fin de línea — el más común en MySQL
admin'--SQL Server / PostgreSQL-- comentario estándar SQL, funciona en múltiples backends
admin'-- -MySQL varianteEl espacio después del -- es necesario en algunos contextos
' OR '1'='1UniversalSiempre true — accede como el primer usuario en la tabla
' OR 1=1--UniversalSiempre true con comentario estándar

¿Por qué usamos # y no -- ? El stack Apache+Debian con PHP típicamente usa MySQL, y # es el comentario de línea nativo de MySQL. -- también funciona en MySQL pero requiere un espacio después (-- ), lo que puede complicarse en URL encoding.

Lecciones Aprendidas

⚔ Para el Atacante

  • Apache+PHP+MySQL → siempre probar SQLi en formularios de login
  • Payload básico: admin'# — 90% de éxito en MySQL sin WAF
  • %23 = URL encoding de # para evitar interpretación bash
  • Si manual falla: sqlmap -u URL --data="user=a&pass=b" --dbs
  • Nmap -sC revela el título "Login" → señal de formulario objetivo

🛡 Para el Defensor

  • Usar Prepared Statements siempre — es el fix definitivo
  • Validar y sanitizar inputs (rechazo de caracteres SQL)
  • WAF: ModSecurity detecta ' y # en campos de login
  • Nunca mostrar errores SQL al usuario (revelan estructura DB)
  • Principio de mínimo privilegio: el usuario DB solo necesita SELECT
