Mongod Machine

MongoDB expuesto en el puerto 27017 sin autenticación. Enumeración de bases de datos, listado de colecciones y extracción directa de la flag almacenada.

AutorYoandy Ramírez Delgado
TipoHTB Starting Point

Objetivoroot

Resumen Ejecutivo

Mongod expone MongoDB 3.6.8 en el puerto 27017 sin ningún tipo de autenticación. Cualquier cliente puede conectarse y leer todas las bases de datos. La base de datos sensitive_information contiene una colección flag con la flag en texto plano. Un solo comando — mongoexport — es suficiente para extraerla directamente a la terminal sin ninguna contraseña.

nmap → 27017 MongoDB
→
nmap lista DBs (sin auth)
→
mongoexport → flag
→
FLAG ✅

¿Qué es MongoDB y por qué es peligroso sin auth?

MongoDB es una base de datos NoSQL orientada a documentos. En vez de tablas y filas usa colecciones y documentos JSON. Puerto por defecto: 27017. Cuando se instala sin configurar autenticación, cualquier cliente en la red puede conectarse, leer, escribir y borrar cualquier dato.

⚠ MongoDB por defecto NO tiene autenticación activada

En versiones antiguas (incluyendo 3.6.8), la instalación por defecto no requiere usuario ni contraseña. Basta con conectarse a la IP y puerto para tener acceso completo a todos los datos. En 2017-2019 se publicaron miles de bases de datos MongoDB expuestas en internet con datos reales de usuarios.

Reconocimiento — Paso a Paso

Paso 1 — Reconocimiento

Nmap — Dos hallazgos en un solo escaneo

¿Por qué nmap ya revela el contenido de MongoDB?

Nmap incluye el script NSE mongodb-databases que se activa automáticamente con -sC al detectar el puerto 27017. Este script se conecta al MongoDB como cliente anónimo y solicita la lista de bases de datos. Si no hay autenticación, MongoDB la entrega sin preguntar — y nmap la muestra en el output. En este caso ya vemos sensitive_information antes de lanzar ningún otro comando.

nmap -p22,27017 -sC -sV -n 10.129.6.232
-p22,27017 → puertos específicos descubiertos en fase 1 (SSH y MongoDB)
-sC → scripts NSE por defecto — activa mongodb-databases automáticamente al ver puerto 27017
-sV → versión exacta del servicio
-n → sin DNS
Output clave
PORT      STATE SERVICE VERSION
22/tcp    open  ssh     OpenSSH 8.2p1 Ubuntu 4ubuntu0.5
27017/tcp open  mongodb MongoDB 3.6.8

| mongodb-databases:
|   databases
|     0: name = admin
|     1: name = config
|     2: name = local
|     3: name = sensitive_information
|     4: name = users
|_  ok = 1.0
¿Qué nos dice este output?
  • MongoDB 3.6.8 → versión antigua (2018). Sin auth por defecto.
  • mongodb-databases script conectó sin autenticación y devolvió lista completa de DBs
  • sensitive_information → nombre exacto de la DB objetivo. Contiene algo que no debería ser público.
  • users → DB extra — puede contener credenciales para otras máquinas o servicios
  • No necesitamos explorar más — el objetivo está identificado: sensitive_information
Decisión tomada

MongoDB sin auth confirmado. DB objetivo identificada: sensitive_information. Conectar directamente y volcar la colección flag.

Explotación — El Comando que Obtuvo la Flag

⚡ Comando exacto que extrajo la flag
mongoexport --host 10.129.6.232 --port 27017 --db sensitive_information --collection flag
Output en terminal
2026-03-16T09:42:57.137-0400  connected to: mongodb://10.129.6.232:27017/
{"_id":{"$oid":"630e3dbcb82540ebbd1748c5"},"flag":"1b6e6fb359e7c40241b6d431427ba6ea"}
2026-03-16T09:42:57.484-0400  exported 1 record
Paso 2 — Explotación

¿Qué hace exactamente cada parte del comando?

¿Por qué mongoexport y no mongo ni mongosh?
  • mongo shell y mongosh modernos (v7+) usan el comando hello para el handshake inicial, introducido en MongoDB 5.0+. MongoDB 3.6.8 no lo entiende y cierra la conexión con error.
  • mongoexport usa el protocolo de wire version 6 directamente, compatible con MongoDB 3.6.x sin problemas.
  • Regla Ghost-Ops: para MongoDB <4.2 usar siempre mongoexport o mongodump, nunca el shell interactivo moderno.
mongoexport --host 10.129.6.232 --port 27017 --db sensitive_information --collection flag
mongoexport → herramienta oficial de MongoDB para exportar colecciones a JSON/CSV. Compatible con versiones antiguas del servidor.
--host 10.129.6.232 → IP del servidor MongoDB objetivo
--port 27017 → puerto estándar de MongoDB (se puede omitir si es el default, pero es buena práctica incluirlo)
--db sensitive_information → nombre de la base de datos a consultar — identificada previamente con nmap
--collection flag → nombre de la colección dentro de la DB — contiene el documento con la flag
⚠ No se pasa --username ni --password → se conecta anónimamente. MongoDB sin auth acepta la conexión.
¿Qué significa el output JSON?
  • "_id": {"$oid": "630e3..."}} → identificador único del documento en MongoDB (ObjectId). No nos interesa.
  • "flag": "1b6e6fb359e7c40241b6d431427ba6ea" → el campo flag del documento. Ese es el valor que necesitamos.
  • exported 1 record → solo hay 1 documento en la colección flag — lo hemos volcado todo.
Resultado

Flag obtenida con 1 comando. Sin exploits, sin credenciales, sin bypass — solo conectarse a un servicio mal configurado y pedir los datos.

Bonus — Exploración completa

Volcado total con mongodump

¿Qué más hay en el servidor?

En un pentest real, no nos limitamos a la flag. Volcamos todo para buscar credenciales, datos de usuarios, configuraciones internas. mongodump descarga todas las bases de datos en formato BSON.

mongodump --host 10.129.6.232 --port 27017 --out /tmp/mongod_loot
mongodump → descarga todas las DBs del servidor en formato BSON (binario MongoDB)
--out /tmp/mongod_loot → directorio donde guardar el dump completo
Output
writing sensitive_information.flag → flag.bson  (1 document)
writing users.credentials → credentials.bson    (N documents)
writing admin.system.users → ...
done dumping sensitive_information.flag

Panel de Agentes Ghost-Ops

⚔ RTO — Red Team

Puerto 27017 = MongoDB. Sin auth = acceso total. mongoexport para versiones antiguas. En red real: volcar DB users para credenciales, pivoting a otros servicios. RCE posible en versiones vulnerables via $where JS injection.

🛡 BTA — Blue Team

Nivel de ruido: MUY BAJO. Una conexión TCP al 27017 = 1 entrada en el log. Sin IDS básico, pasa invisible. MongoDB no genera alertas de acceso anónimo por defecto. Detectable solo con firewall de red bloqueando 27017.

🎭 BHM — Black Hat

Nmap ya lista las DBs antes de conectar — objetivo identificado sin tocar el servidor MongoDB directamente. "Mongod" = el nombre del proceso MongoDB → el nombre de la máquina es la pista del servicio. DB users puede tener creds para SSH.

Lecciones Aprendidas

⚔ Para el Atacante

  • Puerto 27017 → MongoDB → intentar siempre sin auth primero
  • Nmap -sC con mongodb-databases script ya lista las DBs sin conectarse manualmente
  • Usar mongoexport/mongodump para MongoDB <4.2 — mongo/mongosh modernos son incompatibles
  • DB llamada sensitive_information → objetivo evidente. Siempre buscar nombres descriptivos
  • Volcar también users → posibles credenciales para SSH u otros servicios

🛡 Para el Defensor

  • Habilitar autenticación en MongoDB: security.authorization: enabled en mongod.conf
  • Bloquear puerto 27017 en firewall — solo acceso desde localhost o IPs internas
  • Actualizar MongoDB a versión reciente (7.x) con auth habilitada por defecto
  • Monitorizar conexiones entrantes al 27017 en SIEM
  • Usar bind_ip para limitar interfaces: net.bindIp: 127.0.0.1
